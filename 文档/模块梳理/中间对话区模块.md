# 中间对话区模块 Codemap

## Responsibility

- 负责当前会话消息流的展示、渲染与滚动管理，承担多会话缓存切换的核心数据层
- 仅服务于 `/chat` 会话主路由，不再承担左侧导航或其它一级功能页的页面切换职责
- 支持新会话草稿态与正式会话的无缝过渡，管理父子会话回退导航与上下文继承
- 工作台通过 LRU 池保留最近访问的 5 个正式会话；新建会话先进入独立草稿视图，首条消息提交后在同一实例内切换成正式会话
- 会话活跃态由 `useSessionLruPool` 与 `SessionTabArea` 托管；主路由切换离开 `/chat` 时通过 `KeepAlive` 保活，不允许改由路由参数直接控制会话实例生命周期
- 当前单会话工作台已经升级为“消息区 + 操作区 + 右侧功能区”三列；操作区内的文件预览标签、Markdown 自动保存状态、块级 AI 动作、大文件窗口状态、代码高亮状态和当前活动文件都跟随 `SessionTabContent` 实例保活，而不是落到全局 store。
- 实现历史消息按 user 轮次分页与向上扩窗机制；默认仅显示最近 3 轮，支持“展开更早 3 轮 / 折叠旧轮次 / 加载更早历史”的显式可见性控制；草稿不再做切换恢复
- 提供回到底部/脱离底部的滚动状态管理，处理输入框工作台的完整编排逻辑
- **完整支持富内容渲染：文本、思考过程、工具调用、工具结果、图片**
- 负责保证消息头部、输入区和展开内容块在 ridge 主题下通过 surface 层次分隔，而不是依赖裸 `border-*`，避免 `currentColor` 造成意外深色分割线。
- 负责把右侧文件树的文件点击事件转成独立操作区标签，而不是让文件树组件直接承担文件预览和编辑职责。
- 负责把 Markdown 文件在操作区内升级为“源码编辑 + 组件化块渲染 + 块级 AI 动作”，并把块上下文直接注入当前会话 composer，而不是落本地评论存储。
- 负责把 code/text 文件在操作区内升级为独立的只读预览，支持代码高亮、行号和大文件窗口加载，而不是继续复用普通文本 `<pre>`。
- 负责把 HTML 文件从代码预览链路中拆出，走独立 iframe sandbox 只读预览，避免正文直接进入主 DOM。
- 首期仅处理 `ask` 阻塞式交互卡片；文件mention与附件协议、虚拟列表渲染、语音输入等高级能力仍未覆盖

## Design

### Architecture Pattern

**分层数据流架构** — 中间对话区采用"状态中心 + 投影视图"的分层设计：

### Key Abstractions

#### 1. Pi 原始消息协议

```ts
import type { PiMessage } from '@pi/protocol'

type PiMessage = Message
```

- `PiMessage` 是前后端共享的真实协议消息实体，直接承载 user / assistant / toolResult 三类消息
- assistant 消息内部继续保留 Pi SDK 原生 `text/thinking/toolCall` 内容数组，不再在 Web 层重新发明 `contentBlocks`
- 由 `@pi/protocol` 统一导出，被 server、`session-snapshot.ts`、`usePiChat`、`usePiChatCore` 共同消费
- 关联文件：`packages/protocol/src/index.ts`、`packages/web/src/lib/types.ts`

#### 2. UiConversationMessage 渲染包装

```ts
interface UiConversationMessage {
  message: PiMessage
  pending?: boolean
  localId?: string
}
```

- 这是前端唯一允许附加 UI 元数据的消息包装层，只负责 `pending/localId`
- `message` 始终保持 Pi 原始协议，不在包装层写回 `role/content/details` 等业务字段
- 用于乐观用户消息、流式 assistant 占位消息以及消息列表渲染 key
- 关联文件：`packages/web/src/lib/types.ts`、`packages/web/src/lib/conversation.ts`

#### 3. 会话快照缓存桶 (Session Cache Entry)

```ts
type CachedSessionEntry = {
  snapshot: UiSessionSnapshot
  hydratedAt: number
}

interface UiSessionSnapshot extends Omit<SessionSnapshot, 'messages'> {
  messages: UiConversationMessage[]
  historyMeta: SessionHistoryMeta
}
```

- 协议层快照 `SessionSnapshot` 到达前端后，会立即被包装成 `UiSessionSnapshot`
- 包装只发生一次，集中在 `wrapUiSessionSnapshot()` 与 `session-snapshot.ts`，避免 `usePiChat` / `usePiChatCore` 各自组装
- 被 `sessionCache` 按 `sessionId` 分桶持有，支撑切会话秒切、SSE 回填与历史扩窗
- 关联文件：`packages/web/src/composables/session-snapshot.ts`、`packages/web/src/lib/conversation.ts`

#### 4. 流式消息本地占位

```ts
let currentStreamMessageLocalId = ''
```

- SSE 收到 `message_start` 时，前端先生成一个只带 `pending/localId` 的 `UiConversationMessage`
- `message_end` 到来后，用同一个 `localId` 替换占位消息，避免流式过程中重复插入
- 这套逻辑同时存在于 `usePiChat` 与 `usePerSessionChat`，但底层快照装配已收敛到共享模块
- 关联文件：`packages/web/src/composables/usePiChat.ts`、`packages/web/src/composables/usePerSessionChat.ts`

### Design Patterns

#### 1. 分桶缓存 + 去重加载 (Request Deduplication)

```ts
const loadSessionInFlightById = new Map<string, Promise<UiSessionSnapshot>>()
const loadSessionRequestSeqById = new Map<string, number>()
```

- 防止并发加载同一 session 产生重复请求
- 去重后的返回值始终是 `UiSessionSnapshot`，不会在不同入口混入 raw `SessionSnapshot`
- 应用于：`usePiChat.ts`、`usePiChatCore.ts`、`session-snapshot.ts`

#### 2. 协议快照集中装配

```ts
const wrapUiConversationMessage = (message: PiMessage) => ({
  message,
})

const wrapUiSessionSnapshot = (snapshot: SessionSnapshot): UiSessionSnapshot => ({
  ...snapshot,
  messages: snapshot.messages.map(wrapUiConversationMessage),
})
```

- 把协议消息包装成 UI 消息、把 raw snapshot 包装成 UI snapshot 的逻辑统一放进 `conversation.ts`
- `session-snapshot.ts` 负责 hydrate / patch / stream snapshot 三条入口共用这套装配逻辑
- 这样修改协议字段时，只需要改共享装配层，不会再出现两套 composable 分别演进
- 关联文件：`packages/web/src/lib/conversation.ts`、`packages/web/src/composables/session-snapshot.ts`

#### 3. 最终回答与过程消息分层

```ts
const conversationLayout = computed(() => ({
  rounds: [
    {
      userMessage,
      processMessages,
      finalMessage,
    },
  ],
}))
```

- 每轮消息会把最后一个带 assistant 文本内容的消息识别为 `finalMessage`
- 其余 thinking / toolCall / toolResult / 图片消息都进入 `processMessages`
- UI 层通过 `ProcessMessagesFold` 折叠过程消息，保持回答主线简洁
- 关联文件：`packages/web/src/components/workbench/chat/WorkbenchMessageStream.vue`

#### 4. 组件化协议内容渲染

```vue
<ChatMessageItem :message="userMessage" />
<ProcessMessagesFold :messages="processMessages" />
<ThinkingCard :content="content" />
<ToolCallCard :tool-call="content" />
<ToolResultCard :message="toolResultMessage" />
```

- `ChatMessageItem` 只负责用户消息和最终 assistant 文本
- `ProcessMessageItem` 负责协议型内容块渲染：text / thinking / toolCall / image / toolResult
- `ToolResultCard` 单独处理 ask 工具结果与普通工具结果，避免再把 toolResult 混回 `ToolCallCard`
- 所有类型守卫都优先在脚本层完成，模板层只消费已收窄的数据
- 关联文件：ChatMessageItem.vue, ThinkingCard.vue, ToolCallCard.vue

## Flow

### Flow 1: SSE 流式富内容接收流程

```
SSE onmessage
    │
    ▼
解析 StreamEvent
    │
    ├─► type: 'message_start'
    │   currentAssistantMessageId = createLocalId()
    │   currentAccumulatedBlocks = extractContentBlocks(payload.message)
    │   appendMessageToSession({ role: 'assistant', contentBlocks, pending: true })
    │
    ├─► type: 'message_update'
    │   │
    │   ├─► assistantMessageEvent.type: 'text_delta'
    │   │   patchSessionSnapshot -> 更新对应 text block
    │   │
    │   ├─► assistantMessageEvent.type: 'thinking_delta'
    │   │   patchSessionSnapshot -> 更新/创建 thinking block
    │   │
    │   ├─► assistantMessageEvent.type: 'toolcall_start'
    │   │   currentPendingToolCall = { type: 'toolCall', ... }
    │   │   添加到 contentBlocks
    │   │
    │   ├─► assistantMessageEvent.type: 'toolcall_delta'
    │   │   累积 JSON 参数到 currentPendingToolCall.arguments
    │   │
    │   └─► assistantMessageEvent.type: 'toolcall_end'
    │       currentPendingToolCall = null
    │
    └─► type: 'message_end'
        finalBlocks = extractContentBlocks(payload.message)
        patchSessionSnapshot -> 设置 pending: false, contentBlocks: finalBlocks
        重置 currentAssistantMessageId = '', currentAccumulatedBlocks = []
```

关键特性：
- 实时累积各种内容块（text/thinking/toolCall）
- 增量更新避免全量重渲染
- 工具调用参数流式累积

### Flow 2: 消息渲染流程

```
WorkbenchMessageStream.vue 消息列表
    │
    ▼
v-for="message in messages" :key="message.id"
    │
    ▼
ChatMessageItem :message="message"
    │
    ▼
根据 message.role 确定布局（user右/assistant左）
    │
    ▼
v-for="(block, index) in message.contentBlocks"
    │
    ├─► block.type === 'text' -> 渲染纯文本
    ├─► block.type === 'image' -> 渲染图片
    ├─► block.type === 'thinking' -> ThinkingCard（可折叠）
    ├─► block.type === 'toolCall' -> ToolCallCard（显示调用参数）
    └─► block.type === 'toolResult' -> ToolCallCard（显示执行结果）
```

关键特性：
- 组件化渲染每种内容类型
- ThinkingCard 支持展开/收起
- ToolCallCard 区分调用和结果状态

### Flow 3: 历史消息加载与扩窗

```
用户进入多轮历史会话
    │
    ▼
GET /api/sessions/:id?rounds=3
    │
    ▼
server 按 user 轮次返回最近 3 轮
    │
    ▼
WorkbenchMessageStream 默认显示已加载窗口中的最近 3 轮
    │
    ├─► 顶部按钮“展开更早 3 轮”
    │      │
    │      ▼
    │   visibleRoundCount += 3
    │      │
    │      ▼
    │   UI 展示已加载窗口内更多旧轮次
    │
    ├─► 顶部按钮“折叠旧轮次”
    │      │
    │      ▼
    │   visibleRoundCount = 3
    │
    └─► 顶部按钮“加载更早历史”
           │
           ▼
       GET /api/sessions/:id?rounds=current+3
           │
           ▼
       server 返回更大的轮次窗口
           │
           ▼
       Vue nextTick 后恢复滚动位置
           │
           ▼
       UI 继续按“已加载轮次”与“当前可见轮次”两层状态渲染
```

关键特性：
- 后端分页单位改为 user 轮次，不再按消息条数 80/400 硬截窗
- 前端默认只显示最近 3 轮，历史折叠是显式交互，不再偷偷隐藏
- “展开更早 3 轮” 只影响当前已加载窗口；“加载更早历史” 才会触发后端请求
- 折叠/展开只影响渲染层，不影响 sessionCache 与服务端快照数据流
