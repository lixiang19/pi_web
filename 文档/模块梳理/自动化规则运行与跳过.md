# 自动化规则运行与跳过

## 职责

自动化模块负责按计划创建普通 AI 会话并发送提示词。它不是 DAG 工作流引擎，不提供条件分支、审批流或自动化专属消息协议。

## 数据结构

- `automation_rules`
  - `automation_id`：规则 ID。
  - `name`：规则名称。
  - `enabled`：是否启用。
  - `scope`：`workspace` 或 `project`。
  - `project_id`：项目自动化绑定的项目 ID。
  - `cwd`：工作空间规则的运行目录；项目规则创建时保存快照，运行时以项目表路径为准。
  - `agent_name`、`explicit_model`、`explicit_thinking_level`：显式运行配置，空值表示使用系统默认。
  - `schedule_json`：每日、每周或间隔计划。
  - `prompt`：发送给 AI 的普通消息。
  - `next_run_at`、`created_at`、`updated_at`：调度和审计时间。
- `automation_runs`
  - `run_id`：运行记录 ID。
  - `automation_id`：所属规则。
  - `status`：`success` / `failed` / `skipped`。
  - `reason`：失败或跳过原因。
  - `session_id`：成功创建的会话 ID。
  - `created_at`：触发时间。
- 启动时会自修复 `automation_runs` 缺表和缺列，并确保 `idx_automation_runs_rule_created` 存在，避免迁移元信息与真实表结构脱节时运行记录不可写。

## 运行规则

- workspace 自动化在服务端工作空间运行。
- server 本地项目自动化在服务端运行。
- 桌面项目自动化必须在项目绑定设备运行。
- 桌面项目设备离线时不转移到其他设备，本次触发写 `skipped` run。
- 归档项目触发时写 `skipped` run。
- 项目不存在、项目类型不允许或桌面调度器不可用时写 `failed` run。
- 成功运行写 `success` run，并保存创建的 `session_id`。

## 会话创建

- 服务端运行：创建普通 session record，应用 agent/model/thinking 选择，写 session index，然后调用 `session.prompt()` 发送提示词。
- 桌面运行：服务端先写轻量 `session_index`，`run_location='desktop'`，再通过 desktop bridge 发送 `create_session` 和 `send_message` 请求。
- 桌面会话的 `session_index.workspace_path` 写项目实际运行路径，用于会话列表和排障定位；服务端不对桌面项目路径做本机 `realpath` 校验。

## 通知契约

- `automation.skipped`：severity=`warning`，related=`automation/<automation_id>`。
- `automation.failed`：severity=`error`，related=`automation/<automation_id>`。
- 通知动作包含打开自动化和重试。
- 通知中心执行自动化重试时重新调用自动化调度逻辑，并写新的 run；如果仍失败或跳过，会生成新的通知。
- `automation.skipped` 即使生产方没有显式写 `actions_json`，通知中心也会派生 `retry` 动作。

## API 与前端

- `GET /api/automations` 返回规则列表和最近运行记录。
- `POST /api/automations` 创建规则。
- `PATCH /api/automations/:automationId` 编辑规则。
- `POST /api/automations/:automationId/toggle` 启停规则。
- `POST /api/automations/:automationId/run` 立即运行并返回本次 run。
- `DELETE /api/automations/:automationId` 删除规则。
- 自动化页展示规则列表、运行上下文、项目绑定、计划、prompt 和运行历史。
- 运行历史中的失败/跳过记录提供直接重试按钮，调用同一个立即运行接口。
