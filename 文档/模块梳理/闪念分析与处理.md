# 闪念分析与处理

## 职责

闪念模块负责把临时捕捉内容保存到 `fleeting_notes`，异步启动不可见内置 `fleeting-agent`，由真实 Agent 直接沉淀为日记、笔记、剪藏、任务、里程碑或正式附件。

## 入口

- `packages/server/src/routes/fleeting.ts`：闪念 CRUD、附件、分析重试和正式处理 API。
- `packages/server/src/fleeting-analysis.ts`：`fleeting.analyze` 入队、worker、内存 Agent 会话和完成工具。
- `packages/server/src/built-in-agents.ts`：内置不可见 `fleeting-agent` 配置。
- `packages/server/src/fleeting-attachments.ts`：闪念临时附件存储、清理和迁移。
- `packages/web/src/composables/useInbox.ts`：闪念列表、轮询、处理动作和重试。
- `packages/web/src/components/workspace/InboxView.vue`：闪念 UI。

## 处理链路

1. 创建闪念后写入 `fleeting_notes`，`analysis_status='unanalyzed'`。
2. 文件/音频附件通过 multipart 上传到临时附件区；旧的 base64 JSON 只保留给兼容 capture API，不作为前端文件/音频默认路径。
3. 临时附件保存到 `<workspaceDir>/.ridge/fleeting-attachments/<noteId>/`，并写入 `fleeting_attachments`。
4. `createFleetingAnalysisRunner().run(noteId)` 入队 `background_jobs.job_type='fleeting.analyze'`。
5. worker 领取 job 后把闪念置为 `analyzing`。
6. worker 读取临时附件正文：Markdown/text/json/yaml 等文本类只读文件前缀；PDF/Word/音频/图片等二进制附件通过 Python Converter 转为 Markdown/转写/OCR，再按总字符预算截断。
7. worker 创建内存 `AgentSession`，使用 `fleeting-agent` 的 system prompt、普通会话同源的记忆/Wiki 注入、Pi 默认 settings/model/auth、工作空间 ResourceLoader、规划工具和 `complete_internal_task` 工具。
8. `fleeting-agent` 是真实 Agent：可读写工作空间、可运行 bash、可使用任务/里程碑规划工具；权限配置只写显式拒绝 `ask: deny` 与 `subagent: deny`。
9. worker 不持久化本次内部会话记录；后台 job 只保存统一完成工具返回的 summary 或错误。
10. Agent 完成实际沉淀动作后调用 `complete_internal_task`：
    - `status='completed'`：`fleeting_notes.status='processed'`，`analysis_status='processed'`，`recommendation_text` 保存处理摘要。
    - `status='failed'`：`analysis_status='failed'`，`last_error` 保存错误，闪念保持未处理以便用户查看/重试。
11. Agent 退出但未调用完成工具时，worker 视为失败并进入后台 job 重试/最终失败路径。

## 内置 Agent 配置

```yaml
name: fleeting-agent
description: 自动处理闪念
display_name: Fleeting Agent
mode: all
visible: false
enabled: true
inherit_context: false
permission:
  ask: deny
  subagent: deny
```

`visible:false` 同时表示：

- `/api/agents` 不返回。
- 普通主会话不能选择。
- 普通 Agent 不能通过 `subagent` 工具调用。
- 服务器内部 `fleeting.analyze` worker 可以按名称启动。

## 失败与重试

- Agent 运行异常、未调用完成工具或完成工具参数非法时，未耗尽次数则回到 `unanalyzed`，写入 `last_error` 和 `retry_count`。
- 最终失败时置为 `failed`，后台任务队列负责失败通知。
- Agent 主动调用 `complete_internal_task(status='failed')` 时记录错误并完成 job，不自动重试。
- `POST /api/fleeting/:noteId/analyze` 会取消旧 job、清空错误并重新入队。

## 处理类型

- `journal`：追加到今日日记。
- `clip`：保存到 `剪藏/<标题>.md`，创建 `clips`，并触发 RAG。
- `task`：创建 `workspace_tasks`。
- `milestone`：创建 `workspace_milestones`。
- `attachment`：仅迁移闪念附件为正式附件。
- `delete`：仍保留 HTTP 删除入口；自动处理是否删除由 `fleeting-agent` 真实动作决定。
