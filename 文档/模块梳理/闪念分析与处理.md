# 闪念分析与处理

## 职责

闪念模块负责把临时捕捉内容保存到 `fleeting_notes`，异步给出处理建议，并在用户确认后沉淀为日记、剪藏、任务、里程碑或正式附件。

## 入口

- `packages/server/src/routes/fleeting.ts`：闪念 CRUD、附件、分析重试和正式处理 API。
- `packages/server/src/fleeting-analysis.ts`：`fleeting.analyze` 入队、worker 和后台 AI 分析。
- `packages/server/src/fleeting-attachments.ts`：闪念临时附件存储、清理和迁移。
- `packages/web/src/composables/useInbox.ts`：闪念列表、轮询、处理动作和重试。
- `packages/web/src/components/workspace/InboxView.vue`：闪念 UI。

## 分析链路

1. 创建闪念后写入 `fleeting_notes`，`analysis_status='unanalyzed'`。
2. 文件/音频附件通过 multipart 上传到临时附件区；旧的 base64 JSON 只保留给兼容 capture API，不作为前端文件/音频默认路径。
3. 临时附件保存到 `<workspaceDir>/.ridge/fleeting-attachments/<noteId>/`，并写入 `fleeting_attachments`。
4. `createFleetingAnalysisRunner().run(noteId)` 入队 `background_jobs.job_type='fleeting.analyze'`。
5. worker 领取 job 后把闪念置为 `analyzing`。
6. worker 读取临时附件正文：Markdown/text/json/yaml 等文本类只读文件前缀；PDF/Word/音频/图片等二进制附件通过 Python Converter 转为 Markdown/转写/OCR，再按总字符预算截断。
7. worker 用 `createPiDefaultSettingsManager(workspaceDir)` 读取 Pi 默认模型设置，优先使用 `settings.json` 的 `defaultProvider/defaultModel/defaultThinkingLevel`，并通过 `ModelRegistry` 复用 `models.json/auth.json` 中配置好的服务商和鉴权。
8. worker 直接调用 `@mariozechner/pi-ai complete(model, context, options)`，不创建 `AgentSession`，不进入 `ResourceLoader/AGENTS/context/skills/prompt templates/extensions`。
9. `context.systemPrompt` 是 ridge 闪念分析器提示，`context.tools` 只包含 `fleeting_analysis_result`，user prompt 包含闪念内容、附件列表和截断后的附件正文。
10. 模型必须调用该工具返回 `recommendationType/recommendationText/draft/requiresInput`；普通 JSON 文本不再被接受。
11. 服务端从 assistant `toolCall.arguments` 校验结构并写回 `fleeting_notes`，成功状态为 `suggested`。

## 失败与重试

- 分析失败未耗尽次数时回到 `unanalyzed`，写入 `last_error` 和 `retry_count`。
- 最终失败时置为 `failed`，后台任务队列负责失败通知。
- `POST /api/fleeting/:noteId/analyze` 会取消旧 job、清空错误并重新入队。

## 处理类型

- `journal`：追加到今日日记。
- `clip`：保存到 `剪藏/<标题>.md`，创建 `clips`，并触发 RAG。
- `task`：创建 `workspace_tasks`。
- `milestone`：创建 `workspace_milestones`。
- `attachment`：仅迁移闪念附件为正式附件。
- `delete`：由前端按建议执行删除，不是后台自动删除。
