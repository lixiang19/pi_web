# 后台任务队列

## 职责

后台任务队列负责调度系统维护任务和后台 Agent，避免阻塞主对话。

## 入口

- `packages/server/src/background-jobs.ts`
- 数据表：`background_jobs`
- 失败通知表：`notification_events`
- 诊断入口：`GET /api/workspace/knowledge/diagnostics`

## 能力

- `enqueue` 创建任务。
- `claimNext` 领取可运行任务。
- `complete` 标记完成。
- `fail` 记录失败、有限重试，达到上限后标记失败。
- `complete` 和 `fail` 只接受 `running` 任务，避免 pending 任务被绕过领取流程。

## 并发规则

- `memory.maintain` 全局串行。
- `summary.daily` 按 `relatedType=daily` 和 `relatedId` 串行。
- `task.review` 按 `relatedType=workspace` 和 `relatedId=workspaceDir` 去重，避免同一工作空间同时跑多次任务回顾。
- 同一个 `job_type + related_type + related_id` 的活跃任务去重。

## 重试规则

- 失败后按退避时间写入 `run_after` 和 `next_retry_at`。
- 未到 `next_retry_at` 的任务不会被 `claimNext` 领取。
- `retry_count` 达到 `max_attempts` 后进入 `failed`。
- `notify_on_failure=1` 且最终失败时写入 `notification_events`。

## 已接入任务类型

- `summary.daily`：会话结束后写 daily。
- `memory.maintain`：daily L1 Atom 后维护 L2 Scenario 与 L3 `MEMORY.md`。
- `fleeting.analyze`：闪念创建后异步分析，只开放 `fleeting_analysis_result` 自定义工具，成功后把结构化建议写回 `fleeting_notes`。
- `task.review`：手动或定期扫描任务、里程碑、处理会话索引和最近 daily，输出 `task_review.suggestion` 通知；只写建议，不直接改正式任务或里程碑。
- `rag.index`：RAG 索引刷新。
- `file.convert`：文件转换。

## 用户可见诊断

V2 阶段 5 起，知识诊断入口会读取 `background_jobs`，返回：

- 按 `pending/running/completed/failed/cancelled` 聚合的任务数量。
- 按 `job_type` 拆分的状态数量。
- 最近失败任务、错误、关联对象和关联通知。

诊断只读，不直接重跑后台任务；具体重试仍走通知中心动作或领域端点。
