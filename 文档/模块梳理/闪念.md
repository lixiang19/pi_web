# 闪念

## 职责

闪念是临时捕获队列。Web 端和桌面端统一叫“闪念”，不再使用“收件箱”概念。

- 闪念 DB 只保存未处理闪念。
- 处理成功后删除原闪念记录。
- 采集时不选择项目、不分类、不打标签。
- 闪念 Agent 只生成一个最佳处理建议，不自动执行。
- 固定处理按钮始终可用，不依赖 AI 分析结果。

## 入口

- API: `GET/POST/PATCH/DELETE /api/workspace/fleeting`
- 前端: `InboxView.vue`（内部仍用 Inbox 命名，UI 统一叫闪念）
- DB 表: `fleeting_notes`、`fleeting_attachments`

## 状态机

### 主状态

- `pending`: 待处理（默认）
- `processing`: 处理中

### 分析状态

- `unanalyzed`: 未分析（默认）
- `analyzing`: 分析中
- `suggested`: 已建议
- `failed`: 分析失败（不打扰用户，固定按钮仍可用）

## 字段

见 `数据库与迁移.md` 闪念表字段。

## 附件

  - 临时附件目录: `~/ridge-workspace/.ridge/fleeting-attachments/<noteId>/`
  - 正式附件目录: `~/ridge-workspace/附件/`
  - 未处理和临时附件不进 RAG、不进图谱、不可被 workspace MCP 读取。
  - 删除闪念时执行临时附件清理检查（安全检查路径边界，禁止越界删除）。
  - 处理成功后，临时附件迁移到正式附件目录，保留原文件名；冲突时循环追加短 ID 去重直到唯一。
  - 迁移中途失败会删除已复制的正式文件，并将 DB `final_path` 重置为 NULL，保留 note 和临时附件。
  - 只有目标系统确认创建成功后，才迁移附件并删除原闪念。
  - 处理失败保留原闪念和临时附件。

## API 行为

- `POST /api/workspace/fleeting`: 支持 JSON `content` 纯文字，也支持 multipart/form-data `content` + `attachments`（兼容 `files`）。
- 允许“有附件但文字为空”的文件/截图闪念；无文字且无附件仍拒绝。
- 上传文件副本保存到 `.ridge/fleeting-attachments/<noteId>/`，DB 记录引用关系。
- 返回 note 时包含 attachments 数组/元数据。
- 创建后默认 `status=pending`、`analysis_status=unanalyzed`、`type=text`（无附件）或 `type=file`（有附件）。
- 不写 `search_index_status`，不进 RAG。
- `PATCH /api/workspace/fleeting/:id`: 支持 content/status/analysisStatus/suggestion 更新，验证枚举。
- 分析失败只改 `analysisStatus`/`suggestion`/`updatedAt`，不改变主 `status`。
- `DELETE`: 删除原记录 + 附件 DB 记录 + 临时目录清理。
- process journal/clip 成功后：先创建目标（日记/剪藏），再迁移临时附件到 `附件/`，最后删除原闪念和临时目录。
- task process 未接入，保留原闪念和临时附件。

## 前端行为

- 全局保存按钮保存后广播 `ridge:fleeting-created` 事件。
- 轮询只对 `unanalyzed/analyzing` 状态执行；`suggested/failed` 不轮询。
- 固定处理按钮（日记/任务/剪藏/删除）始终可点击；"按建议处理"仅在 `suggested` 时启用。
- `failed` 状态显示“分析失败”，不弹扰用户。
- Web 附件上传入口：真实 file input，选择后真实上传（multipart/form-data）。
- 列表展示附件数量/名称。
- 保存成功清空附件选择；失败保留附件。
- 按钮允许“只有附件无文字”。

## 边界

- 旧 `/api/fleeting` 路径保留兼容（与 `/api/workspace/fleeting` 同时挂载）。
- 测试数据库 bootstrap SQL 必须与生产 migration 字段一致。
