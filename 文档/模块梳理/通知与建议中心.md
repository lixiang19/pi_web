# 通知与建议中心

## 职责

通知与建议中心承接后台任务、文件处理、RAG 和后续 Agent 产生的用户待处理事件。它不是聊天助手，也不进入 RAG。

## 数据表

- 表：`notification_events`
- 关键字段：
  - `event_id`：通知 ID。
  - `event_type`：来源事件类型，例如 `file_processing.convert_failed`、`rag.index_failed`、`background_job.failed`、`task_review.suggestion`。
  - `source`：来源模块，例如 `file_processing`、`rag`、`background_jobs`。
  - `severity`：`info` / `warning` / `error`。
  - `status`：`unread` / `pending` / `handled` / `dismissed` / `failed`。
  - `related_type`、`related_id`：关联对象，用于打开文件、任务、会话等。
  - `actions_json`：动作 schema，至少包含 `id` 与 `label`。
  - `payload_json`：事件载荷；建议事件通过 `payload_json.suggestion` 描述正式写入动作。
  - `created_at`、`updated_at`、`read_at`、`handled_at`。

## API

- `GET /api/notifications?filter=unhandled|all|failed|suggestions|handled`
  - 默认 `unhandled`。
  - `unhandled` 默认排除 `info` 类型；信息类只在 `all` 中可查，避免主动打扰。
  - 返回 `notifications` 和各筛选计数。
  - 后端会为旧数据补推导 `type/source/related/actions`，但新写入应显式写字段。
  - `related` 回退推导只按 `event_type` 白名单读取 payload：文件/RAG 读取 `filePath`，后台任务读取 `jobId`，任务回顾更新建议读取目标任务或里程碑 ID。
- `POST /api/notifications/:eventId/actions`
  - 请求体：`{ "actionId": "..." }`。
  - 支持：
    - `dismiss`：写 `dismissed`。
    - `mark_handled`：写 `handled`。
    - `accept_suggestion`：执行 `payload_json.suggestion` 后写 `handled`。
    - `reject_suggestion`：写 `dismissed`。
    - `retry`：按事件类型重试。
    - `open_related`：只返回当前通知，前端负责导航。
- 动作接口只允许执行当前通知派生出的可用动作，避免客户端直接 POST 隐藏动作。
- 通知中心服务端拥有安全生命周期动作：非终态通知默认补充 `mark_handled` 和 `dismiss`；生产方动作负责领域语义，例如 `retry`、接受/拒绝建议或打开关联对象。

## 建议写入契约

接受建议时才写正式对象。

支持的 `payload_json.suggestion.kind`：

- `task.update`：调用任务更新逻辑。
- `task.create`：创建任务。
- `milestone.update`：调用里程碑更新逻辑。
- `milestone.create`：创建里程碑。

建议 patch 仍走任务系统的状态机与 actor 约束。Agent 建议不能绕过任务完成确认边界。
非法或缺失的 `payload_json.suggestion` 会返回错误，并保持通知状态不变。
建议应用和通知标记已处理在同一数据库事务中完成，避免正式对象已写入但通知仍待处理。

## 失败重试契约

- `file_processing.convert_failed` / `file_processing.index_failed`
  - 将 `file_processing_status` 从失败态恢复为 `pending`。
  - 取消旧 `file.convert` 待运行/失败任务并重新入队。
  - Python 转换服务未启用、后台队列不可用或找不到失败状态记录时返回错误，不更新通知状态。
  - 状态恢复、取消旧任务和重新入队在同一数据库事务中提交。
- `rag.index_failed`
  - 将 `search_index_status` 恢复为 `pending`，清空错误。
  - 取消旧 `rag.index` 待运行/失败任务并重新入队。
  - 后台队列不可用或找不到失败状态记录时返回错误，不更新通知状态。
  - 状态恢复、取消旧任务和重新入队在同一数据库事务中提交。
- `background_job.failed`
  - 将对应 `background_jobs` 记录恢复为 `pending`，清空锁与错误。
  - 找不到 failed 状态的后台任务时返回错误，不更新通知状态。

重试成功提交后通知标记为 `handled`；真正处理结果由对应 worker 继续写状态，若再次最终失败会生成新通知。

## 前端

- 左侧固定入口「通知」打开 `feature:notifications` 单例标签。
- 左侧入口显示未处理数。
- `NotificationCenterView.vue` 负责筛选、列表、动作按钮和关联对象导航。
- 动作按钮在请求期间按 `notificationId:actionId` 禁用，避免重复提交。
- 文件关联对象发出 `openFile`，任务/里程碑发出 `openTasks`，会话发出 `openSession`，项目发出 `openProject`，自动化发出 `openAutomation`。
- 不支持打开详情页的关联类型（例如 `background_job`）不返回 `open_related` 动作，避免按钮无响应。
