# 工作空间菜单路由重构方案

## 任务背景

用户要求进行一次大型信息架构调整：删除当前菜单路由体系，把终端、自动化、设置等功能收敛到工作空间内部，通过工作空间顶部系统菜单打开，并以工作空间标签页承载。旧的底部聊天和项目入口删除，AI 会话后续在工作空间主页标签中启动。

本方案基于 scout 代码探索、pm 产品审查和用户确认后的产品决策编写。

## 已确认决策

1. **路由策略**：只保留根路由/工作空间路由，`terminal`、`automations`、`settings`、`chat`、`search`、`spaces`、`datasets` 等独立页面路由全部移除，不做旧路由重定向兼容。
2. **顶部菜单**：工作空间顶部新增独立系统菜单行，承载 `终端 / 自动化 / 设置`，`AI 会话`后续在工作空间主页标签承载。
3. **标签承载**：终端、自动化、设置都作为工作空间 SplitGrid 标签页打开。
4. **终端模型**：允许多个工作空间终端标签页；每个标签页绑定一个终端实例；终端内部不再显示二级多标签栏。
5. **聊天与项目**：旧侧栏聊天入口和项目入口删除。本轮不实现项目选择能力；后续工作空间内支持引入项目后再设计。
6. **设置入口**：设置不放侧边栏，不作为独立路由；与终端、自动化一样点击后打开工作空间标签页。

## 当前架构理解

### 现状入口链路

```text
main.ts
  -> App.vue
    -> router/index.ts
      -> PlatformShell.vue
        -> WorkbenchSidebar.vue
        -> RouterView
          -> /chat WorkbenchPage.vue
          -> /workspace WorkspacePage.vue
          -> /terminal TerminalPage.vue
          -> /automations AutomationsPage.vue
          -> /settings SettingsPage.vue
```

### 关键现有模块

- `src/router/index.ts`：定义全局路由与 `PlatformShell` 子路由。
- `src/layouts/PlatformShell.vue`：旧全局布局壳，左侧 `WorkbenchSidebar` + 右侧 `RouterView`。
- `src/components/workbench/WorkbenchSidebar.vue`：旧侧栏主导航、聊天会话树、项目入口、设置入口。
- `src/composables/useWorkbenchPrimaryNavigation.ts`：旧一级导航和聊天会话打开逻辑。
- `src/pages/WorkspacePage.vue`：工作空间主页面，已有文件树、固定视图入口、SplitGrid 分屏标签系统。
- `src/composables/useSplitPanes.ts`：工作空间标签页和分屏状态引擎，当前标签类型为 `home/view/file`。
- `src/pages/TerminalPage.vue`：旧终端独立页面，内部通过 `TerminalTabBar` 提供二级多终端标签。
- `src/composables/useTerminalPool.ts`：终端实例池。
- `src/pages/AutomationsPage.vue`：旧自动化独立页面。
- `src/pages/SettingsPage.vue`：旧设置独立页面。

## 目标信息架构

```text
App.vue
  -> WorkspacePage.vue
    ├── 工作空间顶部系统菜单
    │   ├── 终端：打开 terminal 标签页
    │   ├── 自动化：打开 automation 标签页
    │   └── 设置：打开 settings 标签页
    ├── 工作空间左侧内容导航/文件树区域
    └── SplitGrid 分屏标签区
        ├── home 标签页
        ├── view 标签页
        ├── file 标签页
        ├── terminal 标签页
        ├── automation 标签页
        └── settings 标签页
```

## 改动点

### 1. 路由与全局壳

- 精简 `router/index.ts`，只保留根路径进入工作空间。
- 移除 `PlatformShell` 对左侧全局菜单和 `RouterView` 子页面的承载职责。
- 删除旧独立页面路由：
  - chat
  - terminal
  - automations
  - settings
  - search
  - spaces
  - datasets
- 不实现旧路径重定向兼容。

### 2. 工作空间顶部系统菜单

- 在 `WorkspacePage.vue` 或拆分的新组件中增加顶部系统菜单行。
- 菜单动作不调用 vue-router。
- 菜单动作直接调用工作空间标签页打开能力。
- 推荐拆分新组件：`WorkspaceTopMenu.vue`，避免 `WorkspacePage.vue` 继续膨胀。

### 3. SplitGrid 标签类型扩展

- `useSplitPanes.ts` 的 `SplitTabItem.kind` 增加：
  - `terminal`
  - `automation`
  - `settings`
- 为系统标签提供稳定标题、图标和内容渲染分支。
- 保持原有 `home/view/file` 行为不变。

### 4. 终端 workspace 化

- 从旧 `TerminalPage.vue` 中抽出可嵌入工作空间标签页的终端内容组件。
- 工作空间每打开一个 `terminal` 标签页就创建一个终端实例。
- `terminal` 标签内容只渲染对应终端的 `TerminalViewport` 和必要操作区。
- 不再渲染 `TerminalTabBar`。
- 终端关闭、重启、重命名等能力如需要保留，应落在当前工作空间标签页自己的操作区，而不是二级标签栏。
- 关闭工作空间终端标签页时，按终端池接口关闭对应终端实例。

### 5. 自动化 workspace 化

- 将 `AutomationsPage.vue` 的规则列表与编辑器拆成可嵌入的工作空间标签页组件。
- 顶部菜单点击“自动化”打开或激活 `automation` 标签页。
- 自动化 CRUD 继续使用现有 `useAutomations` 与后端 API。
- `runNow` 不再通过旧 `useWorkbenchPrimaryNavigation.openChatSession()` 跳转 `/chat`。
- 本轮在 AI 会话新入口未实现前，立即运行只完成服务端创建动作和前端状态反馈，不打开旧聊天路由。

### 6. 设置 workspace 化

- 将 `SettingsPage.vue` 内容拆成可嵌入的设置标签页组件。
- 顶部菜单点击“设置”打开或激活 `settings` 标签页。
- 设置持久化继续使用现有 Pinia settings store 和 `/api/storage/settings`。
- 删除旧独立设置路由入口。

### 7. 旧侧栏、聊天、项目入口删除

- 删除页面上旧 `WorkbenchSidebar` 渲染。
- 删除旧底部聊天入口、会话树入口、项目入口和 worktree 操作入口。
- `useWorkbenchPrimaryNavigation` 不再作为路由跳转中心；如仍有必要保留，应只保留与新工作空间标签页动作有关的最小职责。
- 自动化、设置、终端不得继续依赖旧导航 composable。

### 8. 测试与文档

- 已新增测试 spec：`文档/spec/工作空间菜单路由重构.spec.md`。
- 执行阶段必须先根据 spec 更新/新增测试，再实现代码。
- 完成后更新涉及模块梳理文档，至少包括：
  - `文档/模块梳理/工作空间模块.md`
  - `文档/模块梳理/左侧统一导航与会话菜单模块.md`
  - `文档/模块梳理/定时会话自动化模块.md`
  - `文档/模块梳理/Web工作台与Pi服务桥接模块.md`
  - 必要时更新 `文档/记忆/MEMORY.md`

## 风险

1. **终端生命周期风险**：终端从独立页面迁入工作空间标签页后，标签关闭、切换、分屏移动都可能影响 WebSocket 生命周期。
2. **自动化结果入口风险**：旧 `runNow` 会打开聊天会话；本轮删除聊天路由后必须明确停止依赖旧跳转，后续再接入主页 AI 会话入口。
3. **路由删除风险**：大量测试和组件可能引用旧路由名，删除后会出现编译错误。
4. **WorkspacePage 膨胀风险**：不能把所有迁移逻辑直接堆入 `WorkspacePage.vue`，需要拆组件。
5. **旧导航遗留风险**：`useWorkbenchPrimaryNavigation`、`WorkbenchSidebar`、旧页面文件如果未彻底断开，会产生不可见的旧架构依赖。
6. **设置样式适配风险**：设置页从独立页面进入标签页后，宽度、高度、滚动容器需要适配工作空间标签内容区。

## 回退策略

本轮不做旧路由兼容，因此回退策略不是代码内兼容，而是版本级回退：

1. 使用 Git 提交边界回退整次变更。
2. 执行前将实现任务拆为可独立验证的 commit/patch：
   - SplitGrid 类型扩展
   - 终端迁移
   - 自动化迁移
   - 设置迁移
   - 路由和旧侧栏删除
   - 测试与文档更新
3. 若某阶段失败，只回退该阶段实现，不在代码中保留双架构。
