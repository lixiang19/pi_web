# 16 闪念处理为正式对象

## 目标

用户确认后把闪念处理为正式对象：任务、里程碑、剪藏、正式附件。

## 范围

- 处理为任务（task）
- 处理为里程碑（milestone）
- 处理为剪藏（clipping）
- 处理为正式附件（attachment）
- 可在处理时选择项目
- 处理成功后删除原闪念
- 附件迁移到正式附件

## 不做

- 不在采集阶段选择项目
- 不允许 AI 建议自动执行

## API

- POST /api/workspace/fleeting/:id/process

### 目标类型

- task
- milestone
- clipping
- attachment

### 行为

- 处理动作必须由用户触发
- 创建目标对象和附件迁移必须在同一成功边界内（事务）
- 成功后删除闪念记录
- 失败时不删除闪念和临时附件

## 测试

- 处理为任务创建任务并删除闪念
- 处理为附件迁移文件到 `附件/`
- 目标创建失败时闪念仍存在
- 处理时可选择项目

## 实现方案

### 后端

1. `routes/fleeting.ts` 中把 `process/task` 从占位改为真实实现
2. 新增 `process/milestone`、`process/clipping`、`process/attachment` 路由
3. 所有 process 路由在事务边界内：先迁移附件，再创建目标对象，最后删除闪念
4. 任务/里程碑支持 `projectId` 参数

### 前端

1. `lib/api.ts` 新增 `processFleetingToMilestone`、`processFleetingToAttachment`，更新 `processFleetingToTask` 签名
2. `composables/useInbox.ts` 新增 `processToMilestone`、`processToAttachment`，更新 `processToTask`
3. `InboxView.vue` 为闪念卡片提供处理操作 UI（选择目标类型、选择项目）

### 事务边界

- 使用 better-sqlite3 的 `db.transaction()` 包裹：
  1. 创建目标对象（INSERT）
  2. 删除闪念（DELETE FROM fleeting_notes）
- 附件迁移在事务外先执行，如果迁移失败则不进入事务
- 如果目标创建失败（抛出异常），事务回滚，闪念保留

## 验收

- 处理为任务时，创建 workspace_tasks 记录并删除闪念
- 处理为里程碑时，创建 workspace_milestones 记录并删除闪念
- 处理为剪藏时，创建 clips 记录并删除闪念（已有逻辑兼容）
- 处理为附件时，仅迁移附件到正式目录并删除闪念
- 失败时闪念和临时附件均保留
- 可选择项目（task、milestone）
- npm run check 通过
- 后端测试 `fleeting-api.test.ts` 覆盖全部 4 种处理路径
