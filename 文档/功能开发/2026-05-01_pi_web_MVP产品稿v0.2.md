# pi_web MVP 产品稿 v0.2

> 任务：梳理 pi_web 产品定位与 MVP 闭环  
> 日期：2026-05-01  
> 角色：@产品  
> 评审重点：MVP 边界、端到端验收、对象模型复杂度、可转 spec/测试  
> v0.2 更新：采纳开发侧与审查反馈，将 Phase 1 收敛为“创建目标 + 打开会话 + 可回找”，暂不承诺完整验收/沉淀，并明确 Phase 1 只改 `DashboardView` 作为目标输入入口。

## 1. 一句话定位

`pi_web` / `ridge` 是面向个人用户的本地 AI 工作空间：把笔记、待办、文件、会话和 Agent 协作收敛到同一个项目空间里，让用户能把一个目标交给 AI 拆解、执行、追踪，并最终验收沉淀结果。

更短版本：**一个个人用户每天打开的“人类 + AI 协作工作台”。**

## 2. MVP 目标用户

### 主用户

先锁定：**有持续产出目标的个人创作者 / 独立开发者 / 研究学习者**。

共同特征：
- 每天有多个想法、资料、待办和 AI 对话散落在不同地方。
- 希望 AI 不只是聊天，而是围绕一个目标持续推进。
- 愿意在本地工作空间里沉淀上下文、文件和结果。

### MVP 不同时服务的人群

- 不先做团队协作。
- 不先做企业项目管理。
- 不先做完整 Notion / Obsidian / Jira 替代品。
- 不先做移动端碎片化场景，保留为后续愿景。

## 3. 核心问题

当前代码库已有工作空间、文件树、仪表盘、待办、收件箱、终端、自动化、Pi session/agents 等能力。最缺的不是更多功能，而是一个清晰入口：

**用户进入首页后，立刻知道自己要把什么目标交给 AI，并且之后能从首页或待办里回到这次 AI 协作。**

因此 MVP 先把已有能力收敛成一个闭环，而不是扩展成完整功能平台。

## 4. 首页第一屏

### 第一屏要完成的唯一协作动作

**把一个目标创建成工作空间里的“协作任务”，并启动一个带上下文的 AI 会话来推进它。**

### 第一屏信息架构

入口原则：**Phase 1 只改 `DashboardView` 作为目标输入入口**。不要同时维护 `HomePage` 搜索框和 `DashboardView` 目标框两套入口。

判定：
- `DashboardView` 承载目标输入、最近目标、最近 AI 会话入口。
- `HomePage` 搜索框本阶段不改、不新增目标提交逻辑。
- 根路由进入后的第一屏如果仍包含 `HomePage` 标签，Phase 1 不扩大到主页/分屏结构重构；只保证用户进入工作空间后能从固定视图 `DashboardView` 完成目标到会话链路。

第一屏建议包含 4 块：

1. **开始推进**
   - 主输入框：`今天你想让 AI 一起推进什么？`
   - 支持输入一句目标，例如：`帮我整理 pi_web 的 MVP 产品稿`。
   - CTA：`创建协作任务`。

2. **最近目标**
   - 展示最近创建的 `kind=goal` 任务。
   - Phase 1 只要求可回找、可打开关联会话，不引入 `needs_review` 状态。

3. **AI 正在推进**
   - 展示最近创建或打开的 Pi session / Agent 任务。
   - Phase 1 先展示“最近会话 + 入口”，不做实时运行任务中心。

4. **今日上下文**
   - 最近闪念、今日待办、今日日记、最近文件。
   - 作用是告诉 AI 和用户：当前工作从哪里接续。

## 5. MVP 核心闭环

### 闭环路径

1. 用户在首页输入一个目标。
2. 系统创建一个协作任务，并把它放入当前工作空间。
3. 系统启动一个 Pi 会话，把目标、工作空间路径和必要上下文传给 AI。
4. 系统自动发送第一条用户 prompt，内容至少包含目标文本，避免只打开空会话。
5. 用户可从目标记录回到同一个会话继续推进。
6. 后续 Phase 2 再补“待我确认 / 验收状态”。
7. 后续 Phase 3 再补“结果沉淀为笔记 / 文件 / 待办”。

### MVP 成功定义

一个新用户不需要理解所有模块，只要能完成：

**创建目标 → 打开带上下文的 AI 会话 → 刷新后可回找 → 下次继续。**

## 6. 对象模型 v0.2

为避免过度抽象，MVP 只定义 6 个对象，并尽量贴合现有实现。

| 对象 | 含义 | MVP 承载方式 |
| --- | --- | --- |
| Workspace | 用户本地工作空间根目录 | 现有 workspaceDir / 文件树 |
| Goal | 用户想推进的一句话目标 | 复用 Task 作为 canonical record，不新增 Goal 表 |
| Task | 可追踪的行动项 / 验收项 | 现有待办系统，支持独立任务和 markdown checkbox |
| Session | 围绕目标运行的 Pi 会话 | 现有 Pi session / chat 链路 |
| Agent | 执行某类工作的 AI 角色 | 现有 Pi agent / subagent 能力 |
| Artifact | AI 或用户产生的可验收结果 | Phase 1 不建模；Phase 3 再处理文件、笔记、会话输出、待办完成状态 |

### TaskItem 最小扩展建议

Phase 1 以现有 Task 作为 `Goal` 的 canonical record。为避免只靠标题匹配会话，建议对 `.ridge/tasks.json` 中的 `TaskItem` 做最小可选字段扩展：

```ts
type TaskItem = {
  // existing fields...
  kind?: "goal" | "task";
  sessionId?: string;
  source?: "dashboard";
};
```

字段含义：
- `kind="goal"`：表示该任务来自首页目标输入，是一个协作目标。
- `sessionId`：关联创建出的 Pi session，支持刷新后从目标回到会话。
- `source="dashboard"`：记录来自 `DashboardView` 目标输入，便于首页筛选最近目标。

### 关键关系

- 一个 Workspace 有多个 Goal / Task / Session / Artifact。
- 一个 Goal 本质上是一条 `kind=goal` 的 Task。
- 一个 Goal 通过 `sessionId` 关联一个 Session。
- Artifact / 验收状态不进入 Phase 1 对象模型。
- Agent 不在 MVP 中作为独立管理对象暴露给用户，先作为 Session 内的执行能力出现。

### 兼容规则

- 未带 `kind` 的历史 Task 一律按普通任务处理。
- `kind="goal"` 只影响首页最近目标筛选和“回到会话”入口。
- 如果 goal task 的 `sessionId` 已失效、对应 session 被删除或不可用，应给出降级提示，不自动新建 session 并覆盖旧关联。

### 暂不引入

- 不新增 Project / Space / Objective / Run / Job 等多层抽象。
- 不做复杂任务依赖图。
- 不做多 Agent 编排 UI。
- 不做完整审计日志。

## 7. MVP 页面与动作范围

### 必做页面 / 区域

1. **DashboardView / 工作空间仪表盘**
   - 目标输入入口。
   - 最近目标。
   - 最近 AI 推进。
   - 今日上下文。
   - 注意：Phase 1 只在 `DashboardView` 保留目标输入入口，不改 `HomePage` 搜索框。

2. **工作空间主界面**
   - 文件树。
   - 多标签 / 分屏。
   - 文件或笔记打开。

3. **会话区**
   - 从目标启动 Pi 会话。
   - 展示 AI 输出。
   - 支持继续追问。

4. **待办 / 目标列表**
   - 展示由首页创建的 `kind=goal` 任务。
   - 支持点击目标回到关联会话。
   - Phase 1 不做 AI 交付验收状态。

5. **收件箱**
   - 捕捉闪念。
   - 可作为目标或上下文来源。

### MVP 暂不做

- 团队成员、权限、共享空间。
- 移动端专属交互。
- 可视化 Agent 编排器。
- 完整自动化中心 / cron 配置增强。
- 数据库 Bases 的复杂视图增强。
- 全局搜索、知识图谱、长期记忆图谱。
- 多工作空间云同步。

## 8. 关键用户故事

### Story 1：从首页创建目标

作为个人用户，我第一次打开 `pi_web`，可以在首页输入一个目标，并创建一个协作任务，让 AI 开始推进。

验收：
- 首页有清晰主输入框和 CTA。
- 输入目标后生成可追踪记录。
- 创建后打开或关联一个 AI 会话。
- 系统自动发送第一条包含目标文本的用户 prompt。
- 用户能回到首页看到该目标或最近推进记录。

### Story 2：AI 围绕工作空间上下文推进

作为个人用户，我希望 AI 启动时知道当前工作空间和目标，而不是一个孤立聊天。

验收：
- 会话创建时带上当前 workspaceDir / 目标文本。
- 会话标题或上下文区域能反映该目标。
- 会话消息里能看到自动发送的目标 prompt。
- AI 输出与目标相关，而不是泛聊天空态。

### Story 3：刷新后回找目标与会话

作为个人用户，我希望刷新或第二次进入后，还能从首页或待办找到之前创建的目标，并继续同一个 AI 会话。

验收：
- 首页或待办区展示最近创建的 goal task。
- goal task 显示目标标题和关联会话入口。
- 点击 goal task 能打开关联 session。
- 刷新后 `sessionId` 关联仍存在。

### Story 4：用户继续追问同一目标

作为个人用户，我希望回到目标后不是重新开始聊天，而是围绕原 session 继续追问。

验收：
- 打开 goal task 后进入对应 session。
- 用户发送新消息时沿用该 session 的上下文。
- 会话标题或入口能反映目标文本。

## 9. 可测验收标准 v0.2

### A. 首页第一屏

- [ ] 进入根路由后默认看到工作空间首页，而不是孤立聊天页。
- [ ] 首页第一屏能回答三件事：我现在可以推进什么、最近创建了哪些目标、如何回到最近 AI 会话。
- [ ] `DashboardView` 有且只有一个主输入入口，用于创建个人目标 / 协作任务。
- [ ] `HomePage` 搜索框不新增目标提交逻辑，避免与 `DashboardView` 出现两套目标入口。
- [ ] 空状态下有明确引导文案，不依赖用户先理解文件树和会话。

### B. 创建目标到会话

- [ ] 用户输入目标并提交后，系统创建一条 `kind=goal` 的 Task 记录。
- [ ] 系统启动或打开一个与该目标关联的 Pi 会话，并把 `sessionId` 写回 goal task。
- [ ] 会话拥有当前工作空间上下文，不回退到虚假默认 cwd。
- [ ] 系统自动发送第一条用户 prompt，目标文本能在会话消息中看到。
- [ ] 刷新后目标记录和会话入口仍可找到。

### C. 回找目标与会话

- [ ] 刷新后用户能在首页或待办区看到刚创建的 goal task。
- [ ] 点击 goal task 能打开对应 session，而不是创建新 session。
- [ ] 如果关联 session 不可用，展示降级提示，不自动新建并覆盖 `sessionId`。
- [ ] 用户可以在对应 session 内继续追问。
- [ ] goal task 与普通 task 在列表中可区分，至少内部数据可区分。
- [ ] 未带 `kind` 的旧 task 仍按普通 task 展示与操作。

### D. Phase 1 不承诺项

- [ ] 不要求展示 AI 交付后的待确认状态。
- [ ] 不要求把会话结果保存为笔记 / 文件。
- [ ] 不要求 Artifact 关系建模。
- [ ] 不要求新增 `needs_review` 状态。

### E. 范围边界

- [ ] 不要求实现团队协作。
- [ ] 不要求实现多 Agent 编排 UI。
- [ ] 不要求实现自动化运行记录中心。
- [ ] 不要求实现数据库 Bases 新能力。
- [ ] 不要求重构 Tauri，Tauri 仍只作为桌面壳。

## 10. 开发拆分建议

### Phase 1：创建目标 + 打开会话 + 可回找

目标：不用扩展大模型能力，先打通首页创建目标到会话的入口，并保证刷新后可回找。

可能任务：
- 明确 `DashboardView` 为 Phase 1 唯一目标输入入口，避免改动 `HomePage` 搜索框。
- 新增“目标输入 → 创建 `kind=goal` Task → 创建/打开 session → 写回 `sessionId` → 自动发送目标 prompt”交互。
- 接入当前 workspaceDir，确保 session 不回退到虚假默认 cwd。
- 在首页或待办区提供 goal task 回到 session 的入口。
- 为 session 不可用场景提供提示，不自动新建覆盖关联。

### Phase 2：待确认与验收

目标：让 AI 输出不只停留在聊天里，而能进入待确认状态。

可能任务：
- 定义待确认项的最小状态。
- 接入待办系统或简单 metadata。
- 首页展示待确认列表。

### Phase 3：沉淀与回访

目标：让用户第二天回来能继续。

可能任务：
- 会话结果保存为笔记 / 文件 / 待办。
- 首页展示最近沉淀内容。
- 会话与目标记录建立可恢复入口。

## 12. v0.2 变更记录

- Phase 1 从“创建目标 → AI 推进 → 用户验收 → 结果沉淀”缩小为“创建目标 → 打开会话 → 可回找”。
- `Goal` 明确复用 Task 作为 canonical record，不新增 Goal 表或 markdown 模型。
- 建议最小扩展 `TaskItem.kind/sessionId/source`，用于区分目标、关联会话和记录来源；`source` 使用 `dashboard`。
- “待我确认”“结果沉淀”“Artifact 关系”全部后移到 Phase 2/3。
- 首页第一屏的第二块从“待我确认”改为“最近目标”，避免提前引入 `needs_review` 状态机。
- Phase 1 只改 `DashboardView` 作为目标输入入口，不维护 `HomePage` 与 `DashboardView` 双入口。
- Phase 1 要自动发送第一条目标 prompt；session 不可用时提示，不自动新建覆盖关联；旧 task 无需迁移。

## 13. Phase 1 spec 前硬判定

1. **自动发送 prompt**：目标提交后必须自动向新 session 发送第一条用户 prompt，内容至少包含目标文本；否则不算完成 AI 协作入口。
2. **只打开原 session**：点击 goal 时只打开已有 `sessionId`；如果 session 不存在、已删除或不可用，只展示失效/修复提示，不自动新建并覆盖原关联。
3. **入口只在 DashboardView**：Phase 1 只改 `DashboardView`，不改 `HomePage` 搜索框，不做主页/分屏结构重构。
4. **类型与兼容覆盖**：`TaskItem.kind/sessionId/source` 必须同步覆盖前后端类型、API 创建/更新白名单、测试 fixture；未带 `kind` 的旧任务必须按普通任务显示。

建议 Phase 1 spec：`文档/spec/首页目标到Pi会话.spec.md`。
