> 日期：2026-04-30
> 状态：按日期存储已实现；全量质量门禁受既有无关错误阻塞
> 审查人：架构师 + PM

## 1. 总体判断

**当前是一个文件列表，不是收件箱。**

技术架构正确（纯 .md 文件 + composable 抽离 + provide/inject 共享），但产品语义缺失——缺少收件箱的三个核心特征：

1. **随时捕捉** — 闪念入口仅在收件箱页面内，违背"摩擦最小"原则
2. **时间感知** — 平铺列表无分组、无已读/未读区分，50+ 条时不可用
3. **处理闭环** — 缺少"已处理"标记和"转为"能力，收件箱变垃圾场

综合评分：**2.5/5**

## 3. 关键问题详解

### 3.1 闪念捕捉不「随时」 🔴

- `⌘Enter` 只在收件箱页生效，无全局快捷键、无命令面板入口
- 竞品（Notion/Things 3/Obsidian）全部支持全局快速捕捉
- 当前路径：导航到收件箱 → 等加载 → 输入 → 提交，至少 3 步
- **灵感来时多一步导航就可能丢失**

### 3.2 没有处理闭环 🔴

GTD 收件箱的核心是「收集→处理→清零」：

- 无法标记「已处理」——笔记放一年和放一分钟没有视觉区别
- 无法「转为待办」「转为笔记」——闪念的出口只有归档和删除
- **积压死循环**：无清零机制 → 收件箱变垃圾场 → 用户放弃（GTD 工具最常见死因）

### 3.3 交互模式与 TaskView 不一致 🔴

| 维度     | TaskView             | InboxView        | 一致性 |
| -------- | -------------------- | ---------------- | ------ |
| 状态切换 | 一键点击圆圈         | 无状态机制       | ❌     |
| 编辑方式 | 内联编辑（点击即改） | 弹窗确认         | ❌     |
| 分组     | 今日/待处理/已完成   | 无分组           | ❌     |
| 操作步数 | 1-2步                | 删除4步、归档5步 | ❌     |

同一个工作空间里两套操作哲学，用户困惑。

### 3.4 平铺列表不可扩展 🟡

- 无日期分组（今天/昨天/本周/更早）
- 无未处理/已处理区分
- 一个月 200+ 条时列表不可用
- 侧边栏 badge 只显示总数，不区分已读/未读

### 3.5 收件箱与 AI 对话完全隔离 🟡

- ridge 是 AI 工作平台，对话中产生的想法无法一键扔进收件箱
- AI 建议保存的内容需要手动切换页面

### 3.6 情感设计：焦虑而非掌控 🟡

收件箱应该传达：**安全感**（放心扔进来）、**轻松感**（捕捉零负担）、**掌控感**（知道哪些处理了）

当前传达：**焦虑**（越积越多无清零希望）、**困惑**（哪些看过了？）、**被忽视**（错误状态不展示）

## 9. 按日期存储闪念的实施方案

> 2026-04-30 追加确认：用户明确要求「闪念按日期存储」，且「不要 checklist」。旧的一条闪念一个 md 尚未上线，不需要迁移和兼容。

### 9.1 新产品定义

- **闪念不是正式笔记**：用户捕捉的是临时想法，不应每条生成一个独立 `.md` 文件。
- **收件箱不是文件浏览器**：界面展示的是闪念条目列表，隐藏底层日期文件。
- **日期是存储单元**：每天一个 Markdown 文件，所有当天闪念追加到同一个文件。

### 9.2 存储结构

```text
收件箱/
  2026-04-30.md
  2026-05-01.md
```

单个日期文件内容：

```md
# 2026-04-30 闪念

## 10:21

想给收件箱加一个快速捕捉入口

## 10:35

用户可能不想每条闪念都生成文件
```

约束：

- 不使用 checklist。
- 不做旧文件迁移。
- 捕捉新闪念时：如果当天文件不存在则创建；如果存在则追加。
- 每条闪念用二级标题记录时间，正文保留用户原始输入。
- UI 展示时解析日期文件中的每个 `## HH:mm` 段落为一条闪念。

### 9.3 前端展示

用户看到的是条目列表，不是日期文件列表：

```text
今天
  10:35 用户可能不想每条闪念都生成文件
  10:21 想给收件箱加一个快速捕捉入口

昨天
  18:02 ...

更早
  2026-04-27 09:12 ...
```

展示规则：

- 默认按时间倒序展示。
- 分组：今天 / 昨天 / 更早。
- 每条显示：时间、正文预览、来源日期。
- 搜索匹配闪念正文，不再只匹配文件名。
- 侧边栏 badge 显示闪念条目总数，不显示日期文件数。

### 9.4 操作范围

第一版只实现与日期存储强相关的最小闭环：

1. 捕捉闪念：追加到 `收件箱/YYYY-MM-DD.md`。
2. 加载收件箱：读取 `收件箱/*.md`，解析为闪念条目。
3. 搜索：按条目正文搜索。
4. 日期分组：今天 / 昨天 / 更早。
5. Dashboard：展示最近 3 条闪念条目，计数为条目数。
6. 错误态：加载失败时在 UI 显示错误。

不在第一版做：

- checklist / 已处理状态。
- 旧文件迁移。
- 每条闪念独立重命名。
- 每条闪念独立归档。
- 批量操作。
- 转为待办/笔记/日记。

原因：用户明确不需要 checklist；当前首要目标是纠正存储模型，避免文件爆炸。

### 9.7 验收标准

- [ ] 捕捉 10 条闪念后，文件系统只生成当天一个 `YYYY-MM-DD.md`。
- [ ] 当天文件内容按时间追加，用户输入正文不丢失。
- [ ] 收件箱界面展示 10 条闪念条目，而不是一个日期文件。
- [ ] 条目按今天 / 昨天 / 更早分组。
- [ ] 搜索能匹配所有条目正文。
- [ ] Dashboard 最近闪念不受收件箱搜索关键词影响。
- [ ] 侧边栏 badge 显示闪念条目数。
- [ ] 加载失败时用户能看到错误提示。
- [ ] `npm run check` 通过。

### 9.8 回退策略

- 实现集中在 `useInbox` 与展示组件，若验证失败，可恢复为旧的文件列表模型。
- 新存储仍是 Markdown 文件，不引入数据库；回退不会破坏工作空间根目录结构。
- 因旧设计未上线，不需要旧数据迁移回退。
