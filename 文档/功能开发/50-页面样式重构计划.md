## 一、现状分析

### 1.1 技术栈

| 维度 | 当前状态 |
|------|----------|
| 框架 | Vue 3 + TypeScript |
| 样式引擎 | Tailwind CSS v4（PostCSS pipeline） |
| UI 组件库 | shadcn-vue（28 个组件集） |
| 主题系统 | 运行时 CSS 变量注入（7 个可切换主题） |
| 字体 | Manrope Variable（已改为 Inter Variable） |
| 图标 | lucide-vue-next |
| 工具 | `cn()` = clsx + tailwind-merge，CVA 变体管理 |

### 1.2 路由与页面结构

Vue Router 仅处理 3 个顶层路由，全部工作空间导航由 **SplitGrid 自定义分屏系统**管理：

```
/login       → LoginPage.vue
/            → WorkspacePage.vue（核心）
/*           → NotFoundPage.vue
```

WorkspacePage 是一个 1248 行的巨型组件，内部通过 `<SplitGrid>` 插槽渲染 10+ 种标签页类型。

### 1.4 当前样式痛点

#### A. 巨型组件与内联样式混杂

- `WorkspacePage.vue` **1248 行**，左侧边栏的 HTML 结构直接写在页面模板中，Tailwind 类多达数百个，无法快速定位某个入口的样式定义。
- 项目列表、固定入口、文件树、新建按钮全部挤在同一个 `aside` 中，缺乏结构分层。

#### B. 魔法数字泛滥

```
text-[9px], text-[10px], text-[11px], text-[12px], text-[13px]
h-6, h-7, h-8, h-9
px-1, px-1.5, px-2, px-2.5
py-0.5, py-1, py-1.5, py-2
```

这些离散值在全站重复出现，未形成统一的**字号/间距阶梯**。

#### C. 边框与背景透明度不一致

同一类元素使用不同透明度：
- `border-border/30`
- `border-border/40`
- `border-border/50`
- `border-border/60`
- `bg-muted/20`
- `bg-muted/30`
- `bg-muted/35`
- `bg-muted/40`

#### D. 全局样式残留旧主题痕迹

- `style.css` 滚动条颜色仍基于旧的「山脊」暖灰色调：`rgba(120, 113, 108, ...)`
- 字体 import 仍为 `@fontsource-variable/manrope`，但 default.css 已改为 `Inter Variable`
- `::selection` 使用 `hsl(var(--primary) / 0.3)`，但当前主题已改用 hex 值，混合模式可能不匹配

#### E. 硬编码颜色

`src/components/ui/` 中存在 3 处硬编码颜色：
- `Switch.vue`: `bg-white`
- `button/index.ts` destructive variant: `text-white`
- `badge/index.ts` destructive variant: `text-white`

#### F. shadcn 组件未完全统一

部分组件仍使用 `text-white` 而非 `text-primary-foreground`，导致切换到浅色主题时可能出现不可见性。

## 二、重构原则

1. **只动样式，不动逻辑** —— 不修改任何 JavaScript/TypeScript 业务逻辑、状态管理、API 调用、路由配置。
2. **保持视觉等价** —— 重构后的页面必须与原页面视觉表现一致（或更好），不引入功能退化。
3. **遵循 Tailwind v4 + shadcn-vue 规范** —— 不引入新的样式方案（SCSS、CSS-in-JS、CSS Modules）。
4. **设计系统驱动** —— 所有样式变更必须通过 CSS 变量（theme tokens）或 Tailwind 工具类表达，禁止硬编码颜色。
5. **组件化拆分** —— 将 WorkspacePage 中内联的侧边栏结构拆分为独立子组件。
6. **可验证** —— 每个重构阶段必须能通过 `npm run check`（ESLint + TypeScript），且不破坏现有测试。

## 三、重构目标

### 3.1 短期目标（Phase 1-3）

- 统一全局基础样式（字体、滚动条、选择高亮、阴影、圆角）
- 规范化设计 token（字号阶梯、间距阶梯、边框透明度标准）
- 拆分 WorkspacePage 左侧边栏为独立组件
- 修复硬编码颜色和旧主题残留

### 3.2 中期目标（Phase 4-6）

- 重构各标签页组件的模板结构，提取通用布局模式
- 统一卡片、列表、按钮、表单元素的视觉层级
- 规范化所有 `text-[Npx]` 为设计系统字号

### 3.3 长期目标（Phase 7+）

- 建立完整的 Design Token 文档
- 考虑引入暗色/浅色模式自动切换的优化
- 所有占位标签页接入真实组件或移除

## 四、分阶段实施计划

### Phase 1: 全局基础样式修复（1-2 天）

**目标**：消除旧主题残留，统一全局基础样式。

**任务清单**：

| # | 任务 | 文件 | 改动点 |
|---|------|------|--------|
| 1.1 | 更新字体 import | `style.css` | 将 `@fontsource-variable/manrope` 改为 `@fontsource-variable/inter`，确保与 default.css 一致 |
| 1.2 | 修复滚动条颜色 | `style.css` | 将暖灰色调 `rgba(120, 113, 108, ...)` 替换为基于 `--muted-foreground` 的灰蓝色调 |
| 1.3 | 修复 `::selection` | `style.css` | 改用 `color-mix(in oklab, var(--primary), transparent 70%)` 替代 `hsl(var(--primary) / 0.3)`，适配 hex 主题 |
| 1.4 | 清理 `ridge-panel-*` 工具类 | `style.css` | 检查 `ridge-panel-header` 和 `ridge-panel-inset` 在新主题下的视觉效果，必要时调整为 `color-mix(in oklab, var(--background) 80%, var(--muted))` |
| 1.5 | 修复硬编码颜色 | `components/ui/switch/Switch.vue` | `bg-white` → `bg-primary-foreground` |
| 1.6 | 修复 destructive 前景色 | `components/ui/button/index.ts` | `text-white` → `text-destructive-foreground` |
| 1.7 | 修复 badge destructive | `components/ui/badge/index.ts` | `text-white` → `text-destructive-foreground` |

**验收标准**：
- `npm run check` 通过
- 明暗模式下滚动条、选择高亮、卡片 header 视觉正常
- shadcn destructive 按钮在浅色模式下文字可见

### Phase 2: 设计 Token 规范化（2-3 天）

**目标**：建立统一的字号、间距、边框透明度标准。

**任务清单**：

| # | 任务 | 说明 |
|---|------|------|
| 2.1 | 定义字号阶梯 | 在 `style.css` 或新文件 `src/assets/design-tokens.css` 中定义： `text-2xs: 10px, text-xs: 11px, text-sm: 12px, text-base: 13px, text-md: 14px, text-lg: 16px`，并通过 `@theme inline` 暴露为 Tailwind utilities |
| 2.2 | 定义间距阶梯 | 统一内联 padding/margin 的常用组合：`space-1: 0.25rem, space-2: 0.5rem, space-3: 0.75rem`，避免 `px-2.5`、`py-1.5` 等离散值泛滥 |
| 2.3 | 定义边框透明度标准 | 规范化为 3 档：`border-subtle: border-border/30`（装饰性边框）、`border-default: border-border/50`（组件边框）、`border-strong: border-border/80`（强调性边框） |
| 2.4 | 定义背景透明度标准 | 规范化为 3 档：`bg-subtle: bg-muted/20`、`bg-soft: bg-muted/40`、`bg-hover: bg-muted/60` |
| 2.5 | 更新主题契约 | `theme-contract.css` 补充新的 token 映射，确保所有主题文件同步更新 |

**验收标准**：
- 新增 token 在 `npm run check` 下无类型错误
- 至少一个组件（如 `HomePage.vue`）完成试点替换，视觉与原版本一致

### Phase 3: WorkspacePage 侧边栏拆分（3-4 天）

**目标**：将 WorkspacePage 的 1248 行模板拆分为可维护的子组件。

**当前结构**：

```
WorkspacePage.vue (1248 行)
├── aside (260px 侧边栏)
│   ├── 固定视图入口（fixedEntries 循环）
│   ├── 项目列表（sidebarProjects 循环）
│   ├── 工作空间会话（sortedWorkspaceSessions 循环）
│   ├── 新建按钮行（4 个 Tooltip 按钮）
│   ├── 文件树（FileTreePanel）
│   └── 归档入口
└── main (SplitGrid 分屏区域)
```

**拆分方案**：

| 新组件 | 来源 | 职责 |
|--------|------|------|
| `SidebarFixedEntries.vue` | 固定视图入口 | 渲染 9 个入口按钮 + Badge 计数 |
| `SidebarProjectList.vue` | 项目列表 | 渲染项目折叠/展开、会话子列表、标签 |
| `SidebarWorkspaceSessions.vue` | 工作空间会话 | 渲染非项目会话列表 |
| `SidebarCreateActions.vue` | 新建按钮行 | 4 个新建按钮（笔记/文件夹/Canvas/Base） |
| `SidebarArchivedEntry.vue` | 归档入口 | 单个归档按钮 |
| `WorkspaceSidebar.vue` | 组合以上 | 统一 260px 容器、边框、背景、滚动行为 |

**验收标准**：
- WorkspacePage.vue 行数降至 400 行以内
- 所有 props/emits 通过 WorkspaceSidebar 透传，不丢失原有事件链
- `npm run check` 通过
- 现有测试（SplitGrid、WorkspacePage 相关）不受影响

## 五、风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| 拆分 WorkspacePage 引入事件传递 bug | 中 | 高 | 拆分前先提取 props/emits 接口定义；拆分后跑完整工作空间测试 |
| Tailwind v4 `@theme inline` 新 token 构建失败 | 低 | 高 | 小步增量添加，每次变更后 `npm run check` + 本地 dev server 验证 |
| 主题颜色调整导致某主题不可读 | 中 | 中 | 每次改主题后切换全部 7 个主题验证对比度 |
| 视觉等价争议 | 中 | 低 | 重构前截图基准，重构后逐像素对比关键页面 |
| 破坏 shadcn 组件升级路径 | 低 | 中 | 所有 shadcn 组件修改只改 class 字符串，不改组件结构；保留原始 reka-ui 用法 |
