# 右键删除静默失败修复方案

## 问题诊断

### 根因（双重 bug）

**Bug 1（P0，直接原因）：`AlertDialogAction` 同步关闭竞态**

reka-ui 的 `AlertDialogAction` 是 `DialogClose`，点击时**同步**执行 `onOpenChange(false)` → `onDeleteDialogOpenChange(false)` → `deleteTarget.value = null`，然后 `@click="confirmDelete"` 才执行，此时 `deleteTarget` 已为 null，`emit("delete")` 永远不触发。

```
用户右键 → 上下文菜单出现"删除" ✅
点击删除 → AlertDialog 确认框弹出 ✅
点击确认 → AlertDialogAction 内部同步关闭 → deleteTarget = null → confirmDelete 执行时 deleteTarget 已为 null → emit 永远不触发 ❌
```

**Bug 2（P0，workbench 场景）：`WorkspaceFileTree.vue` 未绑定事件**

该组件只绑定了 @select/@toggle-expand/@toggle-favorite/@refresh，缺少 @delete/@rename/@create-folder。即使 Bug 1 修复后，workbench 场景下的事件也无监听者。

### 两条路径对比

| 路由         | 页面组件                         | 文件树组件                        | @delete 绑定 | 结果         |
| ------------ | -------------------------------- | --------------------------------- | ------------ | ------------ |
| `/workspace` | WorkspacePage                    | FileTreePanel（直接）             | ✅ 已绑定    | 理论上能工作 |
| `/chat`      | WorkbenchPage → ProjectFilePanel | WorkspaceFileTree → FileTreePanel | ❌ 未绑定    | **静默失败** |

### 测试为何没发现

- **FileTreePanel.test.ts**：只测了 `emit('delete')` 触发 ✅，没测事件是否被父组件接收处理 ❌
- **e2e/file-tree.spec.ts**：只测了取消删除 ✅，没测确认删除后的效果 ❌
- **WorkspaceFileTree.vue**：无任何测试 ❌

测试覆盖了"组件会发事件"，但没覆盖"事件被正确处理"——**测试只证明了前半段没断，后半段根本没有**。

---

## 修复方案

### 核心改动

**废弃 `WorkspaceFileTree.vue`**，让 `ProjectFilePanel.vue` 直接使用 `FileTreePanel`，与 `WorkspacePage.vue` 统一入口。

理由：

- 两套组件共存是根源，每给 FileTreePanel 加新功能都可能只在一侧生效
- WorkspaceFileTree 是薄包装层，仅多加了 `@select`/`@toggle-favorite` 逻辑
- 合并后只需维护一条事件绑定路径

### 改动清单

1. **`ProjectFilePanel.vue`**：将 `WorkspaceFileTree` 替换为 `FileTreePanel`，补上 `@delete`、`@rename`、`@create-folder` 事件绑定和处理函数
2. **`WorkspaceFileTree.vue`**：标记废弃或直接删除
3. **提取共享 composable `useFileTreeActions`**：将 `handleDelete`、`handleRename`、`handleCreateFolderInTree` 从 `WorkspacePage.vue` 提取到 composable，`ProjectFilePanel.vue` 复用，避免重复代码
4. **操作反馈**：所有文件操作成功/失败时显示 toast 通知

### composable 设计

```ts
// composables/useFileTreeActions.ts
export function useFileTreeActions(
  workspaceDir: Ref<string>,
  refreshTree: () => void,
) {
  async function handleDelete(entry: FileTreeEntry) {
    try {
      await trashFileEntry(workspaceDir.value, entry.path);
      refreshTree();
      toast.success(`已删除 ${entry.name}`);
    } catch (err) {
      toast.error(`删除失败: ${err.message}`);
    }
  }
  // handleRename, handleCreateFolderInTree 同理
  return { handleDelete, handleRename, handleCreateFolderInTree };
}
```

### 测试补充

1. **`useFileTreeActions.test.ts`**：测试 composable 的 handleDelete 调用 API + 刷新 + toast
2. **`ProjectFilePanel` 组件测试**：验证 FileTreePanel 的 delete 事件被正确处理
3. **e2e 补充**：B4 测试增加确认删除后的效果验证

| 文件                                                | 变更类型                                     |
| --------------------------------------------------- | -------------------------------------------- |

| `composables/__tests__/useFileTreeActions.test.ts`  | 新增                                         |
| `e2e/file-tree.spec.ts`                             | 补充确认删除测试                             |

## 风险

- ProjectFilePanel 可能依赖 WorkspaceFileTree 的某些特有行为，需对比确认功能对等
- toast 组件需确认项目已有 sonner 或类似方案

## 回退策略

如果合并后 workbench 场景出问题，回退方式是恢复 WorkspaceFileTree 并补上缺失的事件绑定（临时方案）。
