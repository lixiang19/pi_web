# ridge 闪念系统 PRD v0.1

## 1. 产品定位

闪念是 ridge 的临时捕获队列。

它解决的问题不是“写笔记”，而是：

> 用户在任何工作场景里，先快速丢下任何内容；系统后台分析这条内容最适合被处理成什么正式对象；用户稍后点击处理；处理成功后清空这条闪念。

闪念的核心不是分类，而是 **处理建议**。

---

## 2. 核心原则

### 2.1 捕获安静

用户输入闪念时，只保存，不分类、不追问、不打扰。

捕获阶段不要求用户选择：

- 类型；
- 标签；
- 重要级别；
- 目标系统；
- 目标文件。

### 2.2 后台自动分析

闪念保存成功后，系统自动交给闪念 Agent 分析。

Agent 只生成一个最佳处理建议。

Agent 可以自动分析，但不能自动处理。

### 2.3 用户点击处理

所有处理动作都必须由用户点击触发。

处理成功后，原始闪念从 DB 中彻底删除。

### 2.4 DB 是临时队列

闪念 DB 只存未处理闪念。

目标是逐步清空 DB，而不是把 DB 做成长期知识库。

### 2.5 固定按钮永远可用

每条闪念都展示固定处理按钮。

用户可以绕过 AI 建议，直接选择固定处理方式。

用户处理优先级高于 AI 分析。

---

## 3. 范围

### 3.1 做

- 网页端右下角常驻闪念入口；
- 快速输入任意内容；
- 保存到 DB；
- 保存后轻提示；
- 后台自动调用闪念 Agent 分析；
- 每条闪念生成一个最佳处理建议；
- 收件箱页面查看待处理闪念；
- 每条闪念展示 AI 推荐处理；
- 每条闪念展示固定处理按钮；
- 支持处理到任务系统；
- 预留处理到日记系统；
- 预留处理到剪藏系统；
- 支持删除闪念；
- 处理成功后彻底删除原始闪念；
- 用户在 AI 分析完成前手动处理时，取消该闪念后台分析。

### 3.2 不做

- 捕获时分类；
- 捕获时打标签；
- 捕获时选择重要级别；
- AI 自动处理；
- 一条闪念推荐多个并列最佳动作；
- 把已处理闪念长期保留在 DB；
- 忽略状态；
- 闪念历史审计；
- 多端同步入口；
- 桌面全局快捷键；
- 安卓通知栏入口；
- 日记完整系统；
- 剪藏完整系统。

---

## 4. 用户角色

### 4.1 捕获中的用户

用户正在工作、对话、阅读或排查问题。

此时用户不想整理，只想快速保存一句话、一段材料、一个链接或一个待办意图。

成功体验：

> 不离开当前页面，也不思考分类，就能把内容丢进闪念。

### 4.2 清空收件箱的用户

用户稍后进入闪念收件箱。

系统已经为每条闪念准备好一个最佳处理建议。

用户只需要判断：

- 按 AI 建议处理；
- 改用固定处理按钮；
- 删除。

成功体验：

> 像清空邮件收件箱一样清空闪念队列。

### 4.3 闪念 Agent

闪念 Agent 负责理解原始输入，并推荐一个最佳处理方式。

它不直接创建正式对象，除非用户点击处理。

---

## 5. 核心处理关系

```txt
闪念
  ↓ 后台分析
处理建议
  ↓ 用户点击
正式系统对象
  ↓ 成功后
删除原闪念
```

第一版正式处理目标：

```txt
任务系统
日记系统（预留）
剪藏系统（预留）
删除
```

说明：

- 任务系统已经存在，闪念可以处理成正式任务对象；
- 日记和剪藏尚未存在，第一版只设计入口和边界，不要求完整实现；
- 删除不是忽略状态，而是放弃该闪念并从 DB 移除。
- 闪念处理成功后原始记录消失，不和目标对象建立长期关联。

---

## 6. 核心流程

### 6.1 快速捕获流程

```txt
用户点击右下角闪念按钮
↓
输入任意内容
↓
保存
↓
系统写入 DB
↓
显示轻提示
↓
后台启动闪念 Agent 分析
```

约束：

- 保存动作不能要求用户补充分类信息；
- 保存成功后不打开收件箱；
- 保存失败时保留用户输入；
- 空内容不能保存。

### 6.2 后台分析流程

```txt
闪念进入 DB
↓
进入待分析队列
↓
闪念 Agent 分析内容
↓
生成一个最佳处理建议
↓
收件箱中展示建议
```

约束：

- AI 分析失败时后台自动重试；
- 分析失败不打扰用户；
- 用户仍可使用固定按钮处理；
- 用户在分析完成前处理成功时，取消后台分析。

### 6.3 收件箱处理流程

```txt
用户打开闪念收件箱
↓
查看闪念内容
↓
查看 AI 推荐处理
↓
点击按建议处理
或点击固定处理按钮
或删除
↓
目标系统处理成功
↓
删除原闪念
```

约束：

- AI 推荐处理不能自动执行；
- 固定按钮始终可见；
- 处理失败时不得删除原闪念；
- 只有目标系统确认创建成功后，才删除原闪念。

---

## 7. 闪念状态

闪念主状态只表达是否还在队列中。

```txt
待处理
处理中
```

处理成功后不进入“已处理”状态，而是直接删除。

### 7.1 分析状态

分析状态独立于主状态。

```txt
未分析
分析中
已建议
```

失败不作为面向用户的常驻状态。

系统后台记录失败并自动重试。

### 7.2 删除规则

以下情况删除闪念：

- 用户主动点击删除；
- 用户按 AI 建议处理成功；
- 用户按固定按钮处理成功。

以下情况不得删除闪念：

- AI 分析成功；
- AI 分析失败；
- 处理动作开始但未完成；
- 处理失败；
- 目标系统返回失败。

---

## 8. 处理类型

### 8.1 AI 推荐处理

每条闪念最多展示一个 AI 推荐处理。

推荐文案应表达成用户能理解的动作。

示例：

```txt
建议：转为任务
建议：写入今天日记
建议：保存为剪藏
建议：整理成笔记
```

第一版主路径：

```txt
[按建议处理]
```

### 8.2 固定处理按钮

第一批固定按钮：

```txt
日记
任务
剪藏
删除
```

说明：

- 固定按钮不依赖 AI 分析结果；
- 即使建议仍在生成中，固定按钮也可点击；
- 用户点击固定按钮后，系统按该按钮对应的处理方式处理闪念；
- 处理成功后删除原闪念。

### 8.3 删除

删除是独立动作。

删除后闪念从 DB 移除。

第一版不设计“忽略”。

---

## 9. 任务系统处理规则

闪念处理为任务时，必须进入正式任务系统。

不得只写入任务 md 或临时列表。

依据现有任务系统 PRD：

- 任务绑定 workspace；
- 任务使用 DB 存储；
- 新建任务默认状态为待处理；
- 任务必须归属一个里程碑；
- 用户未选择里程碑时，进入系统默认“未归属”里程碑；
- 优先级为普通 / 重要 / 紧急；
- 标题、优先级、完成标准必填。

### 9.1 从闪念生成任务

处理为任务时，系统需要从闪念内容中生成：

- 任务标题；
- 优先级；
- 完成标准；
- 截止日期，若能明确识别；
- 所属里程碑，若能明确识别，否则进入“未归属”。

用户点击“任务”固定按钮时，如果必要信息不足，系统可以打开任务确认表单。

用户点击 AI 推荐“转为任务”时，优先展示 Agent 已生成的任务草稿。

### 9.2 成功条件

只有任务对象在任务系统中创建成功后，原闪念才删除。

如果任务创建失败，闪念保留在 DB 中，并允许用户重试。

---

## 10. 日记处理边界

日记系统尚未设计。

第一版闪念只定义日记处理意图：

```txt
把闪念沉淀为某一天的日记内容。
```

未来日记系统应至少支持：

- 按日期组织；
- 写入今天；
- 允许用户确认写入内容；
- 写入成功后通知闪念系统删除原闪念。

日记系统未完成前，日记按钮可以作为预留入口，不进入完整实现。

---

## 11. 剪藏处理边界

剪藏系统尚未设计。

第一版闪念只定义剪藏处理意图：

```txt
把闪念沉淀为外部资料、链接、摘录或可稍后阅读内容。
```

未来剪藏系统应至少支持：

- 链接；
- 摘录；
- 来源说明；
- 原文标题；
- 保存时间；
- 可搜索。

剪藏系统未完成前，剪藏按钮可以作为预留入口，不进入完整实现。

---

## 12. 页面结构

### 12.1 全局闪念入口

当前只有网页端。

网页端采用：

```txt
右下角常驻闪念按钮
```

点击后显示轻量输入浮层。

浮层只包含输入能力，不包含分类能力。

### 12.2 闪念收件箱页面

收件箱页面用于清空队列。

建议结构：

```txt
顶部：
- 待处理数量
- 分析中数量

主体：
- 闪念卡片列表
```

每张闪念卡片展示：

```txt
原始内容
创建时间
AI 推荐处理状态
AI 推荐处理文案
按建议处理按钮
固定处理按钮：日记 / 任务 / 剪藏 / 删除
```

### 12.3 空状态

当 DB 中没有未处理闪念时，展示：

```txt
闪念已清空
```

---

## 13. 闪念 Agent

### 13.1 定位

闪念 Agent 是专门判断闪念最佳处理方式的系统 Agent。

它不同于任务 Agent。

任务 Agent 负责处理任务；闪念 Agent 负责判断闪念应该变成什么。

### 13.2 输入

闪念 Agent 至少需要：

- 原始闪念内容；
- 可用处理目标说明；
- 任务系统约束；
- 日记和剪藏的当前可用状态。

### 13.3 输出

闪念 Agent 只输出一个最佳处理建议。

建议应包含：

- 推荐处理类型；
- 面向用户的推荐文案；
- 处理所需草稿内容；
- 是否需要用户补充信息；
- 无法推荐时的兜底建议。

### 13.4 推荐优先级

推荐时应遵循：

1. 明确行动项优先推荐任务；
2. 明确个人记录、感受、复盘优先推荐日记；
3. 明确外部资料、链接、摘录优先推荐剪藏；
4. 无法判断时，优先给出最保守且不会污染正式系统的建议。

---

## 14. 数据存储建议

闪念 DB 存储未处理闪念及其当前分析结果。

不存长期历史。

建议保存的信息包括：

- 原始内容；
- 创建时间；
- 主状态；
- 分析状态；
- AI 推荐处理；
- 处理草稿；
- 后台重试信息。

处理成功后，整条闪念记录删除。

---

## 15. 成功标准

第一版完成后，用户可以：

1. 在网页任意页面通过右下角入口快速保存闪念；
2. 保存时不需要分类、不需要选择目标；
3. 保存成功后看到轻提示；
4. 系统后台自动分析闪念；
5. 每条闪念最多得到一个 AI 推荐处理；
6. 用户可以在收件箱看到所有未处理闪念；
7. 用户可以点击 AI 推荐处理；
8. 用户可以使用固定按钮处理为任务；
9. 用户可以看到日记和剪藏作为预留处理目标；
10. 用户可以删除闪念；
11. 任务处理成功后，任务进入正式任务系统；
12. 处理失败时闪念仍保留；
13. 处理成功后原始闪念从 DB 中彻底删除；
14. 当所有闪念处理或删除后，收件箱显示清空状态。

## 16. 待确认问题

1. 日记系统是否以“每日一篇”为核心，还是允许同一天多条独立日记？
2. 剪藏系统是否只处理链接，还是也处理纯文本摘录？
3. AI 推荐为任务时，任务草稿是否必须弹出确认表单？
4. 日记和剪藏系统未完成前，对应按钮是隐藏、禁用，还是展示为“即将支持”？
