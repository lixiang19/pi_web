# 任务系统 DB 真源与 AI 操作模型设计 v0.4

> 任务：task #6 任务系统 DB 真源与 AI 操作模型设计  
> 日期：2026-05-01  
> 产品决策：任务只在专属任务页面 / 列表 / 看板 / 日历任务视图中查看和操作，不作为独立 markdown 文件打开编辑。因此 Task 以 DB 为唯一真源。  
> v0.4 更新：采纳用户最新决策：不做旧 `.ridge/tasks.json` 兼容、回退或导入迁移；任务直接以 workspace SQLite DB 为真源。

## 1. 核心结论

**Task 是 DB 真源；任务页面负责展示和编辑；RAG 通过文本投影读取任务；Markdown 只作为任务来源或导出，不作为任务主存储。**

这意味着：
- 用户手动建任务 → 写 DB；
- AI 创建/编辑任务 → 通过受控工具写 DB；
- 列表 / 看板 / 日历 → 都读同一份 Task；
- 全局搜索 / RAG → 读 Task 生成的 IndexDocument；
- 笔记 / 日记 / 闪念里提到任务 → 只是 source，不是任务真源。

## 2. 为什么 Task 用 DB

任务是状态对象，不是长文档对象。它需要：

- 快速筛选、排序、搜索；
- 看板按状态分组；
- 日历按日期投影；
- AI 创建、编辑、拆解、排期；
- 批量更新；
- 操作日志和撤销；
- 未来同步与迁移。

这些都更适合 DB，而不是 markdown frontmatter。

## 3. 存储边界

### DB 真源

建议路径：

```text
<workspace>/.ridge/ridge.db
```

Task 相关表：

- `tasks`：正式任务真源；
- `ai_operations`：AI 操作日志；
- `task_relations`：后置。MVP 先在 `tasks` 内联 `source_type/source_id/source_path/source_date` 或 `source_json`。
- `task_views`：后置。MVP 看板/日历只是同一 Task 的查询/排序视图，不新增独立 view 表。

### 非 DB 真源

- `笔记/*.md`：笔记正文真源；
- `日记/YYYY/MM/YYYY-MM-DD.md`：日记正文真源；
- `收件箱 / Capture`：由 task #7 单独定义；
- `.ridge/rag` / `IndexDocument`：RAG 投影，不作为任务编辑真源；
- `tasks.md` 导出：可读导出，不作为编辑入口。

## 4. Task 最小字段

```ts
type TaskStatus = "todo" | "doing" | "done" | "archived" | "cancelled";
type TaskPriority = "low" | "medium" | "high";
type TaskActor = "user" | "ai";
type TaskSourceType = "manual" | "capture" | "journal" | "note" | "event" | "ai";

type Task = {
  id: string;
  title: string;
  description?: string;
  status: TaskStatus;
  priority?: TaskPriority;
  dueDate?: string; // ISO date/datetime text
  startDate?: string; // ISO date/datetime text
  tags?: string[];
  source?: {
    type: TaskSourceType;
    id?: string;
    path?: string;
    date?: string;
  };
  createdBy: TaskActor;
  updatedBy?: TaskActor;
  createdAt: string;
  updatedAt: string;
};
```

### MVP 必填

- `id`
- `title`
- `status`
- `createdBy`
- `createdAt`
- `updatedAt`

### MVP 可选但建议保留

- `description`
- `priority`
- `dueDate`
- `startDate`
- `tags`
- `source`
- `updatedBy`

### 迁移兼容

现有 `.ridge/tasks.json` 状态需要明确映射：

| 旧状态 | 新状态 |
| --- | --- |
| `pending` | `todo` |
| `in_progress` | `doing` |
| `done` | `done` |

API 层可短期兼容旧值，但 DB 内统一存新状态。

日期字段统一为 ISO text。现有 number/null 格式迁移时转换为 ISO；无日期则为 null/undefined。

## 5. 多视图模型

同一份 Task 数据投影为不同视图：

### 列表视图

- 默认按 `status + dueDate + updatedAt` 排序；
- 适合快速浏览、筛选、批量处理。

### 看板视图

- 按 `status` 分列；
- 拖动卡片本质是更新 `task.status`；
- 不存独立看板数据。

### 日历任务视图

- 展示有 `dueDate` 或 `startDate` 的任务；
- 拖动到日期本质是更新 `dueDate/startDate`；
- 不存独立日历任务副本。

### 甘特图

暂不做。

原因：甘特图需要 `endDate`、`dependencies`、`progress` 等项目管理字段。等未来“创建项目”能力明确后，作为项目视图再评估。

## 6. AI 操作模型

### 6.1 AI 不能裸 SQL

AI 不直接操作 SQLite / SQL。AI 只能调用受控领域工具：

- `list_tasks`
- `get_task`
- `create_task`
- `update_task`
- `schedule_task`
- `complete_task`
- `split_task`
- `archive_task`

工具层负责：

- schema 校验；
- 权限判断；
- 字段白名单校验；
- 写 DB；
- 同事务记录 `ai_operations`；
- 生成 RAG 投影更新事件；
- 必要时提供 undo。

### 6.2 工具权限白名单

自动执行分级必须落到工具级 policy，而不是只停留在产品描述。MVP 建议：

| 工具 | AI 自动执行 | 字段/动作边界 |
| --- | --- | --- |
| `create_task` | 允许 | 必须有 title；status 默认 todo；必须记录 source/createdBy |
| `update_task` | 限制允许 | 只能改白名单字段：title、description、priority、dueDate、startDate、tags、status(todo/doing/done/archived) |
| `schedule_task` | 允许但可撤销 | 只改 dueDate/startDate，不改正文描述 |
| `complete_task` | 允许但可撤销 | 只改 status 为 done |
| `split_task` | 允许但可撤销 | 创建子任务，原任务默认不硬删除 |
| `archive_task` | 限制允许 | 软归档，不物理删除 |
| `delete_task` | 不暴露给 AI | MVP 不做 |
| `bulk_update_tasks` | 不暴露给 AI | MVP 不做 |

所有工具必须通过服务端 schema 校验、权限判断和 operation log，不允许 AI 裸 SQL 或绕过 repository。

### 6.3 自动执行模式

用户希望 AI 可以直接操作，无需每次确认。因此不采用“所有操作都弹确认”，而采用权限边界 + 日志 + 撤销。

建议分级：

#### 低风险：可自动执行

- 创建任务；
- 补充任务描述；
- 添加标签；
- 从 Capture 创建任务；
- 从 Journal/Note 提取任务；
- 归档低价值建议；
- 生成子任务草稿。

#### 中风险：可自动执行，但必须可撤销

- 修改截止日期；
- 修改优先级；
- 拆分任务并归档原任务；
- 安排到日历日期；
- 批量创建任务。

#### 高风险：先不自动执行

- 删除任务；
- 批量删除；
- 批量标记完成；
- 覆盖用户手写描述；
- 修改大量日期；
- 跨项目/跨空间迁移。

## 7. AI 操作日志

如果允许 AI 无确认写 DB，必须有操作日志。

硬边界：**Task 变更与 `ai_operations` 必须在同一事务提交**。不能出现“任务已被 AI 修改但日志缺失”的状态；如果日志写入失败，任务变更也必须回滚。

```ts
type AiOperation = {
  id: string;
  actor: "ai";
  agentId?: string;
  toolName: string;
  targetType: "task";
  targetId?: string;
  action: "create" | "update" | "schedule" | "complete" | "split" | "archive";
  before?: Partial<Task>;
  after?: Partial<Task>;
  reason?: string;
  source?: {
    type: TaskSourceType;
    id?: string;
    path?: string;
    date?: string;
  };
  status: "applied" | "failed" | "undone";
  createdAt: string;
};
```

### 日志用途

- 用户查看 AI 做了什么；
- 支持撤销；
- 调试 AI 工具；
- 给 RAG / 回顾提供上下文；
- 审查高风险自动操作。

## 8. 软删除与撤销策略

### 8.1 禁止默认硬删除

MVP 中 UI 和 AI 删除都优先做软删除：

- 删除任务 → `status = archived` 或 `cancelled`，并记录 `deletedAt` / `archivedAt`；
- AI 不允许直接物理删除；
- 物理删除作为后续清理能力，需单独设计。

这样可以支撑 undo、审计和误操作恢复。

## 8.2 撤销策略

MVP 先做任务级 undo，不做复杂事务回滚。

### 可撤销操作

- create：删除或 archive 该 task；
- update：用 `before` patch 恢复；
- schedule：恢复原 `dueDate/startDate`；
- complete：恢复原 `status`；
- split：归档/删除生成的子任务，恢复原任务状态。

### 暂不做

- 跨多个对象的复杂批量撤销；
- 与 markdown 文件同步修改的事务撤销；
- 长期版本历史。

## 9. RAG / 全局搜索投影

任务虽然 DB 真源，但必须进入全局 RAG。

统一生成：

```ts
type IndexDocument = {
  id: string;
  sourceType: "task";
  sourceId: string;
  title: string;
  text: string;
  metadata: {
    status: TaskStatus;
    priority?: TaskPriority;
    dueDate?: string;
    tags?: string[];
    source?: Task["source"];
  };
  updatedAt: string;
};
```

文本示例：

```text
任务：整理闪念、待办、日历数据模型
状态：todo
优先级：high
截止日期：2026-05-03
来源：capture:cap_123
描述：讨论 Capture / Task / Journal / Calendar 的关系。
```

MVP 先实现纯函数：

```ts
taskToIndexDocument(task): IndexDocument
```

由未来 indexer 调用。

可选后置：生成只读 markdown 投影：

```text
<workspace>/.ridge/rag/tasks/task_123.md
```

但它只是索引/可读投影，不是任务编辑入口。

## 10. 与其他对象的关系

### Capture → Task

- Capture 由 task #7 定义；
- AI 或用户可从 Capture 创建 Task；
- Task.source = `{ type: "capture", id: captureId }`；
- Capture.processedTo 指向 taskId。

### Journal → Task

- 日记仍是 markdown 真源；
- AI 可以从日记中提取任务；
- Task.source = `{ type: "journal", path, date }`。

### Note → Task

- 笔记仍是 markdown 真源；
- AI 可以从笔记中提取任务；
- Task.source = `{ type: "note", path }`。

### Event / Calendar

- MVP 日历任务视图投影 Task 的日期；
- 独立 Event 以后单独定义；
- 如果 Task 需要安排到时间块，可先使用 `startDate/dueDate`，不急着建 Event。

### MVP 不做

- 不做复杂项目依赖；
- 不做完整版本历史；
- 不让 AI 直接裸 SQL；
- 不生成可编辑 task markdown 文件。

## 12. 迁移策略

### Phase 0：架构确认

先评审本文档，不改代码。

### Phase 1：SQLite TaskRepository 最小实现

- 引入 workspace DB 路径：`<workspace>/.ridge/ridge.db`；
- 建 `tasks` schema；
- `/api/workspace/tasks` 直接读写 DB；
- 不做 `.ridge/tasks.json` 兼容层；
- 不做 JSON → DB 导入迁移；
- 不做回退和双写；
- 前端 API 语义尽量保持不变，但底层任务存储直接切 DB；
- 补 repository / API / 基础 CRUD / 状态 / 日期测试。

### Phase 2：AI 操作日志与工具接入

- 暴露受控领域工具；
- 所有 AI 写入必须写 `ai_operations`；
- undo 先支持单任务 create/update/schedule/complete；
- 禁止 AI 硬删除。

### Phase 3：RAG 投影

- 新增 `taskToIndexDocument(task)`；
- 接入全局 RAG indexer；
- 第一版只提供 `buildTaskIndexDocument(task)` / `taskToIndexDocument(task)`；
- md mirror / `.ridge/rag/tasks/*.md` 作为调试导出后置，不进入 MVP。
