# 项目与设备模型 PRD v0.1

## 1. 背景

ridge 的工作空间最初假设运行在服务器上：

- 工作空间在服务器。
- 项目文件也在服务器。
- Pi 在同一环境内运行。
- Pi 可以直接通过 `bash / ls / read / write / git` 操作项目文件。

随着 Tauri 桌面端、用户本机项目、VPS、手机等设备形态出现，项目文件不再一定在服务器上。

因此需要重新定义：

- 工作空间承载什么。
- 项目承载什么。
- Pi 在哪里运行。
- 项目内 Pi 如何访问工作空间数据。

---

## 2. 产品目标

建立一套统一模型，使 ridge 同时支持：

- 云端工作空间。
- 本机项目。
- 服务器项目。
- VPS 项目。
- 后续移动端轻量项目能力。

核心目标：

> 工作空间负责知识与协作，项目负责文件与执行，Pi 跟随项目运行。

---

## 3. 核心结论

### 3.1 工作空间只承载四类内容

工作空间是云端真源，负责：

- 记忆
- 任务
- 聊天
- 资料

工作空间不直接接管项目文件。

---

### 3.2 项目可以创建在任意设备上

项目可以位于：

- 用户电脑
- 服务器
- VPS
- 其他设备

项目文件归所在设备管理。

云端只保存项目元信息和注册关系。

---

### 3.3 Pi 跟着项目运行

如果项目在用户电脑：

> Pi 在用户电脑的项目目录内运行。

如果项目在服务器：

> Pi 在服务器项目目录内运行。

Pi 不远程操作项目文件。

---

### 3.4 项目内 Pi 通过 MCP 访问工作空间

项目内 Pi 需要访问云端工作空间时，通过 ridge 提供的 MCP。

MCP 负责连接：

```text
项目内 Pi
→ ridge MCP
→ 云端工作空间
```

MCP 能力先不展开，后续单独设计。

---

## 4. 概念定义

### 4.1 工作空间

用户的云端知识与协作中心。

包含：

- 记忆
- 任务
- 聊天
- 资料

不包含：

- 本地代码库源码
- 本地项目完整文件树
- 本地 Git 状态真源

---

### 4.2 项目

项目是一个可执行的工作现场。

它可以是：

- GitHub / Git 代码库
- 本地文件夹
- 研究课题文件夹
- 服务器上的工作目录
- VPS 上的自动化目录

项目的核心特征：

- 有所在设备。
- 有本地文件边界。
- 可以运行 Pi。
- 可以注册到云端工作空间。

---

### 4.3 设备

设备是项目所在的位置。

例如：

- 当前电脑
- 云端服务器
- VPS
- 手机

设备提供不同能力：

| 设备类型 | 主要能力 |
|---|---|
| 电脑 | 本地文件、代码、Git、命令执行 |
| 服务器 | 云端项目、长期运行、协作能力 |
| VPS | 自动化、部署、长期任务 |
| 手机 | 查看、闪念、轻量操作 |

---

### 4.4 项目注册

项目在设备本地创建后，注册到云端工作空间。

注册后，云端知道：

- 有这个项目。
- 它属于哪个用户。
- 它在哪个设备上。
- 它可以和工作空间数据产生关系。

但云端不接管项目文件。

---

## 5. 核心用户流程

### 5.1 本地创建项目

```text
用户在本机选择/创建项目目录
→ ridge 本地生成临时项目身份
→ 向云端工作空间注册
→ 云端返回正式项目身份
→ 本地保存绑定关系
```

注册完成后，该项目出现在工作空间的项目列表中。

---

### 5.2 进入项目

```text
用户打开项目
→ ridge 自动在项目目录启动 Pi
→ Pi 获得项目本地文件能力
→ Pi 通过 MCP 获得工作空间访问能力
```

用户不需要手动启动 Pi。

---

### 5.3 项目内使用 Pi

Pi 可以直接操作当前项目：

- 搜索文件
- 读取文件
- 修改文件
- 执行命令
- 操作 Git

Pi 访问工作空间时走 MCP：

- 查询记忆
- 查询任务
- 读取资料
- 写入沉淀内容
- 关联聊天上下文

具体 MCP 能力不在本 PRD 展开。

---

### 5.4 资料进入工作空间

项目文件可以进入工作空间资料库。

但必须是显式行为。

例如：

```text
把这份设计文档加入工作空间资料
把这个 README 作为项目资料沉淀
把这段研究笔记保存到资料库
```

默认不把整个项目文件夹同步到工作空间。

---

### 5.5 任务关联项目

任务属于工作空间。

任务可以关联项目。

但禁止默认关联。

无论是用户创建任务，还是 AI 创建任务，都必须显式选择是否挂到项目。

规则：

> 任务可以关联项目，但不能因为当前在某个项目内就自动关联。

---

## 6. 信息架构

```text
工作空间
├── 记忆
├── 任务
├── 聊天
├── 资料
└── 项目列表
    ├── 本机项目
    ├── 服务器项目
    └── VPS项目

项目
├── 项目文件
├── Git / 命令环境
├── 项目内 Pi
└── MCP 工作空间连接
```

---

## 7. 页面建议

### 7.1 项目列表页

展示所有已注册项目。

每个项目展示：

- 项目名称
- 所在设备
- 项目状态
- 最近访问时间
- 是否可打开

---

### 7.2 项目详情页

用于查看项目与工作空间的关系。

包含：

- 项目概览
- 所在设备
- 关联任务
- 关联资料
- 关联聊天
- 最近记忆
- 打开项目入口

---

### 7.3 项目内工作页

项目真正执行发生在项目内。

包含：

- 文件区
- 聊天区
- 任务/资料引用区
- Pi 状态
- MCP 连接状态

---

## 8. 数据存储建议

### 8.1 云端保存

云端保存：

- 工作空间数据
- 项目注册信息
- 设备关系
- 用户授权关系
- 项目与任务/资料/聊天/记忆的关联

云端不默认保存：

- 项目完整文件
- 本地代码源码
- 本地 Git 仓库内容

---

### 8.2 设备本地保存

设备本地保存：

- 项目文件
- 项目路径绑定
- 项目运行环境
- Pi 本地执行上下文
- 与云端项目身份的绑定关系

---

## 9. 权限原则

### 9.1 MCP 使用用户 token

项目内 Pi 访问工作空间时，使用用户授权。

不使用独立项目 token 作为首版方案。

---

### 9.2 项目文件权限归设备

Pi 操作项目文件时，遵循设备本地权限与用户确认机制。

云端不直接读取或修改本地项目文件。

---

### 9.3 工作空间写入必须可追溯

项目内 Pi 写入工作空间时，需要能追溯来源：

- 哪个用户
- 哪个项目
- 哪台设备
- 哪次会话

---

## 10. 明确不做

首版不做：

- 不做项目文件云端同步。
- 不做本地代码默认索引。
- 不做完整分布式文件系统。
- 不做离线模式。
- 不做自动任务挂项目。
- 不展开 MCP 具体工具能力。

---

## 11. 成功标准

首版成功标准：

1. 用户可以在本机创建项目并注册到云端工作空间。
2. 用户可以在服务器项目中使用同一套项目模型。
3. 进入项目后 Pi 自动启动。
4. Pi 可以在项目目录内操作项目文件。
5. Pi 可以通过 MCP 访问工作空间。
6. 工作空间能展示项目与任务、资料、聊天、记忆的关系。
7. 任务不会被默认挂到当前项目。
8. 项目文件不会被默认同步到云端。

---

## 12. 待确认问题

1. 项目注册失败时，本地临时项目是否允许继续使用？
2. 一个本地项目能否注册到多个工作空间？
3. 一个云端项目是否允许绑定多个设备实例？
4. 项目重命名时，以本地名称为准还是云端名称为准？
5. MCP 断开时，项目内 Pi 是否允许继续仅操作本地项目？
6. 手机设备是否只作为轻量入口，不承载完整项目执行？

---

## 13. 一句话定义

> 工作空间是云端知识层，项目是设备执行层，Pi 跟着项目运行，MCP 连接两者。
