# 添加项目功能重新设计方案

> **v3 — 采用 Agent A 架构方案后的执行版本**
> 核心改变：① 不碰现有文件 API 安全逻辑 ② 新增独立 browse 端点 ③ 保留 Dialog 文件夹选择器 ④ 去掉自动补全

- 状态：已确认
- 日期：2026-04-08
- Owner：开发团队
- 目标版本/里程碑：v1.0

## 1. 背景与问题

- 背景：需要在Sidebar中添加"自定义项目"功能，让用户可以快速添加和管理常用项目
- 现象/痛点：
  1. 原`webkitdirectory`方案无法获取绝对路径（浏览器安全限制）
  2. 第一版Node API方案计划修改现有 `/api/files/tree` 安全逻辑，破坏已有安全边界语义
  3. 第一版交互设计（逐层点击浏览）效率极低，程序员需要6-8次点击才能到达目标目录
  4. 浏览器原生目录选择不可用，必须由 Node 服务层暴露目录浏览 API 给前端 Dialog 使用
- 成功定义：用户可在 Dialog 中浏览 Home 目录下的文件夹并确认选择，路径正确保存，Sidebar 显示项目列表

## 2. 目标与非目标

- 目标：
  - G1: 提供以 Dialog 为核心的文件夹选择交互
  - G2: 正确获取并保存项目的绝对路径
  - G3: 支持项目的添加、删除、打开操作
  - G4: 项目列表持久化到 `~/.ridge/projects.json`
- 非目标：
  - NG1: 不修改Tauri相关代码（Tauri只负责打包）
  - NG2: 不改变现有会话数据结构
  - NG3: 不修改现有 `/api/files/tree` 的安全检查逻辑

## 3. 现状探索（Explorer）

### 3.1 已有基础设施（可直接复用）

| 资源 | 位置 | 用途 |
|------|------|------|
| `GET /api/files/tree` | `packages/server/src/index.js` | 现有文件浏览API，**不修改** |
| `WorkspaceFileTree.vue` | `packages/web/src/components/` | 现有文件树组件，**不复用**（场景不同）|
| `session-metadata.js` 写队列模式 | `packages/server/src/` | 持久化写入模式参考 |
| `api.ts` `request()` 封装 | `packages/web/src/lib/api.ts` | 前端API调用基础 |

### 3.2 现有安全检查（禁止修改）

```js
// packages/server/src/index.js — 不能动这里
isPathInAllowedRoots(rootPath, workspaceScope.allowedRoots)  // 工作区范围限制
ensureWithinRoot(targetPath, rootPath)                        // 子路径校验
```

**解决方案**：新增独立的 `/api/filesystem/browse` 端点，硬编码安全边界为 `os.homedir()`，完全独立，不影响现有任何逻辑。

### 3.3 约束

- 限制1：Tauri只打包，无法提供原生文件选择对话框
- 限制2：用户级持久化数据必须走现有 `storage` 子系统和 `~/.ridge/` 数据目录
- 限制3：不能修改现有API的安全检查代码

## 4. 资料与依据（Librarian）

- 关键结论：
  - Agent A 的核心判断成立：必须新增独立文件系统 browse 端点，不能污染现有 `/api/files/tree` 的安全语义
  - 用户级配置与收藏已存在于 `~/.ridge/`，项目列表应复用同一 storage 子系统
  - 本次按用户明确要求保留 Dialog 交互，不实现自动补全

## 5. 方案概览（主方案）

### 5.1 方案摘要

- 一句话：新增独立文件系统浏览 API + Dialog 文件夹选择器 + 项目列表持久化到 `~/.ridge/projects.json`
- 取舍：采用 Agent A 的边界划分和存储设计，但交互层保留 Dialog 浏览器，不做自动补全
- 影响面：
  - 后端：新增独立 browse 路由和项目 CRUD 路由，不修改现有代码
  - 前端：新增 `ProjectSelectorDialog.vue`、`useProjects.ts`，修改 `SessionSidebar.vue`

### 5.3 风险与回滚

- 风险：
  - Home 目录层级较深时浏览路径较长：通过面包屑、返回上一级和仅显示目录列表降低操作成本
  - 用户选到不存在或无权限目录：后端 `POST /api/projects` 前做 `fs.stat` 校验
- 回滚：配置数据在 `~/.ridge/projects.json` 中，删除该文件即可回滚

## 6. 详细设计（Spec）

### 6.1 交互/行为

**核心交互范式（Dialog 文件夹选择器）**：

```
┌────────────────────────────────────────────────┐
│  选择项目文件夹                                │
├────────────────────────────────────────────────┤
│  Home / Documents / GitHub                     │  ← 面包屑
├────────────────────────────────────────────────┤
│  ← 返回上一级                                  │
│  📁 openchamber                                │
│  📁 pi_web                                     │
│  📁 sandbox                                    │
├────────────────────────────────────────────────┤
│  已选择：/Users/admin/Documents/GitHub/pi_web  │
│                         [取消] [确认添加]      │
└────────────────────────────────────────────────┘
```

**用户流程**：
1. 点击 Sidebar “添加项目” 按钮 → 打开 Dialog
2. Dialog 从 Home 目录开始显示目录列表
3. 用户通过点击目录逐层导航，并选中当前目录或子目录
4. 点击“确认添加”后，前端调用 `POST /api/projects`
5. Dialog 关闭，项目出现在 Sidebar 列表中

**项目列表区域**（Sidebar中）：
```
自定义项目                [+]
────────────────────────────
  pi_web
  ~/Documents/GitHub      [···]  ← 右键/更多按钮：打开/删除/复制路径
────────────────────────────
  openchamber
  ~/Documents/GitHub      [···]
```
- 项目名称（文件夹名）
- 路径摘要（父目录，非完整路径）
- 悬浮全路径 tooltip
- 更多按钮：打开会话 / 删除 / 复制路径

**边界条件**：
- 空态：无项目时显示"暂无项目"文字提示
- 重复项目：路径去重，已存在时提示"该项目已添加"
- 路径不存在：`POST`时服务端校验，返回400错误
- 权限错误：读目录失败时显示错误提示

### 6.2 接口与契约

**新增：文件系统浏览端点**
```
GET /api/filesystem/browse?path=/Users/admin/Documents
→ {
    homeDir: "/Users/admin",
    path: "/Users/admin/Documents",
    parent: "/Users/admin",
    entries: [{ name: "GitHub", path: "/Users/admin/Documents/GitHub", kind: "directory" }, ...]
  }
安全：path 必须在 os.homedir() 范围内；不存在时返回 400
```

**新增：项目 CRUD**
```
GET  /api/projects           → { projects: Project[] }
POST /api/projects           → Project
  body: { path: string }     // 绝对路径，服务端验证路径存在
DELETE /api/projects/:id     → { ok: true }
```

**Project 类型**：
```typescript
interface Project {
  id: string;        // crypto.createHash('md5').update(path).digest('hex').slice(0,8)
  name: string;      // path.basename(projectPath)
  path: string;      // 绝对路径
  addedAt: number;   // Date.now()
}
```

### 6.3 数据与存储

**存储位置**：`~/.ridge/projects.json`

**Schema**：
```json
{
  "version": 1,
  "projects": [
    { "id": "a1b2c3d4", "name": "pi_web", "path": "/Users/admin/Documents/GitHub/pi_web", "addedAt": 1744000000000 }
  ]
}
```

**写入模式**：复用现有 `storage/index.js` 的文件锁与 JSON 持久化模式，新增 `projects.json` 文件。

- 发布步骤：M1（后端） → M2（前端组件） → M3（Sidebar集成）
