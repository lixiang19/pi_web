# 自动化功能方案：定时会话

- 状态：已按新版方向实现首版
- 日期：2026-04-27
- Owner：AI Agent

## 1. 当前结论

自动化首版只做一件事：保存一条定时会话规则，到时间后由 Node 服务创建一条普通 AI 会话，并把规则里配置好的消息发送给 AI。

自动化页不是任务中心，不展示运行记录，不展示运行中状态，不做审批中心，不做工作流节点。触发后的结果就是普通会话，用户在左侧会话列表里自然看到新会话。

## 2. 产品边界

- 页面名称仍挂在一级入口“自动化”，页面内容表达为“定时会话”。
- 前端只管理规则：名称、启用状态、项目、Agent、模型、思考、频率、下一次会话预览、提示词。
- 规则支持三种最小频率：每天、每周、间隔分钟。
- “立即创建”复用同一条规则，马上创建普通会话并发送消息。
- “保存”只保存规则本身，不创建运行记录。
- “删除”只删除规则，不影响已创建的普通会话。

## 3. 技术方案

### 3.1 服务端

- 新增 `automation_rules` 表，存放规则。
- Node 服务启动后读取启用规则，使用一个全局 `setTimeout` 等待最近的 `next_run_at`。
- 到点后查询所有 due 规则，逐条调用普通会话链路：
  - `createSessionRecord`
  - `applySessionAgentSelection`
  - `upsertIndexedSessionRecord`
  - `record.session.prompt(prompt, { source: 'interactive' })`
- 每次触发后重新计算该规则的下一次时间，并重新安排全局 timer。
- 服务重启只恢复规则和下一次触发，不恢复所谓“运行中任务”，因为这里没有自动化运行实体。

### 3.2 前端

- `AutomationsPage.vue` 拆成两栏：
  - 左侧：规则列表，只显示规则名、下一次创建会话时间、启用/暂停。
  - 右侧：规则编辑器，保存定时会话的输入参数。
- 前端不维护运行历史。触发后调用 `refreshSessions()` 和 `openChatSession(sessionId)`，进入新建的普通会话。
- 项目、Agent、模型、思考沿用现有会话选择来源，不单独造一套自动化选择模型。

## 4. 接口

- `GET /api/automations`：返回规则列表。
- `POST /api/automations`：创建规则。
- `PATCH /api/automations/:automationId`：更新规则。
- `POST /api/automations/:automationId/toggle`：启用或暂停规则。
- `POST /api/automations/:automationId/run`：立即创建一次普通会话并发送消息。
- `DELETE /api/automations/:automationId`：删除规则。

## 5. 非目标

- 不做运行记录表。
- 不做自动化状态中心。
- 不做 cron 表达式编辑器。
- 不做文件变化、Git 事件、线程唤醒触发。
- 不做自动化专属消息协议。
- 不做 Tauri 原生定时器。
