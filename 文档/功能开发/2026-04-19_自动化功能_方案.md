# 自动化功能 方案

- 状态：已确认
- 日期：2026-04-19
- Owner：AI Agent
- 目标版本/里程碑：自动化功能首版设计

## 1. 背景与问题

- 背景：Web 工作台已经把 自动化 作为一级主路由预留出来，但当前仍是占位页，真实的自动化定义、触发、调度、执行与结果回放都还没有建立。
- 现象/痛点：自动化能力目前只能分散地藏在会话、子代理、终端、Git 等细节里，用户无法把“定时执行一条普通对话”稳定配置成产品能力，更无法管理运行记录与失败原因。
- 成功定义：首版明确建立一个独立自动化中心，让用户能配置单 Agent 自动化，支持 cron、手动、线程唤醒、文件变化、Git 事件触发，并且每次触发都新建一条普通会话完成执行。

## 2. 目标与非目标

- 目标：
  - 建立自动化领域模型：定义、触发器、运行记录、去重游标、ask/permission 策略。
  - 复用现有普通会话创建与发送链路，每次自动化执行都落成一条真实普通会话，而不是 task panel 或 synthetic message。
  - 在 自动化 一级页面提供列表 + 详情表单 + 运行记录的首版产品形态。
  - 支持五类触发：cron 定时、手动触发、线程唤醒、文件变化、Git 事件。
  - 支持 ask 自动回复固定引导语，命中 permission 请求则直接失败并提示用户调整方案。
  - 只恢复自动化定义与调度状态，不恢复服务重启前运行中的任务。
- 非目标：
  - 不做多 Agent 串并行工作流编排。
  - 不做节点式可视化流程编辑器。
  - 不做 Webhook 触发、外部 SaaS 集成、Tauri 原生调度。
  - 不做运行中任务跨服务重启续跑。
  - 不做独立审批中心或自动化专属消息协议。

## 5. 方案概览（主方案）

### 5.1 方案摘要

- 一句话：新增一个服务端拥有的自动化域层，让自动化定义在 ridge.db 中持久化，由 scheduler 监听多种触发器；每次触发时直接创建一条普通会话并发送 prompt，前端自动化页只负责配置与查看运行记录。
- 取舍：放弃“子代理任务中心”方向，转而复用现有普通会话链路，换来更低的协议复杂度、更好的历史回放一致性，以及与现有聊天体系完全一致的结果语义。
- 部署形态：Linux 部署时，scheduler 作为 Node 服务内的常驻模块运行，由 systemd 或等价守护方式保证服务进程存活；不把业务调度拆给桌面端或外部 cron 脚本。
- 影响面：会新增服务端自动化模型、调度器、触发适配器和自动化接口；会重做 packages/web/src/pages/AutomationsPage.vue 及其拆分组件；会扩展 ridge.db schema 与模块文档。

### 5.2 风险与回滚

- 风险：
  - ask 自动回复固定引导语可能在部分任务上无法给出最优分支：通过把 ask 策略显式配置为 automation.askPolicy，并在运行记录中记录 ask 命中次数缓解。
  - 文件变化和 Git 事件容易重复触发：通过为每个 trigger 建立指纹游标与去抖窗口缓解。
  - permission 直接失败会让部分自动化看起来“不稳定”：通过在详情页明确展示“不适合自动化执行”的失败原因缓解。
  - 服务重启不恢复运行中任务可能造成“中断感”：通过首版明确只恢复定义和调度器，不承诺续跑，并把中断状态写入最近一次运行记录缓解。
- 回滚：
  - 回滚策略：若多触发器方案复杂度过高，可先保留统一数据模型和自动化页面，只启用 手动触发 + cron 两种触发器，其余触发器在同一模型下后续开启，不回退到子代理任务模型。

## 6. 详细设计（Spec）

### 6.1 交互/行为（如适用）

- 用户流程：
  - 用户进入 自动化 页面，在左侧列表查看已有自动化，在右侧详情表单配置名称、项目 cwd、Agent、模型、thinking、提示词模板、触发器、ask 策略与启停状态。
  - 用户保存自动化后，定义被写入 ridge.db，scheduler 会重新装载该定义对应的触发器。
  - Linux 服务启动时，automation scheduler 会从 ridge.db 读取所有 enabled 定义，恢复 cron/file/git/thread-heartbeat 的触发注册，并维护一个“最近到期触发器优先”的单一调度循环。
  - 手动触发时，服务端立即创建一条普通会话，记录本次 run，并向该会话发送由 prompt 模板与触发上下文拼出的最终 prompt。
  - cron 触发时，scheduler 在 Node 进程内根据标准五段 cron 表达式与 IANA timezone 计算 next_due_at，使用单一 setTimeout 唤醒最近到期定义；命中计划时间后创建 run，执行链路与手动触发一致。
  - 线程唤醒触发时，自动化根据配置的 sourceSessionId 与 intervalMinutes 生成心跳调度，并在命中时以 sourceSessionId 作为可选 parentSessionId 新建普通会话执行。
  - 文件变化触发时，不使用 fs.watch 或第三方 watcher；改用服务端轮询扫描器，每 15000ms 对匹配范围内文件计算 relativePath + size + mtimeMs 指纹，并在 debounceMs 窗口后聚合变更生成触发上下文，新建普通会话执行。
  - Git 事件触发时，服务端复用现有 git-service，每 15000ms 轮询 getStatus(cwd)，用 current/tracking/ahead/behind/files 指纹识别 branch-change 与 status-change 事件，再触发执行。
  - 运行中的会话若发出 ask，请服务端自动调用现有 ask respond 链路，回填固定答案“现在处于自动化执行流程，仔细选择最优解”。
  - 运行中的会话若命中 permission 请求，本次 run 立即标记 failed，并在运行记录里写明“命中权限审批，请调整自动化方案或 Agent 权限”。
  - 前端运行记录列表展示 status、trigger、开始/结束时间、关联 sessionId，并允许一键跳到这条普通会话继续查看完整历史。
- 边界条件：
  - 每次触发必须新建普通会话，不复用固定会话，也不创建 task panel。
  - 自动化运行记录可以关联 sessionId，但 session 本身仍是普通聊天会话，不引入自动化专属消息类型。
  - 自动化系统只追踪自己创建的顶层普通会话；若该会话内部再次调用 task 之类工具并生成 child session，这些 child session 不单独计为 automation_runs。
  - 文件变化与 Git 事件都必须有去重游标，避免同一轮变化连续触发多次。
  - 服务重启后只恢复定义与触发器，不恢复运行中的 session 执行状态；错过的 cron 窗口默认不补跑，只重新计算下一次触发时间。
  - v1 只支持“自动化运行记录 -> 普通会话”的单向跳转，不要求在会话详情页反向显示自动化徽标。
  - 若会话中出现多个 ask，则每个 ask 都按同一固定文案自动回复；若 permission 与 ask 异常并存，以 permission 失败收口为最高优先级，停止后续自动回复。
  - 首版不做多 Agent 流程、人工审批挂起、Webhook、可视化节点编排。

### 6.2 接口与契约

- API/函数签名：
  - GET /api/automations -> { automations: AutomationDefinitionSummary[] }
  - POST /api/automations -> AutomationDefinitionDetail
  - PATCH /api/automations/:automationId -> AutomationDefinitionDetail
  - POST /api/automations/:automationId/toggle -> { ok: true, enabled: boolean }
  - POST /api/automations/:automationId/run -> AutomationRunDetail
  - GET /api/automations/:automationId/runs -> { runs: AutomationRunSummary[] }
  - GET /api/automations/runs/:runId -> AutomationRunDetail
  - type AutomationTriggerConfig =
    - { type: 'manual' }
    - { type: 'cron'; expression: string; timezone: string }
    - { type: 'thread-heartbeat'; sourceSessionId: string; intervalMinutes: number; timezone: string }
    - { type: 'file-change'; root: string; include: string[]; exclude?: string[]; debounceMs: number; pollIntervalMs: number }
    - { type: 'git-event'; root: string; events: Array<'branch-change' | 'status-change'>; pollIntervalMs: number }
  - createAutomationRun(definition: AutomationDefinition, trigger: AutomationTriggerContext) -> Promise<AutomationRun>
  - dispatchAutomationRun(runId: string) -> Promise<void>
  - openAutomationSession(definition: AutomationDefinition, trigger: AutomationTriggerContext) -> Promise<{ sessionId: string }>
  - settleAutomationAsk(sessionId: string, askId: string) -> Promise<void>
  - failAutomationPermission(runId: string, sessionId: string, requestId: string) -> Promise<void>
- 错误码/异常：
  - AUTOMATION_NOT_FOUND：自动化定义不存在。
  - AUTOMATION_DISABLED：自动化已禁用但仍被触发。
  - AUTOMATION_TRIGGER_INVALID：触发器配置不合法。
  - AUTOMATION_CONCURRENT_RUN_CONFLICT：同一自动化命中并发冲突策略。
  - AUTOMATION_SESSION_CREATE_FAILED：普通会话创建失败。
  - AUTOMATION_SEND_FAILED：消息发送失败。
  - AUTOMATION_ASK_AUTOREPLY_FAILED：ask 自动回复失败。
  - AUTOMATION_PERMISSION_BLOCKED：命中权限审批，被策略阻断。
  - AUTOMATION_VALIDATION_FAILED：表单字段、trigger 配置或 replyText 校验失败。

- 前端表单约束：
  - name：必填，2-40 字符，项目内唯一。
  - cwd：必填，必须命中已管理项目根。
  - agent_name：必填，必须是当前 cwd 下可发现的 agent。
  - prompt_template：必填，去首尾空格后不能为空。
  - trigger_config_json：至少包含 1 个 trigger；同类 trigger 可多条，但 root/sourceSessionId 不允许为空。
  - cron.expression：采用五段格式 minute hour day month weekday，不支持秒字段。
  - timezone：cron 与 thread-heartbeat 必填，要求 IANA timezone；默认值取 ridge.db app_settings 中的 serverTimezone，没有则取服务启动时的系统 timezone。
  - ask_policy.replyText：必填，trim 后不能为空；全局默认值只保存在 ridge.db 的 app_settings 表，不写回 ridge-settings.json。
  - concurrency_policy：必填，只允许 skip 或 enqueue-latest。

### 6.3 数据与存储（如适用）

- Schema/字段：
  - automations：
    - id：string（主键）
    - name：string（唯一展示名）
    - description：string
    - cwd：string（所属项目根）
    - enabled：boolean
    - agent_name：string
    - explicit_model：string | null
    - explicit_thinking_level：string | null
    - prompt_template：string
    - trigger_config_json：AutomationTriggerConfig[]（数组 JSON）
    - ask_policy_json：{ mode: 'auto-reply', replyText: string }
    - permission_policy：'fail'
    - concurrency_policy：'skip' | 'enqueue-latest'
    - created_at：number
    - updated_at：number
  - automation_runs：
    - id：string（主键）
    - automation_id：string
    - trigger_type：'manual' | 'cron' | 'thread-heartbeat' | 'file-change' | 'git-event'
    - trigger_payload_json：object
    - status：'queued' | 'running' | 'completed' | 'failed' | 'interrupted' | 'skipped'
    - session_id：string | null
    - error_message：string | null
    - started_at：number
    - finished_at：number | null
  - automation_trigger_cursors：
    - automation_id：string
    - trigger_key：string
    - cursor_json：object（用于保存 cron 的 next_due_at/last_fired_at，文件监听的文件指纹，Git 事件的状态指纹）
    - updated_at：number
- 迁移策略：
  - 新增 ridge.db migration 建表，不新增旧 JSON 兼容层。
  - 首版没有历史自动化数据，无需数据迁移，只需在服务启动时保证表存在。
  - 全局默认 ask 回复文案只存放在 ridge.db 的 app_settings 表中，但每条自动化定义允许覆写，不再额外落到 ridge-settings.json。
  - concurrency_policy 语义：skip 表示当同一 automation 已有 queued/running run 时，新触发写入一条 status=skipped 的运行记录但不执行；enqueue-latest 表示始终只保留一个最新 queued run，后来的触发会覆盖旧 queued run 的 trigger_payload_json。

### 6.4 兼容性与发布

- 兼容策略：新功能从零开始，不保留旧自动化协议兼容层；自动化执行结果完全兼容现有普通会话与消息链路。
- Linux 部署策略：自动化调度器与 server 同进程部署，推荐由 systemd 拉起 Node 服务；服务启动后读取 automations 表重建内存调度队列，服务停止期间错过的 cron 不补跑。
- 发布步骤：
  - 第一步：建立 ridge.db 数据模型与服务端自动化域层。
  - 第二步：建立 scheduler 与五类 trigger adapter，但按能力优先级先打通 manual + cron。
  - 第三步：重构 AutomationsPage 为真实页面，接通列表、详情表单和运行记录。
  - 第四步：补齐 ask 自动回复、permission 失败收口和会话跳转联动。

### 6.5 可观测性

- 日志：
  - 自动化定义增删改、启停日志。
  - trigger 命中日志与去重命中日志。
  - 普通会话创建、消息发送、ask 自动回复、permission 阻断日志。
  - run 状态流转日志（queued -> running -> completed/failed/interrupted）。
- 指标：
  - automation 触发次数、成功率、失败率。
  - ask 自动回复命中次数。
  - permission 阻断次数。
  - 单项目活跃监听器数量、trigger 去重丢弃次数。
- 告警：
  - 同一自动化短时间连续失败超过阈值。
  - 同一 trigger 在去抖窗口内重复命中过多。
  - scheduler 重载失败或 automation 表读取失败。
