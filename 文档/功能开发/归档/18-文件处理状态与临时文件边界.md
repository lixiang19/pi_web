# 18 文件处理状态与临时文件边界 — 实现规格

## 目标

实现文件处理状态机和临时文件边界。让文件从上传开始就有完整的状态生命周期，失败可见、可重试，临时文件被严格隔离。

## 范围

- 文件处理任务记录 CRUD。
- 状态：待处理、转换中、已转换、已索引、转换失败、索引失败。
- `.ridge` 临时文件目录严格隔离。
- 会话临时上传不进入处理流程。
- 转换中间文件不进入 RAG。
- 失败重试缓存。
- 失败生成通知事件。
- 用户可重试失败处理。

## 不做

- 不做实际的 PDF/Word/音频转换（属于任务19-20），本任务只做状态记录和边界。
- 不做 RAG chunk 和索引（属于任务22）。
- 临时文件不进入 RAG（本任务确保，但 RAG 扫描属于后续）。
- 临时文件不进入图谱（本任务确保，但图谱属于后续）。
- 临时文件不给 MCP 读取（本任务确保，但 MCP 属于后续）。

## 验收

- 文件上传到正式目录后自动创建 `pending` 处理记录。
- 状态转换正确：`pending` → `converting` → `converted` → `indexed`。
- 转换失败时：状态变为 `convert_failed`，保留原文件，写入错误原因，生成通知事件。
- 索引失败时：状态变为 `index_failed`，保留转换产物，写入错误原因，生成通知事件。
- 用户可通过 API 重试失败处理（回到 `pending`）。
- `.ridge` 下文件不创建处理记录。
- 文件删除时同步删除 `file_processing_status` 记录。
- 前端文件页展示状态并支持重试操作。

## API

### 新增/修改

- `POST /api/files/upload`（已有）— 上传后自动创建 `pending` 记录（仅在正式目录）。
- `PATCH /api/workspace/files/status` — 更新文件处理状态（内部/管理员使用）。
  - Body: `{ path, status, error? }`
- `POST /api/workspace/files/retry` — 重试失败处理。
  - Body: `{ path }`
  - 效果：状态从 `convert_failed` / `index_failed` 回到 `pending`，清除 error。
- `DELETE /api/files/entries`（已有）— 删除文件时同步删除 `file_processing_status` 记录。

## 数据模型

- `file_processing_status(file_path PK, workspace_path, status, content_hash, converted_at, indexed_at, error, updated_at)`
  - 已存在，无需迁移。

## 通知规则

- 转换失败 → `notification_events`:
  - `event_type`: `file_processing.convert_failed`
  - `severity`: `error`
  - `title`: `文件转换失败: {filename}`
  - `body`: 错误信息
  - `payload_json`: `{ filePath, error }`
- 索引失败 → `notification_events`:
  - `event_type`: `file_processing.index_failed`
  - `severity`: `error`
  - `title`: `文件索引失败: {filename}`
  - `body`: 错误信息
  - `payload_json`: `{ filePath, error }`

## 测试要点

1. 上传文件到正式目录后，DB 有 `pending` 记录。
2. 上传文件到 `.ridge` 后，DB 无处理记录。
3. 状态更新成功。
4. 转换失败保留原文件并生成通知。
5. 索引失败保留转换产物并生成通知。
6. 重试 API 将失败状态回到 `pending`。
7. 删除文件时处理记录同步删除。
8. 临时文件读取不返回 processingStatus。
