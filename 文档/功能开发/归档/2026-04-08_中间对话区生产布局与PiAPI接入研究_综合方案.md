## 第一部分：中间对话区生产布局方案
### 1.1 背景与问题
- **背景**：Workbench 已具备会话加载、消息发送、流式更新、草稿持久化和输入编排能力，但中间对话区仍未达到生产级工作台布局。
- **现象/痛点**：头部信息密度过高、消息区沉浸感不足、输入区装饰化过重、空态表达不稳定，导致最低对话功能虽已存在，但整体体验仍像过渡态实现。
- **成功定义**：在不扩展高级能力的前提下，让中间对话区具备可真实发送、可稳定阅读、可支撑明暗主题的生产级三段式布局。
### 1.2 目标与非目标
- **目标**：
  - 完成中间对话区头部、消息区、输入区的稳定三段式布局，强化消息阅读区域。
  - 保留真实消息发送链路和现有模型、agent、thinking 选择能力，并以更紧凑的生产布局呈现。
- **非目标**：
  - 不实现权限卡片、Question 卡片、拖拽上传、自动补全、虚拟列表和移动端适配。
### 1.3 现状探索
- **入口与调用链**：
  - `packages/web/src/pages/WorkbenchPage.vue:1` - Workbench 页面入口，负责组装左侧会话栏、中间对话区、右侧文件树。
  - `packages/web/src/components/workbench/chat/WorkbenchChatPanel.vue:1` - 中间对话区直接容器，负责头部、消息流、输入区三段式拼装。
  - `packages/web/src/composables/useWorkbenchPage.ts:1` - 聚合 usePiChat、资源选择器和会话状态，向 WorkbenchChatPanel 提供对话数据与行为。
  - `packages/web/src/composables/usePiChat.ts:1` - 聊天核心链路，负责消息加载、发送、流式更新、草稿持久化。
- **关键数据结构/状态**：
  - `packages/web/src/lib/types.ts:1` - 定义 ChatMessage、ChatComposerState、SessionSummary、ThinkingLevel 等核心类型。
  - `packages/web/src/components/workbench/chat/WorkbenchMessageStream.vue:1` - 依赖 messages、status、hasMoreAbove、isLoadingOlder 驱动消息区渲染与滚动状态。
  - `packages/web/src/components/workbench/chat/WorkbenchComposer.vue:1` - 依赖 draftText、isSending、资源列表与资源可见性状态驱动输入区。
- **约束与坑点**：
  - 本次只能做最低对话功能和生产级布局，不能把高级输入、权限卡片和虚拟列表混入范围。
  - 不允许以兼容分支和兜底逻辑修补问题，只能直接收敛到新的正确布局结构。
### 1.4 方案概览
#### 1.4.1 方案摘要
- **一句话**：以现有真实对话链路为基础，重构中间对话区的结构层次和视觉节奏，把头部收缩、消息区做主、输入区稳定到底部。
- **取舍**：本次不新增能力，优先修正结构和布局，因为当前根本问题是界面骨架没有进入稳定形态，而不是发送链路不可用。
- **影响面**：会改动 Workbench 中间对话区的四个核心组件，直接影响桌面端用户的消息阅读、发送与空态体验。
#### 1.4.2 风险与回滚
- **风险**：
  - 头部控件压缩后可能挤占标题可读性：通过将标签和次级状态弱化，并让右侧控件带可换行来缓解。
  - 输入区和消息区高度分配可能失衡：通过固定输入区底部面板角色、保持消息区为 flex-1 主区域来缓解。
  - 流式消息滚动可能因容器样式变化产生体验波动：只修改容器与样式，不调整现有自动滚底核心逻辑来缓解。
- **回滚**：如布局重构引发明显回归，按组件粒度回退 WorkbenchChatPanel、WorkbenchMessageStream、WorkbenchComposer 的提交内容，不做兼容分支保留。
### 1.5 详细设计（Spec）
#### 1.5.1 交互/行为
- **用户流程**：进入 Workbench 会话页 -> 中间区显示收缩头部与消息流 -> 无消息时展示简洁空态 -> 输入文本并发送 -> 消息区展示发送中与返回结果 -> 如有历史消息可点击加载更早消息 -> 脱离底部时可点击返回到底部。
- **边界条件**：空会话需要稳定空态；发送中按钮需要切换为停止状态；长消息、多段消息、流式更新不能撑破消息容器；父会话返回入口仅在存在 parentSessionId 时显示。
#### 1.5.2 接口与契约
- **API/函数签名**：
  - `useWorkbenchPage()：() -> { composer, messages, status, submit, loadEarlier, handleModelSelection, handleThinkingSelection, handleAgentSelection, toggleResourcePicker, ... }`
  - `WorkbenchChatPanel：props -> 组合 WorkbenchChatHeader、WorkbenchMessageStream、WorkbenchComposer 的展示与事件分发`
  - `submit：() -> Promise<void>`
- **错误码/异常**：本次不新增接口和错误码，继续遵循现有发送链路的错误处理规则，界面侧只保证状态显示不破版。
#### 1.5.3 数据与存储
- **Schema/字段**：
  - ChatMessage：消息对象（包含 id、role、parts、text 等字段，用于消息区渲染）
  - ChatComposerState：输入区状态（包含 draftText、selectedModel、selectedThinkingLevel、selectedAgent）
  - SessionSummary.status：会话状态（约束消息区滚动和发送中的展示语义）
- **迁移策略**：不涉及数据迁移；仅调整现有状态在界面中的布局和呈现方式。
#### 1.5.4 兼容性与发布
- **兼容策略**：不做旧布局兼容，直接替换为新的生产级结构；仅保留现有业务能力和数据契约不变。
- **发布步骤**：修改组件 -> 本地执行类型/构建验证 -> 检查明暗主题 -> 完成功能文档与模块文档更新 -> 归档方案文档。

## 第二部分：Pi API 接入研究与最佳实践
### 2.1 背景与问题
- **背景**：当前项目正在围绕 Pi 构建 Web 工作台，需要确认 Pi SDK、扩展 API、agent 资源和会话持久化的正确接入方式。
- **现象/痛点**：如果对 Pi 的 API 边界理解不清，容易走向错误的接法，例如在 Web 层伪造会话语义、绕开资源加载器、错误使用 RPC 或把权限控制放到前端。
- **成功定义**：明确 Pi 在当前项目中的推荐接入面、资源发现机制、agent 注入策略和安全边界，并输出一套可直接指导后续实现的最佳实践。
### 2.2 目标与非目标
- **目标**：
  - 梳理 Pi SDK、扩展 API、资源系统、会话系统和 agent 系统的核心能力与适用边界。
  - 给出适用于当前仓库的接入建议和需要避免的反模式。
- **非目标**：
  - 不在本次研究中实现新的 Pi 扩展、RPC 客户端或额外后端能力。
### 2.3 现状探索
- **入口与调用链**：
  - `packages/server/src/index.js:1` - 当前服务端直接使用 `@mariozechner/pi-coding-agent` 的 AuthStorage、ModelRegistry、SessionManager、DefaultResourceLoader、createAgentSession。
  - `packages/web/src/lib/api.ts:1` - Web 端通过 /api/sessions、/api/resources、/api/agents 等接口消费服务端桥接后的 Pi 语义。
  - `packages/web/src/composables/usePiChat.ts:1` - 前端以 sessionId 为核心管理消息、草稿、流式状态和资源目录，不直接 Pi 运行时。
- **关键数据结构/状态**：
  - `packages/web/src/lib/types.ts:1` - 定义 SessionSummary、SessionSnapshot、ChatMessage、ChatComposerState 等前端投影模型。
  - `packages/server/src/agents.js:1` - 定义 agent frontmatter 解析、project 覆盖 user、thinking 与 permission 规范化。
  - `packages/server/src/agent-permissions.js:1` - 定义 permission 编译与 tool_call 写入路径 gate。
- **约束与坑点**：
  - 项目规范明确要求**只允许使用 SDK 接入，禁止 RPC 模式**。
  - Pi 的不少高级能力通过扩展与资源系统完成，不应在 Web 层重复发明另一套平行机制。
### 2.4 资料与依据（研究结论）
- **Pi SDK 核心能力地图**：
  | 能力 | API | 用途 |
  |------|-----|------|
  | 会话管理 | `SessionManager.create/open/inMemory` | 新建/打开/临时会话 |
  | Agent 会话 | `createAgentSession(options)` | 建立真实 Pi runtime 会话 |
  | 资源加载 | `DefaultResourceLoader` | 加载扩展、资源、技能 |
  | 工具注册 | `pi.registerTool(tool)` | 为 LLM 提供可调用的工具 |
  | 命令注册 | `pi.registerCommand(name, options)` | 添加斜杠命令 |
  | 事件监听 | `pi.on(event, handler)` | 拦截会话生命周期事件 |
  | 消息发送 | `pi.sendMessage/sendUserMessage` | 向会话注入消息 |
  | 状态持久化 | `pi.appendEntry(type, data)` | 跨会话持久化扩展状态 |
  | 权限控制 | `tool_call` 事件拦截 | 在服务端 gate 层限制 agent 能力 |
- **当前仓库桥接验证**：
  - `packages/server/src/index.js:1` - 正确导入 `AuthStorage`、`ModelRegistry`、`SessionManager`、`DefaultResourceLoader`、`createAgentSession`
  - `packages/server/src/index.js:322-342` - 通过 `DefaultResourceLoader` 构造资源加载器，挂接 `appendSystemPromptOverride` 注入 agent systemPrompt，通过 `extensionFactories` 注入权限 gate
  - `packages/server/src/index.js:500-560` - 通过 `applySessionAgentSelection` 实现 agent/model/thinking 的解析和设置
  - `packages/server/src/agent-permissions.js` - 实现 `compileAgentPermission` 与 `createPermissionGateExtension`，在 `tool_call` 事件层拦截权限
- **最佳实践提炼**：
 1. **SDK 模式是唯一选择**：项目规范已明确禁止 RPC，坚持使用 `createAgentSession` + `SessionManager`
 2. **权限控制必须在服务端**：通过 `tool_call` 事件 gate 拦截，而非前端按钮显隐
 3. **Agent 配置通过 resourceLoader 注入**：利用 `appendSystemPromptOverride` 注入 systemPrompt，用 `extensionFactories` 挂载权限扩展
 4. **前后端职责分离**：server 负责 Pi runtime 与权限控制，web 仅消费投影后的稳定接口
- **反模式清单**：
  - ❌ 在 Web 层伪造会话语义或重复实现资源发现
  - ❌ 使用 RPC 模式（SDK 不可用时才考虑）
  - ❌ 把权限判断放在前端
  - ❌ 直接依赖 Pi 内部私有字段
### 2.5 方案概览
#### 2.5.1 方案摘要
- **一句话**：继续坚持以 Pi SDK 为核心的服务端桥接方案，把 Pi 作为会话、资源、agent 和扩展运行时，Web 只消费投影后的稳定接口，不直接模拟 Pi 内部机制。
- **取舍**：这样做比 RPC 和前端伪实现更重一些，但它保留了 Pi 的真实会话语义、资源加载、agent 注入和权限模型，长期维护成本更低。
- **影响面**：影响 server 侧的 Pi runtime 接入方式、agent/资源目录策略、权限控制落点，以及 web 侧的接口设计边界。
#### 2.5.2 风险与回滚
- **风险**：
  - 直接依赖私有内部能力会增加 SDK 升级风险：应尽量避免依赖未文档化的内部字段或私有方法。
  - 在前端重复实现资源发现、agent 解析或权限逻辑会导致双源真相：应统一收敛到服务端基于 Pi SDK 的实现。
- **回滚**：若某次 Pi SDK 升级导致桥接不兼容，优先收敛到官方公开 API 能力，移除私有内部调用，而不是切换到 RPC 或前端兜底。
### 2.6 详细设计（Spec）
#### 2.6.1 交互/行为
- **用户流程**：Web 前端请求会话与资源目录 -> server 基于 SessionManager 与 DefaultResourceLoader 构建或打开 Pi session -> 前端通过 HTTP/SSE 消费投影后的会话与消息流 -> 用户发送消息时 server 调用 session.prompt() 驱动真实 Pi 运行时。
- **边界条件**：资源目录获取在无现成 session 时可使用 inMemory 临时 session；agent 权限限制必须在 tool_call 事件层拦截；项目级 agent 需要覆盖用户级同名 agent。
#### 2.6.2 接口与契约
- **Pi SDK API/函数签名**：
  - `createAgentSession(options)`：Pi SDK 建立真实 agent session 的入口。
  - `DefaultResourceLoader({ cwd, settingsManager, appendSystemPromptOverride, extensionFactories })`：Pi 资源与扩展加载器。
  - `pi.registerTool(tool)、pi.registerCommand(name, options)、pi.on(event, handler)`：Pi 扩展 API 三个核心入口。
  - `SessionManager.create(cwd) / SessionManager.open(path) / SessionManager.inMemory(cwd)`：持久化会话、新建会话和临时会话的推荐接口。
- **错误码/异常**：
  - 权限限制应在 tool_call 层返回 block/reason，而不是前端先行判断。
  - agent frontmatter 非法、permission schema 非法、steps 耗尽都应在 server 层明确返回错误。
#### 2.6.3 数据与存储
- **Schema/字段**：
  - SessionSummary：前端消费的会话投影，包含 id、title、status、projectRoot、worktreeRoot、agent、model、thinkingLevel 等。
  - AgentSummary：agent 的前端摘要，包含 name、mode、model、thinking、steps、sourceScope。
  - permission：以 allow/deny 与 edit 路径规则为核心的 agent 能力约束结构。
- **迁移策略**：不迁移 Pi 原生 session 文件；仅允许在服务端持久化额外元数据，例如归档状态和显式选择。
#### 2.6.4 兼容性与发布
- **兼容策略**：坚持 SDK 路线，不引入 RPC 兼容层；若扩展 API 需要 richer UI，也优先走 server 侧 extensionFactories 注入与 Web 投影，而不是换协议。
- **发布步骤**：确认 server 仅依赖公开 SDK 能力 -> 校验 agent/资源桥接接口 -> 前端仅消费投影接口 -> 跟随 Pi 版本升级时做回归验证。
#### 2.6.5 可观测性
- **日志**：agent 发现失败、permission 编译失败、resourceLoader.reload() 失败、session.prompt() 异常都应在 server 层保留日志。
- **指标**：关注会话打开成功率、资源目录加载成功率、消息发送成功率、SSE 流持续性。
- **告警**：Pi SDK 升级后如出现资源目录为空、agent 注入失效或消息流中断，应视为高优先级回归。
