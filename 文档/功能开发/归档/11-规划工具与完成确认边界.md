# 11 规划工具与完成确认边界

## 目标

实现 Agent 可用的规划工具，并严格禁止 Agent 完成任务或里程碑。

## 范围

- 创建任务。
- 更新任务。
- 创建里程碑。
- 更新里程碑。
- 移动任务到里程碑。
- 设置阻塞。
- 设置 reviewing。
- 工具权限纳入 `task` 权限。

## 不做

- 不支持 Agent 设置 completed。
- 不支持批量隐式修改多个任务状态。

## 验收

- Agent 尝试完成任务时失败并返回明确原因。
- 工具成功只返回必要结果。
- 工具失败返回可自我修正的原因。
- 多任务更新必须逐项明确。

## 实现

- 后端新增 `packages/server/src/planning-tools.ts`，提供 `createPlanningToolExecutors`（可单元测试）和 `createPlanningToolsExtension`（注册 7 个 Pi 工具）。
- 7 个工具：`create_task`、`update_task`、`create_milestone`、`update_milestone`、`move_task`、`set_blocked`、`set_reviewing`。
- 所有工具拒绝批量/隐式多任务输入（`tasks`、`task_ids`、`milestones`、`milestone_ids`、数组形式的 `task_id`/`milestone_id`）。
- `update_task` / `update_milestone` 在 schema 层面排除 `completed`，并在执行层显式抛错说明“完成必须由用户人工确认”。
- `set_blocked` 校验当前状态必须是 `in_progress`，保存 `blocked_reason`。
- `set_reviewing` 校验当前状态必须是 `in_progress`，流转到 `reviewing`。
- `create_task` 未指定 `milestone_id` 时挂到默认里程碑，未指定 `project_id` 时继承里程碑项目。
- 权限：`agent-permissions.ts` 中 7 个工具名映射到逻辑权限 `task`；`task: deny` 时从 `activeToolNames` 移除规划工具；`task: ask` 时生成 `permissionRequest`（subject/pattern 均为工具名本身）。
- 会话接入：`session-context.ts` 的 `createSessionResourceLoader` 注册 `createPlanningToolsExtension(deps.defaultWorkspaceDir)`，所有 Agent 会话默认可用（受权限控制）。

## 测试

- 新增 `packages/server/src/__tests__/planning-tools.test.ts`（24 项测试）：
  - 成功创建任务/里程碑并返回必要字段（id, title, status, milestoneId/projectId）。
  - `update_task` / `update_milestone` 尝试 `completed` 的明确错误。
  - 批量/隐式多任务输入被拒绝。
  - `set_blocked` 保存 blockedReason 且状态流转合法，`set_reviewing` 合法推进到 reviewing。
  - 权限：规划工具映射为 `task`；`task: deny` 移除规划工具；`task: ask` 保留；默认无配置保留。

## 边界

- 规划工具不替代 HTTP API（`workspace-tasks.ts`），它们是 Agent 侧写入通道。
- 底层 completed 禁止仍由 `task-system.ts` 的 `assertStatusAllowed` 兜底；规划工具在执行层做前置校验，返回 Agent 可读的错误文本。
- 子代理 `task` 工具（子代理启动）与规划工具共享同一个 `task` 逻辑权限键。
