## 1. 背景与问题

- 背景：现有 ridge 对话链路同时依赖 Pi session jsonl、`~/.pi/ridge-settings.json`、`.pi-web/session-sidebar.json`、`activeSessions` 内存态。列表、上下文、运行时、配置持久化都靠接口层临时拼装，导致热路径越来越重。
- 现象/痛点：
  - 服务端 `packages/server/src/index.ts:1167-1456` 的 `listProjectSessions()`、`toSessionSummary()`、`toSessionSnapshot()` 同时承担扫描、归属解析、树关系、metadata 融合、消息窗口拼装，职责过肥。
  - `packages/server/src/session-metadata.ts:76-166` 与 `packages/server/src/storage/index.ts:59-183` 分别用 JSON 文件存会话 metadata、settings/projects/favorites，形成多份文件式伪数据库。
  - 左栏 `GET /api/sessions` 既承担会话列表，又隐含 project/worktree/context/runtime 字段，导致每次启动和刷新都在重复拼整条链。
  - 单会话快照仍把 summary/messages/runtime/pending requests 混成一个大对象，前端 `packages/web/src/composables/usePiChatCore.ts:291-303`、`usePerSessionChat.ts:326-368` 围绕肥快照耦合很深。
  - 中间区 `packages/web/src/components/workbench/chat/WorkbenchMessageStream.vue:140-198` 每次消息更新都重算轮次布局，且 `301-312` 仍有 `JSON.stringify(last message content)` 热路径坏味道。
  - Pi session 文件本身已经存有 `model_change` 与 `thinking_level_change` 事件，外部真实样本 `/Users/lixiang/.pi/agent/sessions/--Users-lixiang-Documents-myCode-AuroraPlatformWeb--/2026-04-13T18-01-56-476Z_efbcbff9-78f8-45a8-bf5e-771952047570.jsonl:2-5` 可证；现有 bridge 仍重复把部分选择值写入 JSON metadata，边界混乱。
  - 用户已明确：保留 `v-show` 与标签常驻秒切，不接受通过卸载标签或缓存补丁解决；需要从存储分层和接口边界根治。
- 成功定义：建立 `~/.pi/ridge.db` 作为 ridge 自己统一查询索引库与配置库，删除散落 JSON 配置主存储，拆分对话链路接口，让列表/上下文/运行时/消息各走各源，同时保留现有标签常驻秒切体验。

## 2. 目标与非目标

- 目标：
  - 引入 `SQLite + better-sqlite3`，统一持久化 ridge 自己的全局配置、项目、收藏、会话目录索引、上下文投影、会话选择信息。
  - 删除 `~/.pi/ridge-settings.json` 与 `.pi-web/session-sidebar.json` 作为运行时主存储，改为一次性迁移到 `~/.pi/ridge.db` 后由 DB 单独承担。
  - 拆分对话接口为 `sessions / session-contexts / messages / runtime` 四类职责，列表与上下文直接查 DB，消息继续以 Pi jsonl 为真源，runtime 由 Pi session + selection + active runtime 组装。
  - 保留 `v-show`、全部标签常驻、右侧文件/Git 面板常驻，不通过卸载标签解决性能。
  - 优化中间区流式热路径，去掉高频序列化监听和大范围重算。
- 非目标：
  - 不把 Pi session jsonl 替换成 ridge DB；Pi jsonl 仍是消息与 session 事件真源。
  - 不把消息正文入库；DB 只做查询索引，不做消息正文仓库。
  - 不做 ask/permission 崩溃恢复；pending 继续留在 runtime 内存。
  - 不改 Tauri 角色，不引入外置服务型数据库，不做 Postgres/远程同步。
  - 不把浏览器瞬时 UI 状态（输入草稿、展开态、resource picker 开关等）搬入 DB。

## 5. 方案概览（主方案）

### 5.1 方案摘要

- 一句话：引入 `~/.pi/ridge.db` 作为 ridge 统一配置库与会话索引库，删除散落 JSON 配置文件，服务端改成“DB 查列表/上下文 + Pi jsonl 读消息 + runtime 组装执行配置”的分层架构。
- 取舍：放弃“所有状态都用 JSON 文件拼出来”的轻量假象，换取清晰的存储分层；同时明确 DB 不存消息正文，避免复制消息与高频流式写库。
- 影响面：
  - 服务端：新增 DB 层、索引器、消息读取器、runtime 服务；重构 `packages/server/src/index.ts`。
  - 配置存储：退役 `packages/server/src/storage/index.ts` 现有 JSON 文件实现与 `session-metadata.ts` JSON metadata 实现。
  - 前端 API：`packages/web/src/lib/api.ts` 与相关 types 改用新契约。
  - 左栏：改为 DB 驱动的 `sessions + session-contexts` 双源拼装。
  - 中间区：消息接口独立读取；runtime 恢复独立读取；热路径重算收敛。

### 5.2 风险与回滚

- 风险：
  - DB schema 设计过宽，重新制造伪数据库：通过明确“DB 不存消息正文，只存配置与索引”缓解。
  - 删除 JSON 配置文件后，迁移脚本若漏字段会导致设置/项目/收藏/archived 丢失：通过一次性迁移器 + schema 校验 +开发环境手动验证缓解。
  - `messages` 不入 DB 后，单会话消息读取若仍走旧快照逻辑，会导致接口虽拆但实现未根治：通过新增独立 `session-message-reader`，禁止 `messages` 路由复用 `toSessionSnapshot()` 缓解。
  - runtime 来源仍混乱会导致头部选择器恢复错误：通过明确 `agent` 来自 `session_selections`，`model/thinking` 来自 Pi session / opened session，`resolved*` 来自 active runtime 缓解。
- 回滚：
  - 回滚策略为 Git 版本回退。
  - DB 为查询索引与配置库；会话消息真源仍在 Pi jsonl，因此代码回退后可重新执行旧路径或重新构建 DB。
  - 不保留 JSON 配置与 DB 双轨运行；迁移失败时直接停止、修复、重建 DB，不做运行时兼容分支。

## 6. 详细设计（Spec）

### 6.1 交互/行为（如适用）

- 用户流程：
  1. 服务启动后打开 `~/.pi/ridge.db`，执行 migration。
  2. 若发现 `~/.pi/ridge-settings.json` 或 `.pi-web/session-sidebar.json`，一次性导入其 ridge 自有配置到 DB，然后删除旧文件。
  3. 启动时扫描 Pi session 文件，构建/刷新 `sessions` 与 `session_contexts` 目录投影表，不读取消息正文窗口。
  4. 左栏启动后请求 `GET /api/sessions` 与 `GET /api/session-contexts`，前端按 `contextId` 拼 project/worktree 视图。
  5. 用户打开会话时，`GET /api/sessions/:id/messages?rounds=3` 根据 `session_file` 直接读 Pi jsonl / opened session，按 rounds 返回消息窗口。
  6. 头部选择器请求 `GET /api/sessions/:id/runtime`：`agent` 来自 DB 选择表，`model/thinking` 来自 Pi session / opened session，`resolved*` 来自 active runtime。
  7. 流式时 UI 仍靠 SSE 和内存态；`message_end` 后服务端只更新 `sessions.updated_at / last_message_preview / user_round_count / message_count` 等目录投影。
- 边界条件：
  - 标签仍全部常驻；切换标签不能通过卸载来重置消息区或右侧文件/Git 面板。
  - 若 DB 中 session 目录存在但 `session_file` 丢失，视为索引损坏；路由返回明确错误并要求重建索引。
  - 若 `contextId` 无法映射 `session_contexts`，视为索引器或 project-context 逻辑错误，不由前端兜底猜测。
  - 若 active runtime 不存在，`runtime` 接口必须能退回“Pi session + selection”模式，而不是依赖内存态。
  - ask/permission pending 继续留内存，不做重启恢复；服务重启后 pending 清空是 V1 允许行为。

### 6.2 接口与契约

- API/函数签名：
  - `GET /api/sessions`：无入参 -> `SessionListItem[]`
    - 字段：`id, title, status, updatedAt, archived, parentSessionId, contextId, lastMessagePreview?, userRoundCount, messageCount`
  - `GET /api/session-contexts`：无入参 -> `Record<contextId, SessionContextSummary>`
    - 字段：`contextId, cwd, projectId, projectLabel, projectRoot, worktreeRoot, worktreeLabel, branch, isGit`
  - `GET /api/sessions/:id/messages?rounds=3`：`sessionId, rounds?` -> `SessionMessagesPayload`
    - 字段：`sessionId, messages, historyMeta, interactiveRequests, permissionRequests`
    - 实现：直接读 Pi jsonl / opened session，不读 DB message body。
  - `GET /api/sessions/:id/runtime`：`sessionId` -> `SessionRuntimePayload`
    - 字段：`sessionId, agent, model, thinkingLevel, resolvedModel, resolvedThinkingLevel`
  - `GET /api/settings` / `PATCH /api/settings`：保持现有 settings 契约，底层改查 `app_settings`
  - `GET /api/projects` / `POST /api/projects` / `DELETE /api/projects/:id`：底层改查 `projects`
  - `GET /api/favorites` / `POST /api/favorites` / `DELETE /api/favorites/:id`：底层改查 `favorites`
  - `GET /api/sessions/:id/stream?rounds=3`：SSE 首包改为仅含 `sessionId, status, messages, historyMeta, interactiveRequests, permissionRequests`
- 错误码/异常：
  - `404 Session not found`：session 目录不存在或索引未找到。
  - `410 Session file missing`：DB 目录存在但会话文件已被删除。
  - `400 Session context missing`：`contextId` 或 context 投影缺失。
  - `500 Ridge DB unavailable`：SQLite 初始化或查询失败。
  - `500 Session index stale`：投影状态损坏，需要重建目录表。
