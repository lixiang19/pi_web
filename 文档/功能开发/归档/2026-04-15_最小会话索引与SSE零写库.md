# 2026-04-15 最小会话索引与 SSE 零写库

- 日期：2026-04-15
- 状态：已完成
- 目标：收缩 ridge.db 的会话目录索引，删除无价值字段，并让 SSE 链路完全退出数据库写入。

## 1. 背景

上一版会话索引把数据库从“配置库”扩成了“配置 + 会话索引 + 中间优化状态”混合体：

- `sessions` 同时保存目录字段、运行时字段、消息摘要字段
- `projection_state` 只服务增量优化，但没有形成不可替代的用户价值
- 发送消息前和 `turn_end` 都会同步写 SQLite，把 SSE 热路径和目录索引维护绑在一起

这会带来两个问题：

1. 数据库职责边界变浑浊，字段越来越多，但部分字段没有消费方。
2. 流式对话链路被同步写库污染，性能问题会直接落在用户输入到首 token 的路径上。

## 2. 本次整改

### 2.1 会话索引最小化

`sessions` 表收缩为最小目录索引，只保留：

- `session_id`
- `title`
- `cwd`
- `session_file`
- `parent_session_path`
- `parent_session_id`
- `created_at`
- `updated_at`
- `archived`
- `context_id`
- `user_round_count`
- `last_model`
- `last_thinking_level`

删除字段：

- `status`
- `last_message_preview`
- `message_count`
- `indexed_at`

保留理由：

- `user_round_count` 用于历史按轮分页
- `last_model` / `last_thinking_level` 用于未打开会话的 runtime 恢复

删除理由：

- `last_message_preview` 当前没有任何消费方
- `message_count` 当前没有任何消费方
- `status` 属于运行时态，应只由内存 `activeSessions` 覆盖
- `indexed_at` 只是内部优化标记，没有真实消费

### 2.2 删除 projection_state

`projection_state` 被整体移除。

原因：

- 它只用于避免重复读文件的增量优化
- 当前 Pi 会话规模不大，启动时重扫 session 文件成本可接受
- 保留它只会继续扩大数据库复杂度和维护面

目录索引改为：

- 启动时全量 `refreshSessionCatalog`
- 用户显式结构变更时做局部 upsert 或直接更新

## 3. SSE 链路调整

### 3.1 删除发送前写库

`POST /api/sessions/:id/messages` 不再在 prompt 前调用 `persistSessionRecordMetadata`。

结果：

- 发送消息本身不再成为数据库写入口
- agent/model/thinking 的显式持久化继续由 `PATCH /api/sessions/:id` 这类用户意图路径承担

### 3.2 删除 turn_end 写库

会话 runtime 在 `turn_end` 后不再调用：

- `persistSessionRecordMetadata`
- `syncIndexedSessionFromRecord`

结果：

- `turn_end` 只更新内存态并发回 snapshot/status
- SSE 链路彻底零写库
- 重启后的目录一致性由启动 refresh 恢复

## 4. 保留的真实收益

本轮没有彻底删除目录索引，而是保留了仍有明确收益的能力：

- 左栏列表不需要每次重新扫描全部 session 文件并拼 project/worktree 归属
- project/worktree 上下文字典仍能单独查询
- 父子会话树归档/删除仍可通过索引快速处理
- 未打开会话仍可恢复显式选择和最近运行时模型/思考级别

## 5. 验证

已执行：

- `npm run lint`
- `npm run build:server`
- `npm run typecheck`

结果：全部通过。

## 6. 结论

本次改动后，ridge.db 的职责被收回到：

- 配置
- 最小会话目录索引
- 显式会话选择

它不再承担：

- 无消费的消息摘要缓存
- 纯优化性质的 projection 中间状态
- SSE 期间的同步写库职责
