# 子代理能力补全计划

- 日期：2026-04-28
- 状态：已完成
- 关联分析：[子代理系统对比分析](./2026-04-28_子代理系统对比分析_pi-subagents-vs-task.md)
- 关联方案：[内嵌子代理运行时方案](./2026-04-11_内嵌子代理运行时_方案.md)
- 涉及文件：`packages/server/src/subagents.ts`

## 背景

对比 `@tintinweb/pi-subagents` 后，项目 task 子代理在权限体系上有明显优势，但在运行时管理与输出质量上存在 4 项短板。本次补全聚焦这 4 项，不引入过度设计。

## 引入清单

| #   | 能力                     | 优先级 | 说明                                                  |
| --- | ------------------------ | ------ | ----------------------------------------------------- |
| 1   | **优雅关闭**             | P0     | max_turns 到达时先 steer 提醒收尾，给宽限期，再 abort |
| 2   | **会话恢复**             | P1     | 已完成子代理可 resume 复用 session 继续追问           |
| 3   | **并发队列**             | P1     | 可配置并发上限，超出排队                              |
| 4   | **inherit_context 增强** | P1     | 父会话对话完整 fork 注入子代理，而非仅截取文本片段    |

## 详细设计

### 1. 优雅关闭（Graceful Max Turns）

**现状**：到达 max_turns 直接 `session.abort()`

```
while (true) {
  turnCount++
  if (maxTurns && turnCount >= maxTurns) {
    session.abort()  // 一刀切
  }
}
```

**目标**：三阶段关闭

```
turnCount >= maxTurns
  → steer("请立即收尾，给出当前结果")     // 第1阶段
  → 继续等待最多 3 轮                    // 第2阶段（graceTurns=3）
  → 仍未完成则 abort                      // 第3阶段
```

**改动点**：

- `runPromptWithTurnBudget` 在单次 prompt 运行期内记录 `turnCount/graceTurnsUsed`
- `runChildSession` 中修改 abort 逻辑为三阶段，强制停止后状态标记为 `steered`
- 允许在 agent frontmatter 中配置 `grace_turns`（默认 3）

### 2. 会话恢复（Resume）

**现状**：`task` 工具无 `resume` 参数

**目标**：传入 `resume: "sessionId"` 复用已有 child session

```
Agent: task({
  agent: "explore",
  prompt: "再补充一下搜索结果",
  resume: "abc-123"   // 指向已完成子代理
})
```

**改动点**：

- `task` 工具增加 `resume` 参数
- 校验：sessionId 存在、且属于当前父会话
- 若 session 已完成或 `steered`，通过原 `AgentSession.prompt()` 继续
- session 状态从 `completed` 变为 `running`

### 3. 并发队列

**现状**：无并发控制

**目标**：可配置上限，超出自动排队

```ts
const queue = new Map()           // waiting queue
const running = new Set()         // currently running
const MAX_CONCURRENT = 4          // configurable

spawn(entry):
  if running.size >= MAX_CONCURRENT:
    queue.set(entry.sessionId, entry)  // 排队
  else:
    running.add(entry.sessionId)
    execute(entry)

onComplete(entry):
  running.delete(entry.sessionId)
  if queue.size > 0:
    next = queue第一个
    move from queue to running
    execute(next)
```

**改动点**：

- `childRegistry` 旁边加 `runningChildSessions` 与 `queuedChildRuns`
- `task` 后台模式返回 `queued` 状态
- 允许通过环境变量 `RIDGE_SUBAGENT_MAX_CONCURRENT` 覆盖最大并发数

### 4. inherit_context 增强（Fork Parent Conversation）

**现状**：`inherit_context` 取最近 12 条消息拼成纯文本块塞进 system prompt

```ts
// subagents.ts
const sourceMessages = record.session.messages.slice(-12);
// → 拼成 [User]...[Assistant]...[Tool]... 文本块
// → 追加到 system prompt 末尾
```

**问题**：拼接成文本丢失了消息结构，子代理看到的是一大段无格式文本，而非真正的对话。

**目标**：Fork 父会话消息作为子代理 session 的初始 messages

```
父会话 messages（最近N条）
  → 注入为子代理 session 的真实 messages（非文本拼接）
  → 子代理看到的是结构化对话，而非扁平文本
```

**改动点**：

- `runChildSession` 中，在 `createAgentSession()` 之前，将父会话最近消息作为 child `SessionManager` 历史注入
- 不再用 `buildInheritedContextBlock` 拼文本，改为直接 push 消息到 session
- `inherit_context` 行为保持不变（仍是 opt-in），仅改变底层实现方式

## 实施顺序

```
M1: 优雅关闭 → 缓解 max_turns 截断，最小改动最大收益
M2: 会话恢复 → 解决已完成子代理不可追问的问题
M3: 并发队列 + inherit_context 增强 → 架构补强
```

## 验收标准

### M1 验收

- [x] 子代理到达 max_turns 后收到 steer 消息
- [x] 宽限期内可继续执行并正常返回结果
- [x] 超出宽限期后 abort，状态标记为 `steered`（非 `error`）
- [x] agent frontmatter 支持 `grace_turns` 配置

### M2 验收

- [x] `task({ resume: sessionId })` 复用已有 child session
- [x] 非亲生子代理 session 不可 resume（403）
- [x] 运行中子代理不可 resume（与 steer 区分）
- [x] 恢复后 child session 状态回到 running

### M3 验收

- [x] 后台子代理超过并发上限时进入排队
- [x] 运行中完成一个后自动启动下一个排队任务
- [x] 最大并发数可配置
- [x] `inherit_context` 将父会话消息作为 session 消息注入，非文本拼接
- [x] 子代理看到的对话保留原始消息结构（User/Assistant/Tool）
- [x] 未开启 `inherit_context` 时行为不变

## 实施记录

- 子代理状态新增 `queued` 与 `steered`，排队态可通过 `get_subagent_result` 查询，`steered` 表示达到 max_turns 后已收尾或被宽限期终止。
- `task` 新增 `resume`，只允许当前父会话恢复自己已结束的子代理；运行中和排队中的子代理继续使用 `steer_subagent`。
- `inherit_context` 不再拼接 system prompt 文本，而是把父会话最近 12 条原始 `user/assistant/toolResult` 消息注入子会话历史。
- 检测：`pnpm -s --filter @pi/server typecheck`、`pnpm -s exec eslint .`、`pnpm -s typecheck` 均通过。
