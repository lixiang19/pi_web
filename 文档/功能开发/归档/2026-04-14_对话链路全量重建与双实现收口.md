# 2026-04-14 对话链路全量重建与双实现收口

- 日期：2026-04-14
- 状态：已完成
- 目标：消除对话链路中两处根本性错误
  - 服务端写路径的全量目录重建
  - 前端会话快照装配的双实现并行

## 1. 背景

本轮索引库重构虽然已经把读路径从旧聚合链路中拆开，但仍残留两处会把架构重新拉回错误方向的实现：

1. 会话创建、更新、归档、删除后仍直接调用全量 session catalog refresh。
2. usePiChat 与 usePiChatCore 各自维护一套近乎平行的 snapshot 构建、cache patch、hydrate 流程。

这两处都不是“小优化没做”，而是明确的架构错误：

- 第一处会让写路径重新按全量会话规模扩张，projection_state 与增量索引能力形同虚设。
- 第二处会让新协议在两个 composable 中分别漂移，最终回到“修一处漏一处”的状态。

## 2. 根因

### 2.1 服务端根因

服务端已经具备 sessions、session_contexts、projection_state、session_selections 四类持久化结构，也有 syncIndexedSessionFromRecord 这样的局部同步能力，但 create/update/archive/delete 路由仍沿用“写完就 refresh 全目录”的旧思维。

这意味着：

- create/update 这种单条会话写操作，会重新扫描全部 session 文件。
- archive/delete 这种纯 DB 行级变更，也会触发全量目录重建。
- projection_state 虽然存在，但写路径仍默认全量重算。

### 2.2 前端根因

前端在 split APIs 已成定局后，没有把 snapshot 组装逻辑收敛成单一实现，而是让 usePiChat 和 usePiChatCore 各自保留：

- buildSnapshotFromSummary
- buildSnapshotFromPayloads
- patchSessionSnapshot
- patchSessionSummary
- hydrateSession

结果是：

- 同一份接口契约要在两处同步演进。
- context/runtime 字段很容易一边补、一边漏。
- 任何后续字段调整都会再次制造维护分叉。

## 3. 本次整改

### 3.1 服务端整改

#### 新增单会话增量 upsert

在 session-indexer 中新增单会话索引写入能力：

- resolveIndexedSessionContextEntry
- updateParentSessionIds
- upsertIndexedSessionRecord

它的职责是：

- 只为当前会话解析 managed project/context
- 只 upsert 当前 session 行、context 行、projection_state
- 只修正当前会话的 parent_session_id

#### 删除写路径全量重建

以下路由不再调用 refreshSessionCatalogIndex：

- POST /api/sessions
- PATCH /api/sessions/:sessionId
- POST /api/sessions/:sessionId/archive
- DELETE /api/sessions/:sessionId

新的行为是：

- create/update：persist metadata 后直接 upsertIndexedSessionRecord
- archive：只更新 sessions.archived
- delete：只删除 session/projection/selection 行与文件

保留全量 refresh 的场景只剩：

- 启动时初始化目录索引
- project add/remove 这类 managed scope 边界变化

### 3.2 前端整改

#### 抽出共享快照模块

新增共享模块：

- packages/web/src/composables/session-snapshot.ts

统一承载：

- buildSnapshotFromSummary
- buildSnapshotFromPayloads
- syncSessionSnapshot
- patchSessionSnapshot
- patchSessionSummary
- hydrateSessionSnapshot

#### 收敛 usePiChatCore / usePiChat

两个 composable 不再分别维护自己的 snapshot/hydrate 实现，而是统一调用共享模块。

这次收敛后：

- context 字段并入 snapshot 的逻辑只存在一处
- messages/runtime split API 的 hydrate 流程只存在一处
- cache patch / summary patch 的基础语义只存在一处

#### 删除无意义刷新

工作台归档入口不再额外刷新 session-contexts，因为归档并不会改变 context 拓扑。

## 4. 结果

### 4.1 服务端结果

- 单会话写操作不再退回全量目录扫描。
- session index 的“增量写入”与“全量重建”边界被明确分开。
- projection_state 不再被写路径自己绕开。

### 4.2 前端结果

- 会话快照装配不再存在两套近乎平行实现。
- split APIs 的协议演进点收敛到共享模块。
- context/runtime/messages 的组合语义不再在两个 composable 中漂移。

## 5. 验证

本次整改后已执行：

- npm run build:server
- npm run lint
- npm run typecheck

结果：全部通过。

## 6. 后续约束

后续继续改对话链路时，必须遵守下面两条：

1. 单会话写操作默认只能做局部索引更新，除非 managed project 边界发生变化，否则不得调用全量 refresh。
2. 任何 SessionSnapshot / SessionMessagesPayload / SessionRuntimePayload 的协议调整，只能改共享快照模块，禁止在 usePiChat 与 usePiChatCore 各自补一份。
