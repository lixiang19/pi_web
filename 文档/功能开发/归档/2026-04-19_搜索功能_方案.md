# 搜索功能方案

- 状态：已确认
- 日期：2026-04-19
- Owner：GitHub Copilot
- 目标版本/里程碑：当前工作台迭代

## 1. 背景与问题

- 背景：左侧一级导航已经有“搜索”入口，但右侧 SearchPage 仍是占位壳，只记录输入状态，无法真正检索会话。
- 现象/痛点：用户点击左侧搜索后，不能按会话标题检索历史会话，也无法从搜索结果直接回到目标会话。
- 成功定义：用户从左侧搜索入口进入独立搜索页，输入关键词并按 Enter 后，可在同页看到全部会话标题的匹配结果并直接打开会话。

## 2. 目标与非目标

- 目标：
  - 落地独立搜索页的真实会话标题搜索能力。
  - 搜索范围覆盖当前工作台中的全部会话，包括聊天系统项目、普通项目、worktree 与归档会话。
  - 搜索结果使用扁平列表展示，并复用现有会话打开动作。
- 非目标：
  - 不实现文件、命令、资源等多源统一检索。
  - 不新增服务端搜索接口或全文索引库。
  - 不在左侧侧栏重新加入旧的本地过滤输入框。

## 3. 现状探索（Explorer）

- 入口与调用链：
  - packages/web/src/composables/useWorkbenchPrimaryNavigation.ts:10 - 左侧一级导航已定义 search 路由入口。
  - packages/web/src/router/index.ts:35 - search 路由已挂到 SearchPage。
  - packages/web/src/pages/SearchPage.vue:1 - 当前页面只有输入框和说明文案，没有搜索结果与打开动作。
  - packages/web/src/composables/usePiChatCore.ts:510 - 启动时会一次性拉取全部会话列表数据到 core.sessions。
  - packages/web/src/composables/useWorkbenchPrimaryNavigation.ts: - openChatSession 已承载“切到 chat 并激活会话”的统一动作。
- 关键数据结构/状态：
  - packages/web/src/lib/types.ts:1 - SessionSummary 类型由 protocol 暴露，SearchPage 可直接消费 core.sessions。
  - packages/web/src/lib/session-sidebar.ts:29 - 侧栏项目树已经支持 query 过滤，但当前界面没有暴露搜索输入，且结果形态仍是分组树。
- 约束与坑点：
  - 搜索交互必须保留为独立页面，不能把旧的左侧过滤输入框加回来。
  - 首版要求按 Enter 触发，不做输入即搜。
  - 结果需要跨全部会话统一排序与展示，不能受左侧项目树分组限制。

## 4. 资料与依据（Librarian）

- 关键结论：
  - 现有 SearchPage 已是正式主路由，可直接扩展为真实搜索页，无需新增页面骨架。
  - core.sessions 在前端启动后即可拿到全部会话列表数据，首版按标题搜索可直接在前端完成，不必扩展后端接口。
  - 会话打开动作应统一复用导航组合层，避免在 SearchPage 自己拼路由和 LRU 激活逻辑。

## 5. 方案概览（主方案）

### 5.1 方案摘要

- 一句话：把 SearchPage 从占位状态页改造成“输入关键词并按 Enter 后，基于 core.sessions 做标题匹配并展示结果”的独立搜索页。
- 取舍：首版只做会话标题搜索，直接复用现有会话列表数据和导航动作，先把入口闭环打通，再为后续扩展多源搜索预留页面结构。
- 影响面：会改动搜索页组件、可能新增轻量搜索结果展示组件与测试；用户将能从左侧搜索入口完成真实会话检索。

### 5.2 风险与回滚

- 风险：前端本地过滤在会话数很多时可能有渲染压力；缓解措施是首版仅在按 Enter 后执行，并只匹配标题字段。
- 风险：如果 SearchPage 直接依赖过多工作台状态，后续扩展多源搜索会变乱；缓解措施是把“查询状态”和“结果计算”收敛在页面内部或拆成独立组合函数。
- 回滚：若实现不符合预期，可回退为当前 SearchPage 占位态，不涉及数据迁移和后端协议变更。

## 6. 详细设计（Spec）

### 6.1 交互/行为（如适用）

- 用户流程：
  - 用户点击左侧“搜索”进入 SearchPage。
  - 在搜索框中输入关键词，按 Enter 提交搜索。
  - 页面显示按更新时间排序的匹配会话结果列表。
  - 用户点击任一结果，页面切回 chat 并激活对应会话。
- 边界条件：
  - 空查询不展示伪结果，只展示提示态。
  - 无命中时展示明确空态。
  - 会话标题为空时使用既有标题字段值，不额外伪造新规则。

### 6.2 接口与契约

- API/函数签名：
  - SearchPage 内部搜索：query -> SessionSummary[]。
  - 会话打开动作：sessionId -> 导航到 chat 并激活会话。
- 错误码/异常：
  - 首版无新接口，不新增错误码。
  - 若 core.boot 尚未完成，则页面展示当前会话数量和空状态，不伪造失败结果。

### 6.3 数据与存储（如适用）

- Schema/字段：
  - 搜索仅依赖 SessionSummary.title、SessionSummary.updatedAt、SessionSummary.id。
  - 页面本地状态包括 inputQuery、submittedQuery。
- 迁移策略：
  - 不涉及持久化和数据迁移。

### 6.4 兼容性与发布

- 兼容策略：不做旧左侧搜索输入兼容，统一以独立搜索页为准。
- 发布步骤：前端页面改造 -> 单测更新/补充 -> eslint 与 ts 检查通过。

### 6.5 可观测性

- 日志：首版不新增日志。
- 指标：首版不新增 metrics。
- 告警：首版不新增告警。
