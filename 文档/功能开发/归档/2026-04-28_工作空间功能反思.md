# 工作空间功能反思 - 2026-04-28

> 当前完成到 M7(数据库 Bases)。M8 Canvas 未做。
> 以下逐模块检视已实现功能的完成度和缺陷。

---

## 一、M1 骨架 + M2 编辑器 - ✅ 基本完整

### 已做到

- 路由/布局/文件树/标签页/文件预览全链路跑通
- .md 编辑 + 2s 防抖自动保存 + 关闭时 flush
- TabBar 保存状态指示点
- 新建下拉菜单(日记/笔记/闪念/文件夹/数据库)

### 遗留问题

1. **文件树右键菜单缺失** - M2 计划的 rename/delete 右键操作始终没做,目前文件树只有 click 打开、无任何 CRUD 操作入口。这是核心交互缺失。
2. **搜索栏纯静态 UI** - 左侧搜索框只是装饰,无实际搜索逻辑,也不过滤文件树。
3. **新建笔记不走子目录** - `handleCreateNote(_subdir)` 接收了 subdir 参数但完全忽略,新建笔记总是落在 workspace 根目录。应该根据参数创建到 `笔记/` 或 `收件箱/` 子目录。

---

## 二、M3 仪表盘 - ⚠️ 半成品

### 已做到

- 今日日记摘要展示(读取日记内容取前 3 行)
- 最近编辑列表(调后端 recent-files API)
- 快捷操作按钮

### 遗留问题

1. **今日待办卡片是死占位** - 文案写的"待办视图将在后续实现",但 M4 待办已经做了。`DashboardView` 应该接入 `useWorkspaceTasks.todayTasks` 展示真实数据。
2. **日记摘要取前 3 行太粗糙** - 如果日记有 frontmatter(---),前 3 行全是 YAML 元数据,用户看到的是废信息。需要跳过 frontmatter 取正文。
3. **"开始写"按钮只触发创建,不打开已有日记** - `DashboardView` 的 `create-journal` 事件名字有误导。`useDashboard` 的 `todayJournalPath` 已经知道了,"打开"按钮应 emit `open-file` 事件而非 `create-journal`。

---

## 三、M4 待办系统 - ⚠️ 半成品

### 已做到

- 后端 workspace_tasks 表 + CRUD API
- 后端 checkbox 扫描 API(递归扫 .md 文件中的 `- [x]` 语法)
- 前端聚合 composable(独立任务 + checkbox 任务合并)
- 清单/看板/今日三种视图模式

### 遗留问题

1. **checkbox 状态切换无法回写** - 点击 checkbox 行的切换按钮只调用了 `updateWorkspaceTask`(更新独立任务 DB),对 `source.type === "checkbox"` 的行无效。checkbox 来源的行,切换完成状态必须回写 .md 文件对应行(`- [ ]` ↔ `- [x]`),目前后端没有这个 API。
2. **看板视图不可拖拽** - 看板只做了静态分组展示,没有拖拽排序或跨列拖拽。计划文档标注"M9 迭代",但这意味着当前看板是假看板,只是换了排列方式的列表。
3. **清单视图删除按钮缺失** - 看板视图有删除按钮,但清单视图只有切换状态按钮、没有删除入口。独立任务应可删除。
4. **checkbox 扫描全量递归无缓存** - `GET /api/workspace/tasks/checkboxes` 每次请求都递归扫描整个 workspace 下所有 .md 文件,文件多了会非常慢。应有增量或缓存机制。

## 四、M5 日历+日记 - ⚠️ 有功能缺陷

### 已做到

- 月历网格正确渲染
- 有日记的日期标圆点
- 点击日期打开/创建日记
- 底部显示当月日记列表
- 月份前后导航 + "今天"按钮

### 遗留问题

1. **创建日记使用错误 API** - `handleCreateJournal` 调用 `createNote("")` 但这个 API 创建的是 workspace 根目录的空白笔记,然后在 `preview.openFile(journalPath)` 打开一个路径。实际文件并没有创建到 `日记/YYYY/MM/` 目录下,`createNote` 不知道要创建到子目录。这会导致"创建日记"操作实际上在根目录生成了一个空 .md 文件,而日记路径的文件并不存在。
2. **open-file 路径不一致** - `CalendarView` emit 的 `open-file` 路径是相对路径(`日记/2026/04/2026-04-28.md`),但 `handleSelectFile` 期望的是绝对路径(`/full/workspace/path/...`)。这会导致点击日历中的日记条目时路径不匹配。
3. **同上,最近编辑列表的 `open-file` 路径也有问题** - `recent-files` API 返回的 `path` 是绝对路径,`DashboardView` 直接 emit 这个绝对路径,但 `handleSelectFile` 期望的 `entry.path` 也是绝对路径。需要确认路径一致性。

## 五、M6 收件箱 - ⚠️ 功能缺陷

### 已做到

- 闪念捕捉文本区域
- Ctrl+Enter 提交
- 保存到 `收件箱/闪念_时间戳.md`

### 遗留问题

1. **createNote API 不支持指定路径** - `handleCapture` 调用 `createNote(text, filePath)`,但 `createNote` 的签名是 `createNote(name: string)`,只接收文件名。第二个参数被忽略。这意味着闪念笔记不会创建到 `收件箱/` 子目录,而是创建到 workspace 根目录。这是一个根本性的功能缺陷。
2. **收件箱没有历史列表** - 只有输入框,没有展示已有的收件箱笔记列表。用户写完闪念就看不到了。
3. **说明文案提到"拖拽归档"但拖拽功能不存在** - 文案说"拖拽到其他文件夹即可归档",但文件树没有任何拖拽功能。

## 六、M7 数据库 Bases - ⚠️ 骨架级,多处不足

### 已做到

- 后端 .base 文件 CRUD API
- 后端读取时自动解析文件引用行的 frontmatter
- 前端表格视图(双击编辑、select 下拉、checkbox 切换)
- 前端看板/画廊视图(静态展示)
- 视图切换 + 新建视图

### 遗留问题

1. **BaseView 日历视图是空壳** - 只有一句"需要日期列才能显示",没有任何日历渲染逻辑。
2. **看板视图无拖拽** - 和 M4 同样的问题,只做分组展示,卡片不能跨列拖拽。
3. **画廊视图卡片不可交互** - 不能点击展开详情、不能编辑。
4. **列头不可重命名** - 新建列用默认名字,没有重命名入口。
5. **视图不可重命名/删除** - 视图标签只有切换,没有管理功能。
6. **排序/过滤未实现** - `BaseView` 定义了 sort/filters 数据结构,但前端完全没有排序和过滤的 UI 和逻辑。
7. **表格视图列头删除按钮始终不可见** - `opacity-0 hover:opacity-60` 但 hover 目标是 button 本身而非 th,需要改成 th hover 时显示。
8. **新增行没有输入首列内容的入口** - 新增行 `cells: {}` 是空的,用户必须在表格中双击每个单元格逐一填写,体验很差。
9. **.base 文件点击后 previewKind 才为 "base"** - 但后端 resolvePreviewKind 返回 "base" 时,`getFilePreview` 对 base 类型的处理是当作 text/code 返回文件内容。这意味着 tab.content 是 JSON 字符串,而不是 BaseData 对象。BaseView 的 load 是独立调 getBaseData,这没问题,但 getFilePreview 的无用请求浪费了资源。

---

## 七、跨模块通用问题

1. **路径混乱** - 有些地方用绝对路径,有些用相对路径(相对于 workspaceDir)。没有统一的路径规范。这是当前最大的隐患:
   - `preview.openFile()` 接收绝对路径
   - `createNote` API 接收相对路径
   - `CalendarView` emit 相对路径
   - `recentFiles` 返回绝对路径
   - `handleSelectFile` 期望 entry.path 是绝对路径

2. **createNote API 能力不足** - 当前 `createNote(name)` 只能创建到 workspace 根目录,无法指定子目录。M6 闪念笔记需要创建到 `收件箱/`,M5 日记需要创建到 `日记/YYYY/MM/`。后端需要增强。

3. **前端搜索栏完全是空壳** - M6 计划中的全文搜索未做,左侧搜索框没有任何作用。

4. **视图标签页的活跃管理有 bug** - 点击固定视图入口时 `handleSelectView` 会向 viewTabs 追加,但关掉视图标签后重新点击同一视图,标题可能不一致(用了 fixedViews 的 label vs 之前手动写的 title)。

## 结论:哪些该补,哪些该暂停

1. ✅ `createNote` API 增强支持指定子目录路径
2. ✅ 修复 `handleCreateJournal` 用正确的 API 创建日记文件
3. ✅ 修复 `handleCreateNote` 使用子目录
4. ✅ 修复 `InboxView.handleCapture` 使用正确的 API 创建到收件箱
5. ✅ 统一路径规范（全部用绝对路径，只在 API 调用时转为相对路径）
6. ✅ 仪表盘今日待办卡片接入真实数据
7. ✅ 日记摘要跳过 frontmatter
8. ✅ 仪表盘“打开”按钮直接打开日记而非触发 create-journal
9. ✅ checkbox 任务状态切换需回写 .md 文件（后端新增 API）
10. ✅ 清单视图添加删除按钮

### 应该补（体验缺失但不致命）— 已修复 ✅

11. ✅ 收件箱展示历史闪念列表
12. ✅ Base 表格列头 hover 显示删除按钮
13. ✅ Base 新增行时首列可直接输入
14. ✅ Base 列头双击可重命名
15. ✅ Base 视图可删除
16. ✅ Base 日历视图按日期列渲染
17. ✅ Base 列头点击可排序

- Base 过滤 UI
- 全文搜索
- 双向链接/标签(M9)
