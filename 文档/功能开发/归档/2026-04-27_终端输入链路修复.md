# 终端输入链路修复

- 状态：已完成
- 日期：2026-04-27
- Owner：ridge 前端

## 背景

用户反馈 `/terminal` 页面中终端无法输入字符。截图中终端区域能渲染光标，但没有 shell prompt，也没有输入回显。

## 初始假设

- 终端前端渲染层 `@wterm/dom` 已初始化，否则不会出现光标。
- 服务端 `node-pty` 终端池已经存在，否则 `/api/terminals` 不会返回终端记录。
- 断点更可能在 WebSocket 附着链路：前端未连上 `/api/terminals/:id/stream`，导致输入不能写入 PTY，PTY 输出也不能回到页面。

## 探索结论

- `TerminalViewport.vue` 通过 `getTerminalStreamUrl()` 连接当前页面 host 下的 `/api/terminals/:id/stream`。
- 开发环境页面运行在 Vite 端口，REST `/api` 依赖 Vite proxy 转发到 Node 服务端。
- `packages/web/vite.config.ts` 当前使用 `"/api": "http://127.0.0.1:3000"` 字符串代理，没有显式开启 WebSocket 代理。
- 浏览器复现时终端页面显示光标，但 `/api/terminals` 返回 `status: "disconnected"`，说明前端没有附着到服务端 PTY。

## 方案

将 Vite `/api` 代理改为对象配置，显式设置：

- `target: "http://127.0.0.1:3000"`
- `ws: true`

这是终端输入链路的根因修复：保留前端使用同源 `/api` 的接口边界，让 WebSocket 流和 REST 一样走开发代理进入 Node 服务层。

## 验收标准

- [x] `/terminal` 页面打开后终端状态能从 `disconnected` 进入连接态。
- [x] 点击终端后输入 `echo RIDGE_INPUT_TEST` 并回车，终端显示命令和输出。
- [x] `pnpm lint` 通过。
- [x] `pnpm typecheck` 通过。

## 执行结果

- 修改 `packages/web/vite.config.ts`，把 `/api` 代理改为对象形式并设置 `ws: true`。
- 直连验证 `ws://localhost:5174/api/terminals/:id/stream` 可打开，输入 `echo RIDGE_WS_TEST` 能收到 PTY 输出。
- 浏览器刷新 `/terminal` 后输入 `echo RIDGE_BROWSER_OK`，页面显示命令与输出。
