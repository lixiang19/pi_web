# Pi ask 阻塞式交互落地

## 背景
- 现有项目只有 Pi runtime、消息流、资源目录、权限 gate。
- 仓库内虽然有 `question/questionnaire` 示例与研究文档，但 Web 运行态没有真正 ask/question 闭环。
- 用户需求：统一做成 `ask`，不保留 `question`，并且必须是真正阻塞式 tool，而不是伪装消息。

## 本次决策
1. 工具名统一为 `ask`
2. 首期直接支持结构化 `questions[]`
3. 支持单选 / 多选 / 自定义输入
4. ask 语义是“挂起中的 tool”
5. 前端卡片放中间消息流底部，不做弹窗
6. 用户回答后恢复原 tool 执行，最后产出正常 `toolResult`

## 服务端实现
### 1. ask 扩展
新增：`packages/server/src/ask-extension.ts`
- 向 Pi runtime 注册 `ask` tool
- tool execute 时创建 pending ask
- 返回 Promise，等待前端回答后再 resolve
- 支持 abort 时中止 pending ask

### 2. 运行时状态
修改：`packages/server/src/types/index.ts`
- 增加 `AskInteractiveRequest`
- 增加 `AskQuestion/AskQuestionAnswer`
- 增加 `PendingAskRecord`
- `SessionRecord` 增加 `pendingAskRecords`
- `SessionSnapshot` 增加 `interactiveRequests`
- `AgentPermission.question` 改为 `AgentPermission.ask`

### 3. 会话桥接
修改：`packages/server/src/index.ts`
- 新增 ask 请求/回答 schema
- 新增 pending ask 序列化到 snapshot
- 新增 `POST /api/sessions/:sessionId/asks/:askId/respond`
- `updateStatus()` 在 ask pending 时不允许回退到 idle
- session destroy 时会清理 pending ask
- `createSessionResourceLoader()` 注入 ask 扩展

### 4. 权限层
修改：`packages/server/src/agent-permissions.ts`
- 权限工具名从 `question` 收敛到 `ask`

## 前端实现
### 1. 类型与 API
修改：
- `packages/web/src/lib/types.ts`
- `packages/web/src/lib/api.ts`

新增：
- `AskInteractiveRequest`
- `AskQuestionAnswer`
- `respondToAsk()` API

### 2. 会话状态编排
修改：`packages/web/src/composables/usePiChat.ts`
- 暴露 `interactiveRequests`
- snapshot 中同步 pending ask
- 新增 `respondToPendingAsk()`
- 新增 `dismissPendingAsk()`
- status=streaming 时同步更新 `isSending/composer.canAbort`

### 3. ask 卡片
新增：`packages/web/src/components/chat/AskCard.vue`
- 使用 shadcn-vue 组件实现
- 多题 tabs
- 单选 / 多选 / 自定义输入
- 汇总页
- 提交 / 取消

### 4. 消息流渲染
修改：
- `packages/web/src/components/workbench/chat/WorkbenchMessageStream.vue`
- `packages/web/src/components/workbench/chat/WorkbenchChatPanel.vue`
- `packages/web/src/pages/WorkbenchPage.vue`
- `packages/web/src/pages/SessionDetailPage.vue`

行为：
- ask 卡片显示在消息流底部
- 回答后立即移除 pending ask 卡片
- session 继续保持 streaming，等待 tool 恢复完成

### 5. 历史回放修正
后续继续收敛前端行为：
- pending ask 继续保留底部独立交互卡片
- 历史 ask 必须回到普通工具展示，不再做独立历史卡片
- ask `toolCall` 标题显示问题摘要，展开显示完整问题列表
- ask `toolResult` 标题显示答案摘要，展开显示完整问答对

为此补了消息桥接元数据：
- server 序列化消息时保留 `toolCallId/toolName/details/isError`
- web `ChatMessage` / `usePiChat` 同步保留这些字段
- `ChatProcessGroup.vue` 基于真实工具元数据做 ask 专属展示

## 验收点
- agent 能调用 `ask`
- Web 能看到 ask 卡片
- 用户能提交或取消
- 服务端能恢复原 tool 执行
- ask 完成后消息流继续推进
- `question` 权限键已收敛为 `ask`

## 验证
- `npm run lint`
- `npm run check --workspace @pi/web`
- `npm run build --workspace @pi/server`

## 后续建议
1. 后续可补 ask 结果专用 tool renderer，提升回放可读性
2. 后续可给 pending ask 加“回答中”状态，避免重复点击
3. 后续可把 ask 契约抽成通用 interactive request 层，但当前阶段先只支持 ask，避免过度泛化
