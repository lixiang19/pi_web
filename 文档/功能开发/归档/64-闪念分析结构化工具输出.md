# 64 闪念分析结构化工具输出

## 目标

把闪念分析从“提示词要求 JSON + 正则解析 assistant 文本”改为 Pi custom tool 结构化输出，避免模型自然语言或 Markdown 包裹 JSON 导致的不稳定。

## 实现

- `fleeting-analysis.ts` 注册唯一可用工具 schema `fleeting_analysis_result`。
- 分析调用从 `createAgentSession()` 下沉到 `@mariozechner/pi-ai complete()`，不再创建 `SessionManager/ResourceLoader/AgentSession`。
- 模型和服务商仍使用 Pi 配置：读取 `settings.json` 的 `defaultProvider/defaultModel/defaultThinkingLevel`，通过 `ModelRegistry` 复用 `models.json/auth.json`。
- `context.systemPrompt` 是闪念分析专用提示词，`context.tools` 只包含 `fleeting_analysis_result`。
- 前端文件、PDF、Word、Markdown、音频等附件默认走 multipart 上传到 `.ridge/fleeting-attachments/<noteId>/`；创建闪念时可先延迟分析，附件上传完成后再显式触发分析。
- worker 分析前读取附件正文：文本/Markdown/JSON 等只读取前缀；PDF、Word、音频、图片等二进制附件调用 Python Converter 转 Markdown/转写/OCR。
- prompt 只放附件列表和按字符预算截断后的附件正文，不把原始文件、base64 或完整大文档塞给模型。
- prompt 明确要求调用工具，不再要求输出 JSON 文本。
- 服务端只从 assistant `toolCall.arguments` 读取并校验 `recommendationType/recommendationText/draft/requiresInput`。
- 未调用工具、模型错误或 arguments 结构非法都会走后台 job 失败/重试路径。

## 验收

- `pnpm --filter @pi/server test -- fleeting-analysis` 通过。
- 新增测试覆盖：使用 settings 默认模型调用 `pi-ai complete()`，assistant 文本 JSON 不再被接受，必须调用 `fleeting_analysis_result`。
- 新增测试覆盖：二进制附件会先通过 Converter 取 Markdown 正文，并把截断后的正文放入模型上下文。
