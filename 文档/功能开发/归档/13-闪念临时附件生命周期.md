# 13 闪念临时附件生命周期

## 目标

实现闪念临时附件目录和完整生命周期管理。

## 范围

- `.ridge/fleeting-attachments` 目录规划 ✅
- 闪念附件上传 API ✅
- 未处理闪念引用临时附件 ✅
- 删除未处理闪念时清理未引用临时附件 ✅
- 处理成功后迁移到正式附件目录 ✅

## 验收

- 临时附件属于未处理闪念生命周期 ✅
- 处理失败不删除临时附件 ✅
- 只有目标系统确认创建成功后才迁移附件 ✅
- 正式附件保留原文件名，冲突时追加短 ID ✅

## 实现记录

### 数据库

- 新增 `fleeting_attachments` 表（migration v8）
- 字段：attachment_id, note_id, original_name, stored_name, stored_path, mime_type, size, sha256, created_at
- repair columns 同步更新

### 后端

- **新增 `packages/server/src/fleeting-attachments.ts`**：附件核心逻辑
  - `storeFleetingAttachment`：保存到 `.ridge/fleeting-attachments/{noteId}/`
  - `getFleetingAttachments`：按 note_id 查询
  - `deleteFleetingAttachment`：单条删除（DB + 文件）
  - `deleteFleetingAttachmentsForNote`：批量清理
  - `migrateFleetingAttachments`：迁移到 `附件/` 正式目录，冲突时自动重命名

- **更新 `packages/server/src/routes/fleeting.ts`**：
  - `POST /:noteId/attachments`：上传附件（multer, 最多 10 个, 50MB）
  - `GET /:noteId/attachments`：列出附件
  - `DELETE /:noteId`：先清理附件，再删除闪念
  - `POST /:noteId/process/journal`：先迁移附件，再写入日记
  - `POST /:noteId/process/clip`：先迁移附件，再保存剪藏
  - `POST /:noteId/process/task`：处理未成功，不迁移附件（保留临时文件）

### 前端

- **更新 `packages/web/src/lib/api.ts`**：新增 `FleetingAttachment` 类型、`uploadFleetingAttachments`、`getFleetingAttachments` API

- **更新 `packages/web/src/composables/useInbox.ts`**：
  - 加载闪念时自动加载附件列表
  - `uploadAttachments` 方法
  - `getNoteAttachments` 计算属性
  - 处理成功后清理附件 map

- **更新 `packages/web/src/components/workspace/InboxView.vue`**：
  - 捕捉区支持文件选择（点击+拖拽）
  - 选中文件预览列表（可移除）
  - 保存时自动上传附件
  - 闪念卡片展示附件列表（图标+文件名+大小）
  - 拖拽区域高亮反馈

### 测试

- **后端 `packages/server/src/__tests__/fleeting-attachments.test.ts`**（8 项测试）：
  - 上传附件到临时目录并记录 DB
  - 列出闪念附件
  - 拒绝不存在闪念的附件上传
  - 删除闪念时清理临时附件
  - 日记处理时迁移附件到正式目录
  - 剪藏处理时迁移附件到正式目录
  - 文件名冲突时自动追加后缀
  - 任务处理失败时保留临时附件

- **更新现有测试**：
  - `fleeting-api.test.ts`：添加 fleeting_attachments 表结构
  - `useInbox.test.ts`：适配新 API 和附件加载逻辑
  - `InboxView.test.ts`：适配新 API，使用 flushPromises
