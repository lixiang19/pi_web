# 终端页面功能 方案

- 状态：已确认
- 日期：2026-04-19
- Owner：ridge
- 目标版本/里程碑：Terminal Page Phase 1

## 1. 背景与问题

- 背景：工作台已经把终端提升为一级主路由，但当前仍停留在占位页，既没有真实终端运行时，也没有终端级状态、传输协议和页面交互。
- 现象/痛点：
  - `/terminal` 现在只渲染说明卡片，没有任何真实执行能力。
  - 终端运行时不能放进现有 `/chat` 的会话 LRU，否则会把聊天会话与终端生命周期混在一起。
  - 服务端目前只有 HTTP API，没有 PTY 和 WebSocket 终端桥接能力，页面刷新后也无法保持真实终端继续运行。
  - 终端页面缺少全局终端池、分屏、信息侧栏和项目上下文切换，无法承担一级功能页角色。
- 成功定义：`/terminal` 变成真实可用的终端工作区，基于 wterm 渲染终端，Node 服务层通过 PTY + WebSocket 提供真实 shell；支持全局终端标签池、最多双窗格分屏、信息侧栏、显式目录切换；页面刷新后可重新连接运行中的终端，主题与项目上下文保持正确。

## 2. 目标与非目标

- 目标：
  - 用 `@wterm/dom` 替换当前占位页，落地真实终端页面而不是浏览器假 shell。
  - 在 Node 服务层新增终端运行时管理器，负责 PTY 生命周期、终端元数据和 WebSocket 附着。
  - 终端实例采用全局终端池，不从属于聊天会话，不复用 `useSessionLruPool.ts`。
  - 终端页面支持新建/关闭标签、激活切换、最多双窗格分屏、信息侧栏。
  - 目录切换采用“显式以新 cwd 重启当前终端”模型，不向 shell 注入伪命令改 cwd。
  - 页面刷新后通过服务端终端池重新连接既有终端，不重复创建 PTY。
  - 终端主题使用 ridge 自定义 token 映射到 wterm CSS 变量，明暗模式同步。
- 非目标：
  - 不做浏览器内 `just-bash` 假终端。
  - 不让终端成为聊天会话的子面板，也不塞回 `SessionTabContent.vue`。
  - 不做应用重启后的终端进程恢复；应用重启后允许终端全部终止。
  - 不做 SSH、容器终端、远程多主机连接等二期能力。
  - 不做无限制目录浏览和任意路径裸写；cwd 必须经过服务端上下文校验。

## 4. 资料与依据（Librarian）

- 关键结论：
  - `wterm` 的消费者包分为 `@wterm/core`、`@wterm/dom`、`@wterm/react`、`@wterm/just-bash`、`@wterm/markdown`；Vue 项目最适合接 `@wterm/dom`，不需要 `@wterm/react`。
  - `WTerm` 原生支持 `cols`、`rows`、`autoResize`、`cursorBlink`、`onData`、`onTitle`、`onResize`，并提供 `init/write/resize/focus/destroy` 方法；这足够支撑 Vue 组件封装。
  - `wterm` 自带 `WebSocketTransport`，支持自动重连、发送缓冲和二进制消息；但 resize 仍需业务层通过 `onResize` 主动把尺寸信息发给服务端。
  - `wterm` 官方本地 shell 示例用 `node-pty + ws + WebSocket` 连接真实 PTY，说明“真实终端”必须有 PTY 后端，不是 `just-bash`。
  - `wterm` 默认把约 12 KB 的 WASM 内嵌在包里，消费者不必额外构建 Zig；如需分离静态资源，可再补 `wasmUrl`。
  - `wterm` 主题建立在 CSS 自定义变量上，ridge 可以直接桥接项目主题 token 到 `--term-*` 变量，而不是套用官方内置主题名。
  - 参考仓库当前最新版本为 `0.1.8`，已包含 API Reference、Vite 示例和 WebSocketTransport 文档，适合直接作为一期集成基线。

## 5. 方案概览（主方案）

### 5.1 方案摘要

- 一句话：把 `/terminal` 重构为基于 `@wterm/dom` 的真实终端工作区，并在 Node 服务层新增独立终端管理器，用 PTY + WebSocket 驱动全局终端池、双窗格分屏和刷新重连。
- 取舍：
  - 选择 `@wterm/dom` 而不是 `@wterm/react`，因为当前前端是 Vue，直接接 DOM API 更干净。
  - 选择“终端池独立于聊天会话”，而不是把终端做进 `/chat` 的操作区，避免生命周期污染。
  - 目录切换选择“重启到新 cwd”，而不是向 shell 注入 `cd`，确保 cwd 真正由 PTY 进程管理。
  - 选择服务端常驻 PTY + 可重附着 WebSocket，而不是“连接断开即 kill”，从根本上满足刷新无损。
- 影响面：
  - Web：终端页面、终端池 composable、wterm 视图组件、信息侧栏、分屏布局、API 封装。
  - Server：HTTP 启动方式、终端运行时管理器、PTY 依赖、REST + WebSocket 终端接口。
  - Protocol：终端快照与创建/重启/更新契约。

### 5.2 风险与回滚

- 风险：
  - `node-pty` 是原生依赖：需要补 `pnpm.onlyBuiltDependencies`，并验证本地构建链路；缓解措施是把依赖安装契约写进根包并做完整 lint/typecheck。
  - WebSocket 重连可能重复附着：缓解措施是服务端终端管理器把“终端进程”和“连接”解耦，同一终端多次 attach 不重复 spawn PTY。
  - 分屏与多标签一起落地时，布局状态可能复杂：缓解措施是一期只支持最多双窗格，不做任意网格布局。
  - `wterm` resize 不会自动传给后端：缓解措施是统一用 `onResize` 发送官方示例同构的控制消息，由服务端拦截后调用 `pty.resize()`。
- 回滚：
  - 若 PTY 接入无法稳定落地，可整体回退本次新增终端 API、终端页面和依赖，但不保留旧占位页与新页面长期共存分支。

## 6. 详细设计（Spec）

### 6.1 交互/行为（如适用）

- 用户流程：
  - 用户进入 `/terminal`，页面先请求终端列表；若为空则自动创建一个默认终端。
  - 终端页面顶部显示全局标签条，支持新建、关闭、切换；主区域支持单窗格和双窗格两种视图。
  - 每个窗格内挂一个 `wterm` 实例，`onData` 把输入发到对应终端的 WebSocket，服务端 PTY 输出再写回终端。
  - 用户在信息侧栏查看当前终端标题、cwd、连接状态、尺寸、最近退出码，并可显式选择新 cwd 后重启该终端。
  - 页面刷新时，前端重新拉取终端池快照，并按终端 id 重新连接 WebSocket；已有 PTY 继续运行。
  - 用户关闭标签时，对应终端被显式销毁；关闭某个窗格只改变布局，不自动结束另一个终端。
- 边界条件：
  - WebSocket 断开后页面显示断开状态，并依赖 `WebSocketTransport` 自动重连；若服务端已删除该终端，则回收对应标签。
  - 应用重启后终端池为空，页面重新进入默认新建流程，不承诺跨重启恢复。
  - cwd 切换不是给运行中 shell 注入命令，而是显式重启当前终端进程到新目录。
  - 分屏一期最多 2 个窗格，避免任意多窗格引入额外状态复杂度。
  - 移动端不做复杂分屏；小屏下退化为单窗格 + 侧栏 Sheet。

### 6.2 接口与契约

- API/函数签名：
  - `GET /api/terminals`：`void -> TerminalListResponse`
  - `POST /api/terminals`：`TerminalCreateRequest -> TerminalSnapshot`
  - `PATCH /api/terminals/:id`：`TerminalUpdateRequest -> TerminalSnapshot`
  - `POST /api/terminals/:id/restart`：`TerminalRestartRequest -> TerminalSnapshot`
  - `DELETE /api/terminals/:id`：`void -> { ok: true }`
  - `WS /api/terminals/:id/stream`：客户端发送终端输入字节流；当发送 `\x1b[RESIZE:<cols>;<rows>]` 时服务端拦截为 `pty.resize()`；服务端原样回推 PTY 输出。
  - `useTerminalPool()`：`createTerminal / removeTerminal / restartTerminal / setActiveTerminal / splitTerminal / closePane / terminals / panes / activeTerminalId`
  - `useWTermSession(terminalId)`：`mount / unmount / focus / connectionState / title / lastResize`
- 错误码/异常：
  - 创建或重启终端时 cwd 非法、越界或不存在，返回 400。
  - 请求不存在的终端 id，REST 返回 404，WebSocket 在建立后立即关闭并返回错误消息。
  - PTY spawn 失败返回 500，并在页面卡片中展示明确错误。

### 6.3 数据与存储（如适用）

- Schema/字段：
  - `TerminalSnapshot`
    - `id: string`
    - `title: string`
    - `cwd: string`
    - `shell: string`
    - `status: "starting" | "running" | "disconnected" | "exited" | "error"`
    - `cols: number`
    - `rows: number`
    - `order: number`
    - `pane: "primary" | "secondary" | null`
    - `createdAt: number`
    - `updatedAt: number`
    - `exitCode?: number | null`
    - `errorMessage?: string | null`
  - `TerminalRuntimeRecord`
    - `snapshot: TerminalSnapshot`
    - `ptyProcess: IPty`
    - `connections: Set<WebSocket>`
    - `pendingOutput: Uint8Array[]`
  - `TerminalCreateRequest`
    - `cwd?: string`
    - `title?: string`
    - `cols?: number`
    - `rows?: number`
  - `TerminalRestartRequest`
    - `cwd: string`
    - `cols?: number`
    - `rows?: number`
- 迁移策略：
  - 一期不做磁盘持久化，终端池只驻留在 Node 进程内存中；页面刷新通过服务端内存快照恢复，应用重启即清空。

### 6.4 兼容性与发布

- 兼容策略：直接替换现有终端占位页，不保留旧页面或旧 API 分支。
- 发布步骤：
  - 先补 protocol 类型、服务端终端管理器与 PTY/WebSocket 依赖。
  - 再落地前端终端池、wterm 组件、标签/分屏/侧栏布局。
  - 最后补页面验收、模块梳理和记忆文档更新。

### 6.5 可观测性

- 日志：
  - 终端创建、附着、断开、重启、销毁、PTY 退出日志。
  - cwd 校验失败和 PTY spawn 失败日志。
- 指标：
  - 当前运行中的终端数。
  - WebSocket 重连次数。
  - 终端创建失败次数。
- 告警：
  - 终端进入 `error` 或 `exited` 状态时，页面显式展示错误/退出信息，不静默吞掉。
