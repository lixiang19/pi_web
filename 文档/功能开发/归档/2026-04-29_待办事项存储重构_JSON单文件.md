# 待办事项存储重构：JSON 单文件方案

**日期**：2026-04-29  
**状态**：设计中

---

## 1. 背景与问题

### 当前架构（双源）

```
TaskView UI ⟶ useWorkspaceTasks 聚合
            ├── SQLite workspace_tasks 表 (~/.pi/ridge.db)  ← 独立任务（Agent 无法自然操作）
            └── .md 文件 checkbox 扫描                       ← 上下文任务（散布各处）
```

**核心矛盾**：工作空间是 Agent 和 Human 的共同工作区。Agent 的工具集围绕**文件操作**（read_file / write / edit），SQLite 对它不透明。同时双源聚合引入了无谓的复杂性。

### 用户决策

- **放弃散落**：不再扫描 .md 文件中的 checkbox
- **统一集中**：所有任务用一个 JSON 文件管理
- **Agent 可见**：Agent 可通过文件工具直接操作

---

## 2. 设计目标

1. **一源一格式**：`tasks.json` 是唯一数据源，数据结构即存储格式，无转换层
2. **Agent & Human 共享**：Agent 用文件工具操作，Human 用 TaskView UI 操作，同一份数据
3. **并发安全**：mtime 乐观锁，防止 Agent 和 Human 同时写入冲突
4. **保持 CRUD**：粒度细，前端可乐观更新单条，不需要全量重写

---

## 3. 数据模型

### 3.1 存储位置

```
{workspaceDir}/.ridge/tasks.json
```

放在隐藏目录 `.ridge/`，与 ridge 数据放一起，不污染工作空间根目录。

### 3.2 JSON Schema

```json
{
  "tasks": [
    {
      "id": "task-1714060800000-a1b2c3",
      "title": "设计 API 接口",
      "status": "pending",
      "priority": "high",
      "dueDate": 1746144000000,
      "tags": ["api", "backend"],
      "createdAt": 1714060800000,
      "updatedAt": 1714060800000
    }
  ],
  "updatedAt": 1714060800000
}
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识，格式 `task-{timestamp}-{random6}` |
| title | string | 任务标题 |
| status | `"pending"` \| `"in_progress"` \| `"done"` | 任务状态 |
| priority | `"low"` \| `"medium"` \| `"high"` | 优先级 |
| dueDate | number \| null | 截止日期 unix 时间戳 |
| tags | string[] | 标签数组 |
| createdAt | number | 创建时间 unix 时间戳 |
| updatedAt | number | 更新时间 unix 时间戳 |

顶层 `updatedAt` 用于 mtime 乐观锁校验。

### 3.3 与旧模型对比

| 旧字段 | 新字段 | 变化 |
|--------|--------|------|
| task_id | id | 简化命名 |
| status | status | 不变 |
| priority | priority | 不变 |
| due_date | dueDate | camelCase |
| tags (逗号分隔字符串) | tags (数组) | 结构化 |
| note_path | ❌ 删除 | 不再关联笔记 |
| created_at | createdAt | camelCase |
| updated_at | updatedAt | camelCase |

## 4. API 契约

### 4.1 获取任务列表

```
GET /api/workspace/tasks
```

**响应**：
```json
{
  "tasks": [...],
  "updatedAt": 1714060800000
}
```

后端读取 `.ridge/tasks.json` 并返回。文件不存在时返回空数组。

### 4.2 创建任务

```
POST /api/workspace/tasks
```

**请求体**：
```json
{
  "title": "设计 API 接口",
  "priority": "high",
  "dueDate": 1746144000000,
  "tags": ["api", "backend"],
  "_expectedUpdatedAt": 1714060800000
}
```

**响应** (201)：
```json
{
  "task": {
    "id": "task-1714060800000-a1b2c3",
    "title": "设计 API 接口",
    "status": "pending",
    "priority": "high",
    "dueDate": 1746144000000,
    "tags": ["api", "backend"],
    "createdAt": 1714060800000,
    "updatedAt": 1714060800000
  },
  "updatedAt": 1714060800000
}
```

**错误** (409)：
```json
{
  "error": "TASK_CONFLICT",
  "message": "tasks.json 已被修改，请刷新后重试",
  "currentUpdatedAt": 1714060900000
}
```

### 4.3 更新任务

```
PATCH /api/workspace/tasks/:taskId
```

**请求体**（可选字段）：
```json
{
  "status": "done",
  "title": "新标题",
  "priority": "low",
  "dueDate": null,
  "tags": ["done"],
  "_expectedUpdatedAt": 1714060800000
}
```

**响应** (200)：
```json
{
  "ok": true,
  "updatedAt": 1714060800001
}
```

### 4.4 删除任务

```
DELETE /api/workspace/tasks/:taskId
```

**请求体**：
```json
{
  "_expectedUpdatedAt": 1714060800000
}
```

**响应** (200)：
```json
{
  "ok": true,
  "updatedAt": 1714060800001
}
```

## 5. 并发策略（mtime 乐观锁）

### 流程

```
前端读取 tasks.json（记录 updatedAt）
    │
    ├── Human 通过 UI 修改
    │       └── PATCH 请求带 _expectedUpdatedAt
    │              ├── 匹配 → 写入 + 返回新 updatedAt
    │              └── 不匹配 → 409 → 前端 toast + 重新加载
    │
    └── Agent 通过 write 工具修改 tasks.json
            └── 文件 mtime 变化
                    └── 前端下次请求时 updatedAt 已变 → 正常刷新
```

### 关键点

- 所有写操作（POST/PATCH/DELETE）都携带 `_expectedUpdatedAt`
- 后端**先读取文件**检查 `updatedAt` 是否匹配，不匹配直接 409
- 前端收到 409 时 toast 提示 + 重新 GET 列表
- `_expectedUpdatedAt` 是内部字段，不在 JSON 文件中，仅传输用

## 7. 数据迁移

### 策略

**不迁移**。独立任务数据量小，SQLite 中的旧任务不做自动迁移。用户手动重建（影响极小）。

### 理由

- `workspace_tasks` 表是短期数据，不是持久资产
- 引入迁移代码增加复杂度，且一次性
- 删除旧表 + 用户手动创建 = 最简
