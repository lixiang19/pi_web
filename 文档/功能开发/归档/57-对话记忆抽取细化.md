# 57 对话记忆抽取细化

## 背景

当前 ridge 已有 `summary.daily` 和 `memory.maintain`：

- 会话结束后把本次会话追加到 `记忆/daily/YYYY/MM/YYYY-MM-DD.md`。
- 再由 memory agent 维护工作空间级 `记忆/MEMORY.md`。
- 新会话启动时注入 `MEMORY.md` 与 `Wiki/index.md`。

现有问题不是缺少记忆链路，而是 daily 抽取粒度偏自由：同一类信息有时写成决策，有时写成事实，有时直接进入 MEMORY，后续很难稳定判断可信度、覆盖关系和当前应注入的结论。

本任务只做“对话记忆抽取细化”：保留日期主轴，把当前按时间沉淀的 daily 改成结构化 L1 Atom，再从 L1 聚合 L2 Scenario 和 L3 MEMORY。

## 范围

做：

- L1：按日期 Markdown 存储结构化 Atom。
- L2：按场景 Markdown 聚合 L1 Atom。
- L3：`记忆/MEMORY.md` 只保留当前启动注入需要的有效结论。
- 明确日期在可信度、冲突裁决和注入选择中的作用。
- 更新 summary/memory agent prompt、解析、测试和模块梳理文档。

不做：

- 不做长任务上下文压缩。
- 不引入 Mermaid 任务画布。
- 不接入 Kuzu graph。
- 不把 L1 真源改成 SQLite/JSONL。
- 不做旧数据迁移；项目还没有用户，旧 daily 可以保持历史格式，也可以由后续维护任务自然重写新格式。

## 核心判断

日期不是审计附属字段，而是记忆可信度的一等依据。

新的组织方式是：

```txt
按日期沉淀
按字段抽取
按场景使用
按新近程度和显式程度裁决
```

同一主题发生冲突时，默认更新、更明确、更接近当前对话的记忆更可信。但不能机械地只按日期覆盖：显式命令、长期稳定偏好、一次性上下文需要不同权重。

## 存储真源

### L1 Atom

L1 真源仍然是日期 Markdown：

```txt
记忆/daily/YYYY/MM/YYYY-MM-DD.md
```

不新增 L1 数据库作为真源。数据库未来可以作为检索索引或诊断缓存，但不能成为唯一来源。

原因：

- 日期本身就是可信度和冲突裁决的重要信号。
- Markdown 可读、可审查、可手工修正。
- 现有 RAG、Wiki、搜索和备份链路已经覆盖 `记忆/`。
- ridge 当前问题是抽取结构不稳定，不是底层存储不够复杂。

### L2 Scenario

L2 是从 L1 聚合出的场景页：

```txt
记忆/scenarios/<scenario-slug>.md
```

L2 不保存不可追溯的新事实，只保存场景当前结论、覆盖关系和 Atom 引用。

### L3 MEMORY

L3 继续使用：

```txt
记忆/MEMORY.md
```

`MEMORY.md` 只放启动注入需要的当前有效结论。它不是完整记忆库，也不承担审计功能。

## L1 Atom 格式

daily 文件仍按日期和会话追加，但每个会话条目内必须包含结构化 Atom。

示例：

```md
# 2026-05-17

## 10:32 session_xxx 对话记忆抽取讨论

### L1-Atom

- id: mem_20260517_1032_001
  type: decision
  scope: module:memory
  status: active
  confidence: explicit
  observedAt: 2026-05-17T10:32:00+08:00
  sourceSessionId: session_xxx
  evidence: "日期非常好。因为日期代表着可信度"
  content: "对话记忆 L1 继续按日期 Markdown 存储，日期参与可信度和冲突裁决。"

- id: mem_20260517_1032_002
  type: constraint
  scope: module:memory
  status: active
  confidence: explicit
  observedAt: 2026-05-17T10:32:00+08:00
  sourceSessionId: session_xxx
  evidence: "不做这个了"
  content: "本轮记忆细化不接入 Kuzu graph。"
```

字段说明：

| 字段 | 说明 |
|------|------|
| `id` | 稳定 Atom ID，建议由日期、时间、序号生成 |
| `type` | `preference | decision | constraint | lesson | fact | workflow` |
| `scope` | `global | pi_web | module:<name> | task:<id> | repo:<name>` |
| `status` | `active | superseded | disputed` |
| `confidence` | `explicit | inferred` |
| `observedAt` | 记忆发生时间，使用 ISO 时间 |
| `sourceSessionId` | 来源会话 |
| `evidence` | 用户原话或短证据，必须足够支持 content |
| `content` | 原子记忆正文，只表达一个事实、偏好、决策或教训 |

## Atom 类型

- `preference`：用户长期偏好或协作方式。
- `decision`：项目或产品决策。
- `constraint`：禁止项、边界、安全要求、技术约束。
- `lesson`：从错误或审查中沉淀的经验。
- `fact`：当前项目事实。
- `workflow`：可复用流程或 SOP。

抽取规则：

- 一个 Atom 只表达一个点。
- 必须能用 `evidence` 支撑。
- 临时操作、一次性命令、未确认推测不进 L1。
- 敏感信息不写入。
- 当前用户明确纠正旧规则时，必须生成新 Atom，并在 L2/L3 中让新结论生效。

## 冲突与可信度

冲突裁决顺序：

1. 当前用户最新话语优先于所有历史记忆。
2. 显式命令优先于普通对话推断。
3. 同类型同范围冲突时，较新的 `observedAt` 默认优先。
4. 长期稳定偏好不被一次性上下文轻易覆盖。
5. 被覆盖的 Atom 不删除，标记为 `superseded` 或在 L2 中列入“历史覆盖”。

L1 daily 可以追加新 Atom，不强制回写旧日期文件。覆盖关系优先在 L2 Scenario 中表达，避免频繁改历史账本。

## L2 Scenario 格式

场景页示例：

```md
# 对话记忆系统

## 当前结论

- L1 继续按日期 Markdown 存储，日期是可信度字段。
- L2 只做场景聚合，不产生无法追溯的新事实。
- L3 MEMORY 只注入当前有效结论。
- 不做长任务压缩、Mermaid 画布、Kuzu graph 接入。

## 依据 Atom

- `mem_20260517_1032_001`：2026-05-17 明确保留日期主轴。
- `mem_20260517_1032_002`：2026-05-17 明确不接入 graph。

## 历史覆盖

- 旧设想“日期只做审计/回放索引”已被 2026-05-17 讨论覆盖。
```

L2 维护规则：

- 按主题聚合，不按日期建文件。
- 必须引用 L1 Atom ID 和日期。
- 当前结论优先呈现，不把所有历史流水塞入正文。
- 冲突、覆盖、废弃要显式写出。

## L3 MEMORY 规则

`MEMORY.md` 只写当前高价值结论：

```md
# MEMORY

- [module:memory][2026-05-17] 对话记忆 L1 以日期 Markdown 为真源；L2 按场景聚合；L3 只注入当前有效结论。
- [module:memory][2026-05-17] 日期是记忆可信度和冲突裁决的一等依据，同主题冲突时新近且明确的表达优先。
```

规则：

- 每条必须带 scope 和日期。
- 不写低价值进度。
- 不写完整证据；证据在 L1。
- L3 可以删除旧弱记忆，但不能让 L1 证据消失。

## Agent 改造

### summary agent

从“自由生成 decisions/facts/artifacts”改为生成：

```json
{
  "title": "",
  "summary": "",
  "atoms": [
    {
      "type": "decision",
      "scope": "module:memory",
      "confidence": "explicit",
      "evidence": "",
      "content": ""
    }
  ],
  "artifacts": []
}
```

服务端负责补齐：

- `id`
- `status`
- `observedAt`
- `sourceSessionId`

### memory agent

从“维护短句数组”改为：

- 读取当天 L1 Atom。
- 读取现有 L2 Scenario 和 L3 MEMORY。
- 输出：
  - 需要更新的 Scenario 页面。
  - 当前应写入 MEMORY 的 L3 条目。
  - 被新记忆覆盖的旧结论说明。

服务端仍需做敏感信息过滤、去重、长度预算和路径校验。

## API 与 UI

首版不新增用户操作 API。

现有可见面：

- 全局搜索继续能搜到 `记忆/daily`、`记忆/scenarios` 和 `MEMORY.md`。
- 知识诊断增加 Atom/Scenario 状态可以作为后续任务，不阻塞本任务。

## 测试要求

必须覆盖：

- 会话结束后 daily 写入结构化 L1 Atom。
- Atom 自动补齐 `id/status/observedAt/sourceSessionId`。
- summary agent 输出非法 type/scope/confidence 时被拒绝或过滤。
- 敏感信息不进入 L1/L2/L3。
- `MEMORY.md` 条目带日期和 scope。
- 同一主题新旧冲突时，L3 采用新近明确结论。
- L2 Scenario 引用 Atom ID，不写无法追溯的新事实。
- 空 L1/L2/L3 不注入。

修改 `.ts` 后必须运行根目录：

```bash
npm run check
```

## 验收标准

- daily Markdown 从自由段落升级为结构化 Atom。
- 日期仍是 L1 真源组织方式。
- L2 能表达当前场景结论和历史覆盖关系。
- L3 只注入当前有效结论，并带日期。
- 不新增 graph、Mermaid、长任务压缩、L1 数据库真源。
- 模块梳理文档同步更新。
