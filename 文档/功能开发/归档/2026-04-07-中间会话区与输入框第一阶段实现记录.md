# 中间会话区与输入框第一阶段实现记录

## 本次落地范围

本次实现没有去做 openchamber 全量能力复刻，而是优先把文档里第一阶段最关键的底座落到了当前项目代码里：

- 新会话草稿态
- 前端按 sessionId 分桶缓存消息快照
- 邻近会话预取与 hover 预取入口
- 中间区父会话回退入口
- 中间区回到底部与脱离底部状态
- 历史窗口限窗加载与向上扩窗
- 草稿态目录驱动的资源/Agent 同步刷新

## 已完成的关键实现

### 1. 新建会话不再立即创建服务端 session

左栏点击“新建”或“子会话”后，前端现在会先进入显式草稿态，而不是马上调用创建接口。

只有在用户第一次发送消息时，才会真正：

1. 根据草稿态里的 cwd 和 parentSessionId 创建 session
2. 把草稿迁移到真实 session 草稿桶
3. 发送首条消息并进入 SSE 流式生命周期

这保证了“会话工作台”与“真实持久化 session”是两个不同状态，不再混在一个按钮事件里。

### 2. 会话切换改成缓存优先

`usePiChat` 现在维护按 `sessionId` 分桶的 `sessionCache`：

- 已 hydrate 的会话直接秒切
- 未 hydrate 的会话才请求 `GET /api/sessions/:id`
- 同一 session 并发加载会复用同一个 Promise
- 通过请求序号避免慢响应覆盖新结果
- SSE snapshot 会回填缓存桶，而不是只更新当前页面数组

这次性能优化的重点不在样式层，而在状态层去掉“每次切会话都重新拉整份消息”的路径。

### 3. 预取从“没有”变成“有两层”

目前已经落了两类预取：

- 激活当前会话后，自动预取前后相邻会话
- 左栏鼠标 hover 或 focus 到会话节点时，触发该会话预取

这让用户在项目内来回切换会话时，更多时候命中前端缓存，而不是临时等待网络和服务端恢复。

### 4. 中间区开始具备工作台语义

本次没有只改 `usePiChat`，中间区 UI 也补上了工作台关键状态：

- 草稿态标签
- 父会话返回按钮
- 回到底部按钮
- pinned / 非 pinned 滚动状态
- 顶部“加载更早消息”扩窗按钮
- 草稿态下文件树根目录与资源目录同步

这意味着中间区不再只是“消息 map + 底部 textarea”，而是开始承载 session-aware 的工作区语义。

## 当前仍未做的部分

本次没有进入这些更重的能力：

- 阻塞式交互卡片
- 文件 mention 与附件协议
- 命令 / 技能 / agent 的高级补全面板联动
- turn 级时间线阅读器
- 虚拟列表

原因很明确：虽然现在已经补上了基础的历史限窗和向上扩窗接口，但阻塞卡片、时间线和文件进入对话语义仍然需要继续沿着同一条状态链往下做，不能退回页面补丁。

## 结论

这一阶段的重点不是“看起来更像 openchamber”，而是把中间会话区和输入框真正拉到正确的数据结构上。

只有先有：

- 草稿态
- session 级缓存桶
- 预取
- 父子会话回退
- 滚动状态管理

后续再继续加阻塞卡片、文件进入对话、长会话阅读器，才不会把问题重新堆回页面临时状态。
