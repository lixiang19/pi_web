# 2026-04-13 标签新建会话恢复草稿态与项目继承

## 背景

- 多标签工作台重构后，侧栏“新建会话”被改成了立即调用 `createSession()`。
- 这破坏了原本的草稿态语义：用户未发送消息也会落库空会话。
- 同时全局新建会话的 cwd 继承链被截断，只剩 `payload.cwd -> workspaceDir`，丢失当前活动项目上下文。

## 本次目标

- 恢复“新建会话先开草稿，首条消息发送时再创建 session”的主流程。
- 恢复 cwd 继承链：显式 payload 优先，其次当前活动标签 cwd，最后退回 workspaceDir。
- 保持多标签能力，不回退整个标签式工作台实现。

## 实施

### 1. 标签模型恢复草稿能力

- `useSessionTabs.ts` 为标签引入独立 `id`，不再拿 `sessionId` 兼做标签主键。
- 新增草稿标签能力：草稿标签 `sessionId` 为空，由前端管理，不触发服务端持久化。
- 已存在 session 仍按 `sessionId` 去重打开，避免重复标签。

### 2. 会话创建入口恢复为草稿

- `WorkbenchPage.vue` 的 `handleSessionCreate()` 改为只打开草稿标签。
- 删除“点新建即 createSession”的行为。
- 发送首条消息时，由 `usePerSessionChat.ts` 内部的 `ensureSession()` 负责真正创建 session。

### 3. 恢复当前项目继承链

- 全局新建会话的 cwd 解析改为：
  - `payload.cwd`
  - `tabs.activeTab.value?.cwd`
  - `core.info.value?.workspaceDir`
- 这样无论当前活动的是正式会话标签还是草稿标签，都能继承当前项目上下文。

### 4. 草稿标签转正式标签

- 草稿首发消息创建真实 session 后，`usePerSessionChat.ts` 会把当前标签绑定到新的 `sessionId`。
- 标签自身 `id` 保持不变，避免激活状态丢失。
- 标签标题、状态、cwd、父会话 id 继续由会话内容组件回写。

## 验证

- `npm run lint`
- `npm run typecheck`

## 结果

- 侧栏“新建会话”重新回到草稿态，不再制造空会话脏数据。
- 当前项目上下文继承恢复，顶栏加号不会无故掉回 workspace 根目录。
- 多标签架构保留，但草稿态重新成为合法的一等状态。
