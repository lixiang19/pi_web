# 输入框第一阶段 Spec 与里程碑

## 当前状态

- 当前阶段已完成并归档，代码实现已经进入主工作台链路。
- 已落地范围：
  - 会话级草稿保存与恢复
  - 模型、thinking、agent 三联选择
  - `@agent` 第一处有效提及覆盖显式 agent
  - prompt / skill / command 真实资源目录与 `/` 资源选择器
  - 服务端 `model + thinkingLevel + agent` 输入协议扩展
- 已完成验证：
  - 根仓 `npm run build`
  - 根仓 `npm run check`

## 文档目标

- 只讨论当前项目的**输入框模块**，不展开中间消息区和左侧会话列表。
- 明确输入框能力必须以 **Pi 自身能力** 和 **Pi 扩展能力** 为边界，不按 `openchamber` 全量照搬。
- 输入框能力以 7 项核心能力组织：
  - 输入及发送
  - 模型选取
  - 思考能力选取
  - agent 选取
  - prompt 快速发送
  - skill 选取与发送
  - Pi 命令选择与发送
- 产出一份可以直接执行的设计文档，包含：
  - spec
  - 里程碑
  - 任务拆分
  - 明确不做项

## 背景结论

前一份研究文档已经确认：`openchamber` 的输入框本质上不是 textarea，而是输入编排器。

但是本项目现在不能直接照搬它的输入框设计，原因有三个：

1. 本项目的输入能力必须受限于 `@mariozechner/pi-coding-agent` 当前真实支持的能力。
2. 项目顶级约束明确要求：**只允许使用 SDK 接入，禁止 RPC 模式**。
3. `openchamber` 里一部分输入增强能力依赖它自己的 runtime、store、agent 注册体系和 shell 输入模式，本项目当前并不存在这些前提。

所以本轮不是“抄一个好看的输入框”，而是先定义：

- Pi 现在能支持什么输入语义
- 哪些能力需要通过 Pi 扩展自行建设
- 哪些能力当前禁止进入范围

---

## 当前仓库现状

当前 `pi_web` 的输入区只具备一个最小可用版本：

- 文本输入与发送
- 模型选择
- agent 选择
- 手写 quick prompt chips

但以下能力还没有进入真实链路：

- thinking level 选择
- prompt catalog
- skill catalog
- Pi 命令 catalog
- `/` 资源选择与快速发送

也就是说，当前实现还不是 openchamber 那种“输入编排器”，而更接近一个带模型和 agent 选择的基础聊天输入区。

## 一、Pi 侧真实能力边界

根据当前项目已接入的 `@mariozechner/pi-coding-agent` SDK，以及 Pi 扩展 API 文档，当前可以确认：

### 1. Pi 原生 / SDK 侧可依赖能力

- 基于 `SessionManager` + `createAgentSession()` 创建、打开、驱动会话。
- 发送用户输入，触发 agent 执行与流式消息返回。
- 获取 provider / model，并在发送时指定模型。
- 通过扩展注册：
  - `command`
  - `tool`
  - `event handler`
  - `custom UI`
- 通过扩展 API 主动：
  - `sendMessage()`
  - `sendUserMessage()`
  - `setSessionName()`
  - `appendEntry()`
  - `ctx.ui.*` 交互能力
- 从 openchamber 对 Pi SDK 的桥接方式可以进一步确认：
  - prompt 可以作为标准资源被发现并列给 Web 端
  - command 可以通过 catalog 方式列给 Web 端
  - skill 可以作为资源参与 `/` 选择与发送
  - 但这些都需要服务端主动暴露 catalog API，前端不能自己伪造

### 2. Pi 扩展侧可用于本项目的关键能力

输入框如果要支持“agent 选择/agent 提及”，不能假设 Pi 原生就有 openchamber 那套 agent 列表机制，而应该通过扩展自己建立：

- agent 列表来源
- agent 对应的提示词 / 工作流 / 命令封装
- agent 选择后的行为注入
- agent 切换后的会话内状态保持

也就是说，**agent 能力不是输入框自身功能，而是扩展能力投影到输入框中的一种交互入口**。

### 3. 当前明确不可直接照搬的能力

下面这些能力不能直接进入当前项目输入框第一阶段：

- `queue / queued messages`
- `!` 触发 shell 模式
- openchamber 自己的 runtime agent 体系
- openchamber 自己的 command / skill / shell 自动补全体系
- 依赖 openchamber 内部 store 才成立的输入工作流

## 二、本轮输入框的设计原则

### 原则 1：输入框只承载 Pi 已支持的语义

如果 Pi SDK 现阶段没有明确承载某种输入语义，就不应该先在前端伪造交互。

### 原则 2：agent 能力必须被定义为“扩展能力”

输入框只负责：

- 展示可用 agent
- 让用户选择 / 提及 agent
- 把选择结果传给后端桥接层

真正的 agent 行为必须来自 Pi 扩展，而不是在前端硬编码 prompt 模板。

### 原则 3：先做发送链路闭环，再做输入增强

优先级必须是：

1. 新会话草稿态
2. 会话级输入草稿
3. 发送 / 停止
4. agent 扩展接入
5. 其他增强

不能一开始就做一堆补全、快捷操作、复杂输入模式。

### 原则 4：不为了像 openchamber 而引入伪能力

`openchamber` 能做，不代表当前项目现在就该做。凡是依赖它私有 runtime 的能力，一律拆开重新定义。

## 三、本轮明确范围

## 要做

### A. 输入框基础能力

- 支持新会话草稿态下直接输入。
- 支持当前会话输入与发送。
- 支持发送中切换为“停止”。
- 支持会话级草稿保存与恢复。
- 支持基础快捷键发送。
- 支持模型选择继续保留并与发送联动。
- 支持 prompt 快捷注入与发送入口。

### B. 输入框状态分层

输入框至少要拆出四类状态：

- `draft state`
  - 当前文本
  - 会话级本地草稿
- `session send state`
  - 是否发送中
  - 是否可停止
- `selection state`
  - 当前模型
  - 当前 agent（如已启用）
- `ui state`
  - 聚焦
  - 禁用
  - 错误提示

### C. agent 第一阶段能力

本轮允许做 agent，但必须限定为：

- agent 由 Pi 扩展定义，不由前端硬编码
- 输入框提供 agent 选择入口
- 输入框可支持 `@agent` 形式的显式提及
- 当前会话要保存本次 agent 选择
- 后端桥接层要把 agent 选择传入 Pi 会话执行链路

### D. 模型与思考能力选择

本轮输入框必须把模型与 thinking 视为一组联动能力，而不是只保留一个模型下拉：

- 输入框提供模型选择入口
- 输入框提供 thinking level 选择入口
- 会话发送时同时带上模型与 thinking
- agent frontmatter 中声明的 model / thinking 与当前显式选择之间必须有固定优先级

### E. prompt / skill / 命令资源选择

本轮要把这三类能力纳入输入编排，不再停留在手写 quick prompt：

- prompt：Pi 原生资源，支持快速注入或直接发送
- skill：作为 `/` 触发的资源选择项
- command：作为 `/` 触发的可执行资源选择项

它们在 UI 上应共享同一类“资源选择器”入口，但在发送语义上必须区分：

- prompt：注入 prompt 模板文本或按资源语义发送
- skill：注入 `/skill-name` 形式的调用入口
- command：注入 `/command` 并走命令语义

### F. 服务端桥接补齐

当前项目发送接口只有：

- `prompt`
- `model`

这不够支撑 agent 输入。因此本轮 spec 要求服务端补齐输入协议，至少支持：

- `prompt`
- `model`
- `agent` 或 `agentName`
- `thinkingLevel`
- 为后续扩展预留稳定结构

## 四、本轮明确不做

下面这些内容直接排除，不进入本轮实现：

### 1. 不做队列输入

不做：

- queued messages
- queue mode
- Enter / Ctrl+Enter 双语义切换
- 多条消息合并发送

原因：

- Pi 当前接入链路没有这个产品必要性。
- 这会显著抬高输入状态复杂度。
- 这是 openchamber 的工作流产品决策，不是 Pi 的基础能力。

### 2. 不做 `!` 触发 shell 输入模式

不做：

- `inputMode = normal | shell`
- `!` 前缀切换 shell
- shell 专用 placeholder / 发送链路 / UI 状态

原因：

- 当前项目不应伪造 shell agent 语义。
- Pi 侧没有现成的“输入框 shell 模式”契约可直接承接。
- 这会把输入框从会话输入器带偏为命令调度器。

### 3. 不做 openchamber 全量多模式补全体系

不做：

- 文件 mention 自动补全全套工作流
- shell autocomplete
- queued quick action 体系
- command / prompt / files / agents 全部塞进同一个重型多 tab 面板

说明：

- 本轮仍然要做 `/` 触发的 prompt / skill / command 资源选择。
- 但不直接照搬 openchamber 那套覆盖 prompts、agents、files 的完整自动补全系统。
- 当前项目应该先做受控的资源选择器，而不是把所有输入增强一次性端上来。

### 4. 不做语音输入

本轮不接：

- Browser voice
- conversation mode
- 语音转文本自动发送

原因：

- 不是当前输入框第一阶段主路径。
- 与 agent / 扩展 / 会话发送主闭环相比优先级过低。

### 5. 不做复杂附件工作流

本轮不做：

- 拖拽即附件
- inline 文件 mention 自动转附件
- 输入框粘贴文件
- 多来源文件注入

说明：

- 如果后续要做，应该先从 Pi 的真实上下文注入能力重新设计。
- 不能直接照抄 openchamber 的文件输入语义。

### 6. 不做 agent 配置前端编辑页

本轮不做：

- agent 配置 CRUD 页面
- agent YAML 可视化编辑器
- agent 权限表单编辑 UI

原因：

- 这不属于输入框第一阶段范围。
- 它和“输入框是否能正确选择并发送 agent”不是一条链路。
- 当前优先级应该放在输入编排和执行闭环，而不是补一个管理台。

这意味着和输入框相关的后续路线里，agent 配置前端页面明确废弃，不进入本文件里程碑。

### 7. 不做 shell 模式回归

即便 openchamber 里存在 `!` 切 shell 的路径，本轮也明确不做：

- `normal | shell` 双输入模式
- shell mode 专属快捷键和 UI 提示
- shell 相关自动补全与命令面板

原因：

- 用户已经明确禁用这条路线。
- 它会把输入编排和 Pi 主会话语义再次混杂。

## 五、第一阶段目标形态

本轮完成后，输入框应该达到下面这个状态：

### 用户侧可见能力

- 可以在没有现有会话时直接输入，形成新会话。
- 可以在已有会话里继续输入。
- 发送过程中主按钮变成停止。
- 切换会话后，能恢复该会话未发出的草稿。
- 可以选择模型。
- 可以选择 thinking level。
- 如果 agent 扩展已安装并可用：
  - 可以从输入框选择 agent
  - 可以用 `@agent` 明示本次输入要交给哪个 agent
- 可以通过 `/` 调出 prompt / skill / command 资源选择器。
- 可以通过 prompt 快捷入口快速把常用 prompt 放入输入区或直接发送。

### 架构侧达成目标

- 输入框不再只是绑定一个 `input` 字符串。
- 前端有清晰的输入状态模型。
- 服务端输入协议开始支持 agent。
- 服务端输入协议开始支持 thinking level。
- 服务端需要提供 prompt / skill / command catalog。
- Pi 扩展成为 agent 的唯一来源。
- prompt / skill / command 不再由前端硬编码。

## 六、Spec

## 1. 输入模型

建议新增统一输入模型，而不是继续散落在页面变量中：

```ts
interface ChatComposerState {
  sessionId: string | null
  draftText: string
  isSending: boolean
  canAbort: boolean
  selectedModel: string
  selectedAgent?: string
  hasDraft: boolean
}
```

说明：

- `selectedAgent` 是输入编排状态，不是 UI 临时变量。
- `sessionId = null` 时表示新会话草稿态。
- `hasDraft` 用于判断是否需要持久化与恢复。

## 2. 草稿持久化

### 目标

- 每个会话单独保存一份草稿。
- 新会话草稿态也有独立存储键。

### 规则

- 切会话时自动保存当前草稿。
- 进入会话时自动恢复草稿。
- 发送成功后清空当前会话草稿。
- 删除会话时同步清理草稿。

## 3. 发送协议

当前发送接口需要升级为稳定输入协议，而不是只收 `prompt`。

建议最小结构：

```ts
interface SendMessagePayload {
  prompt: string
  model?: string
  agent?: string
  thinkingLevel?: 'off' | 'minimal' | 'low' | 'medium' | 'high' | 'xhigh'
}
```

后续如果需要扩展，再升级为更宽的输入对象，但本轮不要过度设计。

## 4. 停止协议

输入框主按钮在发送中切换为停止按钮。

规则：

- `isSending = false` 时：点击主按钮发送。
- `isSending = true` 时：点击主按钮中止当前 session。
- 中止后回到可编辑状态。
- 不引入延迟兜底，不做模糊状态。

## 5. agent 选择能力

### 定义

agent 不是前端自带枚举，而是**扩展公开给本项目的可选能力集合**。

### 前端职责

- 展示当前可用 agent 列表。
- 允许用户选择一个当前 agent。
- 可选支持在文本中输入 `@agent`。
- 将最终 agent 选择写入本次发送 payload。
- 当前会话保存最近一次 agent 选择。

### 服务端职责

- 暴露 agent 列表接口。
- 暴露当前 session 默认 agent 或最近 agent。
- 发送消息时把 agent 选择传入 Pi 侧。
- 如果显式指定了 thinking level，要传入 Pi 会话执行链路。

### 扩展职责

- 声明有哪些 agent。
- 定义每个 agent 的行为和入口。
- 负责 agent 的实际执行逻辑。

## 6.1 模型与 thinking 规则

模型和 thinking 是同一组控制器，不应该拆成互不相干的两个小下拉。

建议规则：

- 模型由输入区显式选择
- thinking level 由输入区显式选择
- 若当前 agent frontmatter 自带 `model/thinking`，则按下面优先级归一：

1. 当前输入框显式选择
2. 当前 session 最近一次显式选择
3. agent frontmatter 默认值
4. 系统默认值

这样可以避免 agent 一切换就把用户刚选的模型和思考等级静默覆盖掉。

## 6. `@agent` 规则

第一阶段只支持最简单规则：

- 文本中出现 `@agentName` 时，尝试匹配可用 agent。
- 只认第一处有效提及。
- 识别成功后：
  - 发送 payload 带上该 agent
  - 文本本身不需要做复杂二次改写
- 如果未匹配成功，则按普通文本处理。

输入编排层必须把优先级写死，不允许在页面组件和服务端路由里各自判断：

1. 文本中第一处有效 `@agent`
2. 当前输入框显式选择的 agent
3. 当前 session 最近一次 agent
4. 无 agent

这意味着后续实现时必须满足：

- `@agent` 命中后覆盖当前选择器 agent
- 未命中时回退到当前选择器 agent
- 发送前只产出一个最终 `selectedAgent`
- 这一归一过程属于输入编排层职责，而不是页面展示逻辑

说明：

- 这里可以借鉴 openchamber 的 `parseAgentMentions` 思路。
- 但不能把 openchamber 整套 store / runtime / mode 依赖照搬进来。

## 7. 错误处理

必须明确处理：

- 没有模型时禁止发送
- 空输入禁止发送
- agent 被选中但已不可用时禁止发送并提示
- thinking level 非法时禁止发送并回退到当前稳定值
- prompt / skill / command catalog 加载失败时，输入区仍允许普通文本发送
- 发送失败后保留草稿
- 中止后保留草稿

## 8. prompt / skill / 命令选择规则

这三类能力在输入区要统一成“资源选择器”，但语义不同：

### prompt

- 入口一：常用 prompt chips
- 入口二：`/` 触发的 prompt 资源选择
- 作用：快速把标准 prompt 放入输入区，或在未来支持直接按模板语义发送

### skill

- 通过 `/skill-name` 形式插入输入区
- skill 选择器只负责发现和补全，不负责伪造 skill 逻辑
- skill 的实际语义必须由 Pi 资源系统决定

### command

- 通过 `/command` 形式插入输入区
- 内置命令和动态命令都应该走统一列表
- 命令执行语义必须明确区分“本地 UI 命令”和“Pi 命令”

当前项目建议第一阶段只支持 Pi 命令，不扩展本地 undo/redo/timeline 这类 openchamber 私有命令。

## 9. 输入区 UI 与样式方向

openchamber 值得借鉴的不是视觉皮肤，而是控制区层次：

### 桌面端结构

建议输入区采用三层：

1. 顶部控制条
  - 模型
  - thinking
  - agent
2. 中部快速资源条
  - prompt chips
  - `/` 打开资源选择器按钮
3. 底部输入与发送条
  - textarea
  - send / stop

### 自动完成浮层

- 浮层从输入区上方弹出
- 固定为轻量列表，不做 openchamber 那种 prompts / agents / files 三 tab 总面板
- 第一阶段只承载 prompt / skill / command 三类资源

### 视觉原则

- 控制条与输入区分层明显
- send / stop 是唯一主按钮切换
- prompt chips 是辅助入口，不抢主视觉
- 不把所有能力都塞进单行 toolbar

## 七、里程碑

## 里程碑 M1：输入与发送闭环

### 目标

建立一个真正可用的输入框，而不是演示级输入栏。

### 交付结果

- 新会话草稿态
- 会话级草稿恢复
- 发送 / 停止二态按钮
- 前端输入状态模型成型
- 输入区基础布局成型

### 任务

- 梳理现有 `usePiChat` 输入与发送状态

## 九、关键设计决策

## 决策 1：不引入 queue

理由：

- 不是 Pi 当前必须能力
- 复杂度高
- 会污染基础输入闭环

## 决策 2：不引入 shell 模式

理由：

- 不是当前产品主路径
- 不能用 `!` 伪造一套 shell 语义
- 会把输入框架构带偏

## 决策 3：agent 必须走扩展

理由：

- 只有这样才能让 agent 真正绑定 Pi 的能力系统
- 只有这样才能避免前端硬编码 prompt 模板
- 只有这样后续才能扩展成多 agent 工作台

## 决策 4：本轮只做最小 `@agent`

理由：

- 只解决显式声明 agent 的核心问题
- 不引入 openchamber 那套大规模补全系统
- 保持输入框第一阶段复杂度可控

## 十、风险与前置条件

### 风险 1：当前服务端桥接可能还没有 agent 执行入口

如果 Pi SDK 当前会话发送链路没有现成的 agent 参数入口，需要先在扩展层定义 agent 注入方式，再决定服务端如何桥接。

### 风险 2：当前仓库还没有现成的 Pi 扩展接入层

这意味着 M2 不只是前端开发，还包含：

- 扩展加载方案
- 扩展发现方案
- agent 列表暴露方案

### 风险 3：不要把 agent 做成 UI 假象

如果前端只是让用户“选了一个 agent”，但 Pi 执行时根本没有 agent 语义，那这就是错误设计，不能接受。

## 十一、验收标准

### M1 验收

- 新会话可直接输入并发送
- 发送中可停止
- 切会话后草稿能恢复
- 发送成功后草稿清空
- 删除会话后草稿清理

### M2 验收

- 输入框可选择模型、thinking、agent
- 当前会话可记住模型、thinking、agent
- 发送时服务端确实收到 `model + thinkingLevel + agent`
- Pi 执行链路确实消费了这些选择

### M3 验收

- 系统能返回 prompt / skill / command catalog
- `/` 能打开资源选择器
- prompt / skill / command 能正确注入输入区
- prompt chips 来源于真实 catalog，而不是前端常量

### M4 验收

- `@agent` 能识别第一处有效匹配
- 未匹配时按普通文本发送
- agent 选择与 `@agent` 的优先级规则稳定一致
- `@agent` 命中时覆盖当前选择器 agent

## 结论

本轮输入框设计不追求 openchamber 的“全能输入工作台”，而是先把真正能落地的 Pi 能力立起来。

因此本项目输入框第一阶段的正确方向是：

- **做实基础发送闭环**
- **把模型、thinking、agent 作为输入控制核心**
- **把 agent 严格定义为扩展能力**
- **把 prompt / skill / command 接入真实资源目录**
- **拒绝 queue 和 shell 模式**
- **只在 Pi 真正支持的前提下增加输入增强**

后续只要 M1 和 M2 打稳，输入框就会从“聊天框”升级为“Pi 工作台的真实输入入口”，而不是继续停留在页面表层。

## 十三、对标 openchamber 的实现边界

结合本次对 openchamber 的补充探索，当前项目明确采用以下对标方式：

- 借鉴它的输入编排结构：控制条、资源选择器、textarea、send/stop
- 借鉴它的状态分层：草稿、发送、选择、UI
- 借鉴它的 `@agent` 解析和 `/` 资源选择思路

但明确不照搬：

- queue mode
- shell mode
- files / agents / prompts 三 tab 重型自动完成面板
- 本地 undo / redo / timeline 这类 openchamber 私有命令
- 权限系统和附件系统混入输入框主路径
