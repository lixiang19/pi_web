# 非 AI 会话工作台 MVP 范围稿 v0.1

> 任务：task #3 梳理“除了 AI 会话的其他部分”  
> 日期：2026-05-01  
> 产品决策：跳过用户另有设计的仪表盘，不继续扩展 AI 会话链路，先把个人工作空间基础内容层做顺。

## 1. 当前方向

先暂停两条线：

- **暂停仪表盘新设计**：不继续基于现有 `DashboardView` 扩入口、卡片或首页结构。
- **暂停 AI 会话扩展**：不继续新增 session 接线、Agent 编排、AI 交付验收或聊天结果沉淀。

下一步聚焦：**非 AI 会话的个人工作台基础能力**。

一句话目标：让用户不用 AI 会话，也能在 ridge 里完成“捕捉内容 → 整理资料 → 拆成任务 → 打开编辑 → 回访继续”的个人工作空间闭环。

## 2. 产品原则

1. **先内容层，后智能层**：文件、笔记、任务、收件箱、日历这些基础对象先稳定，AI 后续只是叠加在其上。
2. **先闭环，后大而全**：不同时扩展文件管理、数据库、日历、任务高级字段；先打通一个用户每天能用的轻量闭环。
3. **尊重现有实现**：已有可用能力优先保留；只做体验断点和组织方式调整，不重写成熟模块。
4. **冻结不确定区域**：仪表盘新设计由用户另行决定；本阶段不抢先设计。
5. **不误动 AI 会话链路**：除非修复阻塞问题，本阶段开发不改 session、Agent、Pi bridge 主链路。

## 3. 建议优先级

### P0：内容捕捉与整理闭环

这是下一步最建议先做的部分。

闭环：

1. 用户把闪念、资料、链接或临时文字放入收件箱。
2. 用户在收件箱里快速预览、搜索、重命名。
3. 用户把内容归档到项目/笔记/阅读等目录。
4. 用户能从内容中拆出待办，或把 checkbox 待办聚合到待办视图。
5. 用户能通过文件树和标签页重新打开相关内容继续编辑。

为什么先做它：

- 不依赖 AI 会话和仪表盘新设计。
- 与现有 `InboxView`、`FileTreePanel`、`WorkspaceMarkdownEditor`、`TaskView` 能力直接相关。
- 能验证 ridge 是否是一个“每天打开的个人工作空间”。

### P1：任务系统基础体验

目标：让用户能管理非 AI 场景下的行动项。

范围：

- 普通任务创建、编辑、删除、完成。
- markdown checkbox 任务聚合和回写。
- 清单 / 今日 / 看板三种视图保持一致。
- 后续可考虑优先级、截止日期、标签，但不建议第一批就做。

### P2：项目/目标组织结构

目标：定义用户如何组织“一个目标/项目”的资料、任务和文件。

范围：

- 先采用文件夹 + 笔记 + 待办的轻量组织方式。
- 暂不新增 Project 表或复杂目标对象。
- 可定义推荐目录：`项目/`、`笔记/`、`收件箱/`、`日记/`、`阅读/`。

### P3：日历 / 日记回访

目标：让用户按时间回顾和继续。

范围：

- 日记创建与打开。
- 日历入口到当天日记。
- 今日任务与日记关系后续再设计。

### P4：数据库 Bases

目标：作为结构化内容能力保留，但不作为当前优先级。

原因：能力面大，容易变成 Notion Database 重建；建议等文件/笔记/任务闭环稳定后再推进。

## 4. 建议第一个非 AI 开发切片

建议命名：**Phase A：收件箱到工作空间整理闭环**。

根据开发侧核查，Phase A 需要拆成 A1/A2，避免把尚未实现的 markdown checkbox 聚合与回写当作已有能力。

### 用户故事

作为个人用户，我可以快速捕捉一个想法，把它从收件箱整理到合适目录，并在整理过程中产生可追踪待办，之后能从文件树或待办视图继续。

### 目标路径

1. 在收件箱创建一条闪念。
2. 在收件箱列表看到它，能预览、搜索、重命名。
3. 归档到目标目录，例如 `项目/xxx/` 或 `笔记/`。
4. 打开归档后的 markdown 文件继续编辑。
5. 后续再把文件里的 checkbox 待办聚合到待办视图，并支持回写完成状态。

### 优先检查已有实现

当前开发核查显示：A1 大多已有实现但存在小修；A2 是主要缺口，需要单独 spec/测试/实现。

- `InboxView`：捕捉、列表、预览、搜索、排序、归档、重命名、删除。
- `createNote({ path })`：支持指定目录创建。
- `FileTreePanel`：文件树展示与打开。
- `WorkspaceMarkdownEditor`：markdown 编辑与保存。
- `TaskView`：当前主要是 `.ridge/tasks.json` 独立任务；markdown checkbox 聚合与回写需要作为 A2 单独实现。

## 5. Phase A 验收标准

### A1：收件箱归档后打开 + 子目录路径准确性

#### A. 收件箱捕捉

- [ ] 用户能在收件箱创建一条闪念。
- [ ] 新闪念保存到 `收件箱/` 目录，而不是工作空间根目录。
- [ ] 列表中能看到新闪念，包含标题、预览和相对时间。
- [ ] 空内容不创建或有明确提示。

#### B. 整理归档

- [ ] 用户能把闪念归档到已有目录。
- [ ] 如果产品确认支持 `项目/xxx/`，归档目标选择需要支持子目录浏览，而不是只列根目录一级目录。
- [ ] 归档后收件箱列表不再显示该条。
- [ ] 文件树目标目录能看到归档后的文件。
- [ ] 点击归档后的文件能打开 markdown 编辑器。

#### C. 编辑与继续

- [ ] 归档成功后可直接打开目标 markdown 文件。
- [ ] 归档后的 markdown 内容可编辑并自动保存。
- [ ] 刷新后内容仍存在。
- [ ] 文件路径在前端内部保持绝对路径，调后端时转换为相对路径。

### A2：markdown checkbox 聚合与回写

- [ ] markdown 中的 `- [ ]` checkbox 能出现在待办视图。
- [ ] 在待办视图勾选 checkbox 后，原 markdown 文件同步变为 `- [x]`。
- [ ] 刷新后待办状态与 markdown 文件一致。
- [ ] A2 需要单独 spec/测试/实现，不能作为 A1 的已有能力验收。

### 范围边界

- [ ] 不改 AI 会话、Agent、Pi session 接线。
- [ ] 不改用户另有设计的仪表盘结构。
- [ ] 不新增 Project / Goal / Artifact 等新一等对象。
- [ ] 不扩展数据库 Bases。
- [ ] 不做团队协作和云同步。

## 6. 保留 / 调整 / 冻结

### 保留

- 工作空间作为唯一根路由。
- 左侧固定视图入口：待办、日历、收件箱、文件树。
- 文件树 + 分屏多标签页。
- markdown 编辑、文件预览、默认应用打开。
- 收件箱现有捕捉/归档/搜索/重命名/删除能力。
- 待办现有独立任务与 checkbox 聚合能力。

### 可能调整

- 收件箱到目标目录的整理流程是否足够顺。
- 归档后是否需要 toast 提供“打开文件”。
- 待办视图中 checkbox 来源是否需要更明显展示来源文件。
- 文件树新建按钮的默认目录和命名体验。

### 冻结

- 仪表盘新首页设计。
- AI 会话入口、session tab、Agent 编排。
- AI 交付验收、Artifact、待我确认。
- Bases 高级视图与字段系统。
