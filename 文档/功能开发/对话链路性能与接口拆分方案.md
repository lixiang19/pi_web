# 对话链路性能与接口拆分方案

- 状态：草案
- 日期：2026-04-14
- Owner：AI / ridge
- 目标版本/里程碑：当前工作台对话基础链路重构

## 1. 背景与问题

- 背景：当前 ridge 最基础功能是对话。链路横跨左侧会话列表、中间消息区、右侧文件/Git 面板、Web 前端状态层、Node 服务桥接层、Pi session runtime。用户要求先审视整条前后端流程的性能，再决定执行改造。
- 现象/痛点：
  - 服务端 `listProjectSessions()` 同时承担会话扫描、项目归属解析、worktree/git 上下文解析、parent/child 建树、metadata 融合，职责过肥，且被多个热路径接口重复调用。
  - `GET /api/sessions` 当前把会话摘要和上下文重字段混在一起，导致列表接口同时承担“列表显示数据”和“上下文字典数据”两类职责。
  - `GET /api/sessions/:id` 同时承载消息窗口、interactive requests、permission requests、上下文、运行时选择信息，导致单会话快照过大。
  - 中间区流式渲染存在热路径重算：`WorkbenchMessageStream.vue` 使用 `JSON.stringify(props.messages.at(-1)?.content ?? "")` 监听最后一条消息内容；消息轮次布局每次流式更新都重新分组计算。
  - 用户明确要求保留 `v-show` 与全部标签常驻，因为这是当前秒切体验基础；因此不能靠卸载标签页解决问题，必须从接口边界和热路径本身下手。
- 成功定义：保留当前标签页秒切体验与 UI 信息量，把大接口拆成清晰小接口，去掉重复数据和热路径重算，让启动、切会话、发消息、流式更新都更轻。

## 2. 目标与非目标

- 目标：
  - 拆分当前对话链路中的肥大接口，按“列表摘要 / 上下文字典 / 消息窗口 / 运行时信息”分离职责。
  - 保留左栏当前信息表达能力，但改为由 `sessions + session-contexts` 拼装，而不是单个大接口重复返回上下文重字段。
  - 保留 `v-show` 与全部标签常驻策略，不改变当前秒切产品决策。
  - 优化前端中间区流式热路径，移除明显高频重算和大对象序列化监听。
  - 收敛资源、会话、运行时接口边界，避免“拿一点数据却重建整套上下文”。
- 非目标：
  - 不改成标签页按需卸载或 keep-alive/懒挂载方案。
  - 不引入复杂缓存系统、持久索引系统或双轨兼容接口。
  - 不改变会话存储格式，不做 Pi runtime 协议改造。
  - 不在本次顺手重做右侧文件树/Git 功能设计，只解决其对话主链路中的性能耦合点。

## 5. 方案概览（主方案）

### 5.1 方案摘要

- 一句话：保留标签页常驻秒切，拆掉会话链路中的肥大接口，把左栏上下文、单会话消息窗口、运行时信息分离，同时清理中间区流式热路径重算。
- 取舍：放弃“一个接口返回一切”的偷懒方式，换取更稳定的边界、更轻的传输、更少的重复聚合和更低的流式渲染成本。
- 影响面：
  - 服务端：`packages/server/src/index.ts`、相关类型定义、会话聚合逻辑、资源目录逻辑
  - 前端 API：`packages/web/src/lib/api.ts`、类型定义
  - 左栏：会话列表分组和上下文拼装逻辑
  - 中间区：每标签会话状态加载逻辑、消息流重算逻辑

### 5.2 风险与回滚

- 风险：
  - 左栏分组可能因新旧数据源切换出现 project/worktree 归组错误：通过引入显式 `contextId + context dictionary` 契约，并在左栏派生层集中验证缓解。
  - 中间区拆快照后，某些依赖 `SessionSnapshot` 全量字段的组件可能失联：通过先补类型和适配层，再切组件消费路径缓解。
  - `/api/resources` 若同步改太多，容易把 prompt/skill/command 目录发现也带坏：通过把资源接口拆分放进独立里程碑，优先只收敛会话主链路缓解。
- 回滚：
  - 回滚策略为 Git 级版本回退。
  - 不保留旧大接口与新小接口双轨长期共存；如中途验证失败，整批回滚，不做运行时兼容分支。

## 6. 详细设计（Spec）

### 6.1 交互/行为（如适用）

- 用户流程：
  1. 工作台启动后，左栏先请求 `GET /api/sessions` 与 `GET /api/session-contexts`。
  2. 前端用 `contextId` 将 session 摘要与上下文字典拼装成当前左栏需要的完整视图数据，并继续按 project/worktree 分组。
  3. 用户切换标签或新开标签时，不卸载标签内容；激活中的标签按需拉取 `messages` 与 `runtime`，非激活标签保持已驻留状态。
  4. 发送消息时，后端只处理会话运行时与消息追加，不再提前构造整份 session registry 输出。
  5. SSE 流式更新期间，中间区只增量更新当前消息窗口，不再对最后一条消息做整段 JSON 序列化监听。
- 边界条件：
  - 新建草稿会话仍要求先选项目路径，不改变当前产品决策。
  - ask / permission pending 仍固定在消息流底部，不独立成新协议层。
  - 左栏在 `session-contexts` 未返回前，需要可显示骨架或最小摘要，不得阻塞整个工作台可交互。
  - 若某 session 的 `contextId` 丢失或字典缺项，应明确视为服务端契约错误，而不是前端猜测修复。

### 6.2 接口与契约

- API/函数签名：
  - `GET /api/sessions`
    - 出参：`SessionListItem[]`
    - 字段：`id, title, status, updatedAt, archived, parentSessionId, contextId`
  - `GET /api/session-contexts`
    - 出参：`Record<contextId, SessionContextSummary>` 或 `SessionContextSummary[]`
    - 字段：`contextId, cwd, projectId, projectLabel, projectRoot, worktreeRoot, worktreeLabel, branch, isGit`
  - `GET /api/sessions/:id/messages?rounds=3`
    - 出参：`SessionMessagesPayload`
    - 字段：`sessionId, messages, historyMeta, interactiveRequests, permissionRequests`
  - `GET /api/sessions/:id/runtime`
    - 出参：`SessionRuntimePayload`
    - 字段：`sessionId, agent, model, thinkingLevel, resolvedModel, resolvedThinkingLevel`
  - `GET /api/sessions/:id/stream?rounds=3`
    - snapshot 首包改为只推送当前消息窗口与 pending requests，避免再塞整份 summary/context/runtime 混合对象。
  - `POST /api/sessions/:id/messages`
    - 入参维持 `prompt, model?, thinkingLevel?, agent?`
    - 内部依赖改为只确保会话 record 与 runtime，不再构建整份列表摘要。
- 错误码/异常：
  - `404 Session not found`：session 不存在。
  - `400 Session context missing`：session 已存在但无法解析到合法 `contextId` 或上下文字典缺失。
  - `400 Session project is required`：新会话未提供项目路径。
  - `400 当前 agent 的最大轮次已耗尽，请重新选择 agent 或新建会话`：沿用现有运行时约束。

### 6.3 数据与存储（如适用）

- Schema/字段：
  - 新增 `SessionListItem`
    - `id: string`
    - `title: string`
    - `status: 'idle' | 'streaming' | 'error'`
    - `updatedAt: number`
    - `archived: boolean`
    - `parentSessionId?: string`
    - `contextId: string`
  - 新增 `SessionContextSummary`
    - `contextId: string`
    - `cwd: string`
    - `projectId: string`
    - `projectLabel: string`
    - `projectRoot: string`
    - `worktreeRoot: string`
    - `worktreeLabel: string`
    - `branch?: string`
    - `isGit: boolean`
  - 新增 `SessionMessagesPayload`
    - `sessionId: string`
    - `messages: ChatMessage[]`
    - `historyMeta: SessionHistoryMeta`
    - `interactiveRequests: AskInteractiveRequest[]`
    - `permissionRequests: PermissionInteractiveRequest[]`
  - 新增 `SessionRuntimePayload`
    - `sessionId: string`
    - `agent?: string`
    - `model?: string`
    - `thinkingLevel?: ThinkingLevel`
    - `resolvedModel?: string`
    - `resolvedThinkingLevel?: ThinkingLevel`
- 迁移策略：
  - 不改 session 文件与 metadata 存储格式。
  - 迁移只发生在 HTTP 契约与前端消费层。
  - 旧 `SessionSummary / SessionSnapshot` 大对象不保留为运行时主契约，直接拆分替换。

### 6.4 兼容性与发布

- 兼容策略：不保留旧大接口作为长期兼容层；本次直接改新契约。
- 发布步骤：
  1. 先定义新类型和新接口出参。
  2. 服务端拆分聚合函数，产出 `sessions / session-contexts / messages / runtime` 四类数据源。
  3. 前端 API 与类型切到新契约。
  4. 左栏改成双源拼装；中间区改成按需读取 messages/runtime。
  5. 清理消息区流式热路径坏味道。
  6. 执行 `eslint` 与 `ts` 检查，再手动验证秒切与长对话流式更新。

### 6.5 可观测性

- 日志：
  - 服务端为 `sessions / session-contexts / messages / runtime` 四类接口记录基础 timing。
  - 流式链路记录 `snapshot 首包大小`、`rounds`、`message count`，便于比较拆分前后差异。
- 指标：
  - 工作台启动到左栏可见耗时。
  - 切换未缓存会话耗时。
  - 发消息到首个 assistant 增量出现耗时。
  - 流式期间 message stream 每秒更新次数下的主线程卡顿感。
- 告警：
  - 若 `contextId` 无法映射字典，视为高优先级契约错误。
  - 若拆分后左栏丢分组、消息区丢 pending ask/permission、运行时选择丢失，视为阻断级回归。
