# 对话信息流原生组件重构 方案

- 状态：草案
- 日期：2026-04-11
- Owner：AI / ridge
- 目标版本/里程碑：当前工作台对话区重构

## 1. 背景与问题

- 背景：当前 ridge 工作台中间对话区已经接入 `ai-elements-vue` 风格封装，但底层消息源仍然是 Pi 原生 `ChatMessage/ContentBlock` 数据结构。
- 现象/痛点：
  - 对话消息被 `packages/web/src/lib/ai-elements-adapter.ts` 二次转换，出现结构语义错位。
  - 截图已出现明显显示异常：消息块留白失衡、操作按钮悬浮位置异常、输入区布局裁切。
  - 输入区与消息流分别走 ai-elements 包装层，导致维护路径与 `usePiChat.ts` 的真实数据流断裂。
  - 工具调用、思考块、文本消息的展示方式被迫适配第三方结构，不符合 Pi 原生内容块模型。
- 成功定义：彻底移除 ai-elements-vue 在对话信息流中的承载角色，消息流和输入区直接消费 Pi 原生结构，并恢复稳定、简洁、可读的生产工作台显示。

## 2. 目标与非目标

- 目标：
  - 移除对话消息流与输入区对 ai-elements-vue 数据适配的依赖。
  - 基于现有自研组件文件路径重构 `WorkbenchMessageStream.vue`、`ChatMessageItem.vue`、`WorkbenchComposer.vue`。
  - 保留 Markdown 渲染能力，继续使用 `vue-stream-markdown` 正确渲染文本消息。
  - 让 `thinking / toolCall / toolResult` 默认折叠，以简洁摘要方式进入信息流。
  - 清理本次确认无用的 ai-elements 对话相关适配层与引用。
- 非目标：
  - 不在本次引入新的聊天协议、消息存储格式或后端接口。
  - 不做兼容保留式双轨渲染，不保留 ai-elements 作为运行时备用方案。
  - 不在本次做华丽视觉实验，优先交付稳定生产样式。

## 4. 资料与依据（Librarian）

- 关键结论：
  - `vue-stream-markdown` 当前项目内已在 `packages/web/src/components/ai-elements/message/MessageResponse.vue` 使用，说明 Markdown 渲染能力可以直接复用，无需引入新的文本引擎。
  - `usePiChat.ts` 已完整支持 Pi 原生流式事件和内容块累积，本次问题根源主要在前端表现层被错误抽象，而不是后端数据结构不足。
  - 项目明确要求“编辑代码时不要兼容、要从根本解决”，因此本次不应保留 ai-elements-vue 作为兜底路径。
  - 用户已明确确认：消息流与输入区一并回归自研组件体系，工具/思考块默认折叠，文本允许 Markdown 渲染。

## 5. 方案概览（主方案）

### 5.1 方案摘要

- 一句话：移除对话区 ai-elements 适配层，基于现有原生组件文件路径重构消息流与输入区，让工作台直接消费 Pi 原生 `ChatMessage/ContentBlock`。
- 取舍：直接回到单一原生数据模型，牺牲 ai-elements 的现成包装能力，换取架构一致性、展示可控性与后续扩展稳定性。
- 影响面：
  - 中间对话区 UI：`WorkbenchChatPanel.vue`
  - 消息流：`WorkbenchMessageStream.vue`、`ChatMessageItem.vue`
  - 输入区：`WorkbenchComposer.vue`
  - 折叠卡片：`ThinkingCard.vue`、`ToolCallCard.vue`
  - 清理：`ai-elements-adapter.ts` 及明确无用的 ai-elements 对话相关引用

### 5.2 风险与回滚

- 风险：
  - 消息块重构后可能出现滚动贴底回归：通过沿用 `WorkbenchMessageStream.vue` 现有滚动锚点逻辑缓解。
  - Markdown 渲染替换后可能引入代码块/长文本溢出：通过统一容器宽度、`prose` 样式与 `overflow-x-auto` 控制缓解。
  - 工具结果展示如果直接平铺，可能噪音过大：通过默认折叠、摘要头与内部 monospace 内容层缓解。
- 回滚：
  - 回滚策略为 Git 级版本回退；本次不保留运行时双实现开关，不做兼容分支。

## 6. 详细设计（Spec）

### 6.1 交互/行为（如适用）

- 用户流程：
  1. 用户进入工作台后，中间区由 `WorkbenchChatPanel` 统一装配头部、消息流、输入区。
  2. 已有会话时，`WorkbenchMessageStream` 直接渲染 `messages: ChatMessage[]`。
  3. 每条消息由 `ChatMessageItem` 按 `ContentBlock` 顺序输出：文本 Markdown、图片、思考块、工具调用块、工具结果块。
  4. `thinking / toolCall / toolResult` 默认折叠，展示简洁摘要，用户按需展开详情。
  5. 输入区 `WorkbenchComposer` 直接绑定 `composer.draftText`、模型/思考/agent 选择、资源按钮和发送/停止动作。
- 边界条件：
  - 无 active session 时，继续展示 `WelcomeEmptyState`，输入区仍保持当前受控禁用策略或按空态方案接管。
  - 长历史会话继续支持“加载更早消息”。
  - streaming 状态下，若用户停留底部，消息流自动贴底；否则显示回到底部按钮。
  - 工具结果内容为空、图片结果、错误结果都需有明确显示分支。

### 6.2 接口与契约

- API/函数签名：
  - `WorkbenchChatPanel`：继续接收 `messages / composer / status / agents / modelOptions / commands / prompts / skills` 等工作台聚合态，不新增后端接口。
  - `ChatMessageItem`：`message: ChatMessage`，内部按 `message.contentBlocks` 渲染，不再引入 adapter。
  - `ThinkingCard`：`block: ThinkingContentBlock`，默认折叠，显示摘要 + 可展开正文。
  - `ToolCallCard`：`block: ToolCallContentBlock | ToolResultContentBlock`、`isResult: boolean`，默认折叠，头部展示工具名、状态、错误态。
  - `WorkbenchComposer`：保持 `update:value / submit / selectModel / selectThinking / selectAgent / toggleResourcePicker / applyPrompt / injectCommand / injectSkill` 事件契约。
- 错误码/异常：
  - 不新增错误码。
  - `resourceError` 继续由资源选择器负责显示。
  - Markdown 渲染失败时应退化为普通文本展示，不能让整条消息消失。

### 6.3 数据与存储（如适用）

- Schema/字段：
  - 继续使用 `ChatMessage.contentBlocks` 原生字段，不新增消息存储 schema。
  - 继续使用 `ChatComposerState` 管理输入区受控状态。
- 迁移策略：
  - 无数据迁移。
  - 迁移仅发生在前端组件边界：删除 ai-elements 中间映射，回归原生字段直接消费。

### 6.4 兼容性与发布

- 兼容策略：不做旧实现兼容；本次直接替换当前对话区实现。
- 发布步骤：
  1. 重构原生消息流与输入区组件。
  2. 切换 `WorkbenchChatPanel` 引用。
  3. 删除明确无用的 ai-elements 对话相关适配与引用。
  4. 执行 eslint 与 TypeScript 检查。
  5. 手动验证截图中异常场景。

### 6.5 可观测性

- 日志：
  - 保持现有前后端日志，不新增埋点。
  - 必要时在开发期间观察 `usePiChat` streaming 更新与消息块顺序是否稳定。
- 指标：
  - 关注消息渲染正确率、历史扩窗后滚动稳定性、发送后自动贴底正确性。
- 告警：
