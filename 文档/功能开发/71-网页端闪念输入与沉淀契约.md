# 71 网页端闪念输入与沉淀契约

## 背景

网页端闪念不是“让 AI 判断要不要保留”的入口。用户已经主动把内容放进闪念，就表示这个素材需要被收进 ridge；AI 的职责是理解、转化、命名、补元信息、归类和沉淀，不是问用户是否剪藏、是否删除。

因此默认产品语义是：

```text
用户输入素材 -> 临时闪念队列/临时附件库 -> AI 主导整理 -> 正式知识产物
```

## 范围

本契约只约束网页端闪念输入，不覆盖 Android、桌面托盘、Chrome 插件的具体交互实现。

网页端输入类型先按以下几类处理：

- 文本
- 图片附件
- 录音附件
- PDF / Word / Markdown / 文档类附件
- URL

## 总原则

1. 纯文本直接进入 `fleeting_notes.content`。
2. 用户上传的文件先进入临时附件库：`<workspaceDir>/.ridge/fleeting-attachments/<noteId>/`。
3. AI 不判断“是否剪藏/是否保留”；用户输入即表示要沉淀。
4. AI 负责决定标题、摘要、标签、归类、正文结构和目标知识形态。
5. 不再恢复前端人工按钮：按建议、日记、任务、剪藏、附件处理、删除都不是默认交互。
6. 分析失败时保留重试入口；处理失败时保留闪念和临时附件。

## 文本

文本是最简单的输入：

- 写入 `fleeting_notes.content`。
- 不需要转化。
- AI 直接读取内容，生成后续正式产物。

## 图片附件

图片和其他附件必须区分。

图片不需要 OCR，因为当前模型具备视觉能力。默认处理方式：

1. 图片先进入临时附件库。
2. 分析时作为视觉输入提供给模型。
3. 不先转 Markdown。
4. 不走 Python Converter OCR，除非未来某个特殊模型明确不支持视觉。

图片是否进入正式附件库，由后续沉淀产物是否需要引用原图决定；但它不应该被 OCR 文本替代。

## 录音附件

录音是特殊附件。

录音先进入临时附件库，然后需要转写成 Markdown。和 PDF / Word 不同，原始录音本身有长期价值，所以最终应进入正式附件库。

目标流程：

```text
录音文件
  -> 临时附件库
  -> 音频转写 Markdown
  -> 原始录音迁移到正式 附件/
  -> 转写 Markdown 的 YAML 引用正式录音附件地址
```

转写 Markdown 的 YAML 示例：

```yaml
---
source_type: audio
original_name: recording.webm
mime_type: audio/webm
attachment_path: 附件/recording.webm
transcribed_at: 2026-05-21
---
```

规则：

- 转写 Markdown 是可读知识正文。
- 正式音频附件是原始证据。
- Markdown 必须能反向引用正式音频附件。
- 不能只保留转写文本而丢掉原音频。

## PDF / Word / 文档类附件

PDF、Word、Markdown、文本、表格等非图片文档附件先进入临时附件库，然后转 Markdown。

目标流程：

```text
文档文件
  -> 临时附件库
  -> 转 Markdown
  -> 写入剪藏/资料类正式 Markdown
```

默认不把 PDF / Word 原文件迁移到正式附件库。原因：

- 用户真正需要进入知识系统的是可读 Markdown。
- 原文件通常只是转换输入，不一定有长期引用价值。
- 后续如果要保留原件，应由明确的新规则或 AI 产物需要触发，而不是默认保存。

文档 Markdown 的 YAML 示例：

```yaml
---
source_type: document
original_name: spec.pdf
mime_type: application/pdf
converted_at: 2026-05-21
---
```

## URL

网页端输入 URL 时，语义不是“AI 判断要不要剪藏”。用户输入 URL 就表示希望 ridge 收进来。

目标流程：

```text
URL
  -> 闪念正文或 URL 字段
  -> exa_get_contents 提取网页正文
  -> 写入剪藏/资料库
  -> Markdown YAML 记录 URL 基础信息
```

剪藏 Markdown 的 YAML 示例：

```yaml
---
source_type: web
url: https://example.com/article
title: 页面标题
site_name: Example
captured_at: 2026-05-21
---
```

## URL 内容提取决策

URL 内容提取由 `fleeting-agent` 调用 `exa_get_contents` 触发。该工具直接使用 Exa 官方 Contents API，不依赖 Pi MCP 版 Exa，也不走 Python Converter 的 URL 转 Markdown 路径。

- 后端工具读取 `exa_api_key` / `EXA_API_KEY`。
- 工具只接收公开 HTTP/HTTPS URL，拒绝 localhost、私网和非网页 scheme。
- AI 负责理解网页内容、命名、摘要、标签、归类和沉淀策略。
- 写入剪藏或资料 Markdown 时必须带 YAML 基础信息。

## AI 与后端职责

AI 负责：

- 理解输入素材。
- 生成标题、摘要、标签、分类、正文结构。
- 判断沉淀到剪藏、笔记、任务、里程碑或其他正式知识形态。
- 调用受控工具完成必要的转化或写入。

后端负责：

- 保存闪念和临时附件。
- 提供安全的文件转 Markdown 能力。
- 提供 Exa 官方 API 网页正文提取工具。
- 提供音频转写能力。
- 提供正式附件库写入能力。
- 提供正式知识产物写入能力。
- 记录状态、错误、重试和可观测日志。

## 当前实现差距

当前代码已经有闪念 DB 队列、临时附件库、上传、轮询、重试和内置 `fleeting-agent` 方向，但网页端输入契约还需要继续收口：

- 图片应作为视觉输入，不应默认 OCR。
- 录音转写 Markdown 后要迁移原始音频到正式附件库，并在 YAML 中引用。
- PDF / Word 等文档应转 Markdown，但默认不迁移原件到正式附件库。
- URL 默认是待沉淀素材，不是“是否剪藏”的建议项。
- URL 由 `fleeting-agent` 调 `exa_get_contents` 提取正文后沉淀。

## 验收标准

- 网页端上传图片后，分析链路不调用 OCR，而是把图片作为视觉输入。
- 网页端上传录音后，录音先入临时附件库，转写 Markdown，原音频进入正式附件库，Markdown YAML 引用正式音频地址。
- 网页端上传 PDF / Word 后，文件先入临时附件库，转 Markdown，Markdown 带来源 YAML，原文件默认不进入正式附件库。
- 网页端输入 URL 后，系统默认将其作为待沉淀网页素材处理，最终剪藏 Markdown 带 URL YAML。
- AI 不输出“是否剪藏/是否删除”的建议给用户确认。
- 失败时保留闪念和临时附件，并允许重试。
