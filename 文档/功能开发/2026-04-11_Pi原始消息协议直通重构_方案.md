# Pi原始消息协议直通重构 方案

- 状态：草案
- 日期：2026-04-11
- Owner：AI / ridge
- 目标版本/里程碑：当前工作台消息链路纠偏

## 1. 背景与问题

- 背景：当前 Web 工作台虽然已经移除了 ai-elements-vue，但前后端之间仍然不是直接传递 Pi 原始消息，而是由服务端和前端共同维护了一套自定义 `ChatMessage / ContentBlock` 投影协议。
- 现象/痛点：
  - 后端 `packages/server/src/index.ts` 通过 `extractContentBlocks()` / `serializeMessage()` 对 Pi 原始消息做了结构重组。
  - 前端 `usePiChat.ts` 又对 SSE 事件进行二次累积，把多种 block 混入单条 assistant message。
  - UI 层为了适配这套投影协议，被迫在 message、process group、thinking、toolCall 之间做多层嵌套，导致折叠层级混乱。
  - 用户明确要求：**只要 Pi 原始数据，后端禁止任何处理**。
- 成功定义：服务端不再改写 message/content 语义，Web 端直接消费 Pi 原始 message / 原始 stream event，折叠和渲染严格按 message 边界工作，不再依赖投影协议猜测语义。

## 2. 目标与非目标

- 目标：
  - 删除后端对 Pi message/content 的结构化投影处理。
  - Web 端只消费 Pi 原始 message 与原始 stream event。
  - 保留最小 transport 包装字段，例如 `sessionId`、`status`、`timestamp`。
  - 前端折叠逻辑只按原始 message 边界工作，不再跨 block 做伪协议聚合。
  - 清理当前 `ChatMessage / ContentBlock` 自定义链路中与原始协议冲突的部分。
- 非目标：
  - 不在本次修改 Pi SDK 本身。
  - 不做兼容双协议（新旧协议并存）。
  - 不在本次重做输入区资源系统、会话树、文件树等非消息主链路。

## 3. 现状探索（Explorer）

- 入口与调用链：
  - `packages/server/src/index.ts:308-360` - 服务端 `extractContentBlocks()` 将原始 `message.content` 转为自定义 `ContentBlock[]`。
  - `packages/server/src/index.ts:362-377` - `serializeMessage()` 将原始 message 改写成 `SerializedMessage`，并派生 `text/contentBlocks/createdAt`。
  - `packages/web/src/composables/usePiChat.ts:80-132` - 前端再次定义 `extractContentBlocks()`，对 stream message 进行二次投影。
  - `packages/web/src/composables/usePiChat.ts:881-1048` - SSE 更新阶段把 thinking/toolCall/toolResult 累积进单条 assistant message。
  - `packages/web/src/components/workbench/chat/WorkbenchMessageStream.vue` - 当前消息列表层做“过程组 + 最终正文”分组。
  - `packages/web/src/components/chat/ChatProcessGroup.vue` - 当前过程组继续基于自定义投影 message 渲染摘要与折叠。
- 关键数据结构/状态：
  - `packages/server/src/index.ts` 当前 `SerializedMessage`：`id/role/text/contentBlocks/createdAt`。
  - `packages/web/src/lib/types.ts` 当前 `ChatMessage`：`id/role/text/contentBlocks/createdAt/pending`。
  - `packages/server/src/types/pi-sdk.d.ts:124-162` - Pi 原始 `SessionMessage / SessionEvent / MessageContent` 协议定义。
- 约束与坑点：
  - 当前前端多个组件默认依赖 `ChatMessage.contentBlocks`，不能只改后端不改前端。
  - SSE 订阅逻辑会直接影响滚动贴底、pending 状态、最终正文识别。
  - 若直接切协议，现有 `ChatProcessGroup.vue`、`ThinkingCard.vue`、`ToolCallCard.vue` 需要重新定义输入契约。
  - 用户要求非常明确：后端禁止对 `message/content` 做加工，意味着角色改写、text 摘要派生、contentBlocks 规范化都应移除。

## 4. 资料与依据（Librarian）

- 关键结论：
  - Pi SDK 原始协议已经提供 `SessionMessage.content: string | MessageContent[]` 与 `SessionEvent`，理论上足够前端直接消费。
  - 当前问题不是“消息结构太弱”，而是“后端和前端都额外造了一层消息投影协议”。
  - 用户已经明确确认：只允许保留 transport 必需字段，`message/content` 一律保持 Pi 原样。
  - 当前三层折叠混乱本质上来自错误的消息建模与投影，而不仅是样式问题。

## 5. 方案概览（主方案）

### 5.1 方案摘要

- 一句话：删除后端消息投影层，改为 Pi 原始消息协议直通，前端按原始 message 边界进行消息流分组与渲染。
- 取舍：放弃当前自定义 `ChatMessage / ContentBlock` 带来的“看似方便”的前端简化，换取协议真实、链路透明、渲染边界稳定。
- 影响面：
  - 服务端消息接口：`packages/server/src/index.ts`
  - 前端类型：`packages/web/src/lib/types.ts`
  - 前端状态层：`packages/web/src/composables/usePiChat.ts`
  - 消息列表与过程折叠：`WorkbenchMessageStream.vue`、`ChatMessageItem.vue`、`ChatProcessGroup.vue`

### 5.2 风险与回滚

- 风险：
  - 一次性切协议会影响现有消息列表、历史加载、SSE 流式更新：通过分阶段替换类型和渲染入口缓解。
  - 当前 UI 依赖的 `text` 摘要字段移除后，部分摘要逻辑需在前端重新从原始 message 临时派生：必须限定在展示层、且不能反向形成新协议。
  - 历史 session 与流式 session 若原始 message 格式存在细微差异，可能造成前端渲染边界不一致：需要同时验证历史快照与 SSE。
- 回滚：
  - 回滚策略为 Git 级整体回退，不保留双协议兼容。

## 6. 详细设计（Spec）

### 6.1 交互/行为（如适用）

- 用户流程：
  1. 打开会话时，前端获取的消息列表为 Pi 原始 message 数组。
  2. 流式过程中，前端接收 Pi 原始 stream event，不再由后端重写 message/content。
  3. 消息列表按 message 边界区分：用户消息、assistant 过程消息、最后一条 assistant 正文消息。
  4. 过程折叠只按 message 维度展示，不再让 block 折叠层级超过两层。
- 边界条件：
  - 若 assistant 最后一条 message 不含文本正文，则不强行派生正文区。
  - 若一条原始 message 自身带多个 content item，前端按原始顺序平铺显示，不再额外投影成另一种语义结构。
  - pending / streaming 状态仍需保留，但只能附着于 transport 状态，不改写原始 message。

### 6.2 接口与契约

- API/函数签名：
  - `GET /api/sessions/:id`：返回 session transport 信息 + Pi 原始 messages。
  - `GET /api/sessions/:id/stream`：返回 Pi 原始 session events + transport 状态事件。
  - 前端状态层不再暴露自定义 `ChatMessage/contentBlocks` 作为唯一真相，而是以 Pi 原始 message 协议为主。
- 错误码/异常：
  - 不新增业务错误码。
  - 若原始 message 不符合预期结构，前端仅做渲染降级，不在服务端“修正”数据。

### 6.3 数据与存储（如适用）

- Schema/字段：
  - 删除或废弃自定义 `ChatMessage.text`、`ChatMessage.contentBlocks` 作为主数据契约。
  - 保留 transport 必需字段：例如 `sessionId/status/timestamp`。
  - message/content 使用 Pi 原始结构。
- 迁移策略：
  - 无持久化数据迁移。
  - 迁移发生在运行时接口与前端类型层。

### 6.4 兼容性与发布

- 兼容策略：不保留双协议；直接切换到原始协议。
- 发布步骤：
  1. 服务端删除消息投影处理。
  2. 前端类型改为原始协议。
  3. `usePiChat.ts` 改为原始 event/message 状态管理。
  4. 重写消息渲染组件只按 message 边界渲染。
  5. 执行 lint / ts 检查并手动验证截图问题。

### 6.5 可观测性

- 日志：
  - 记录 session snapshot 和 stream event 的原始结构差异点，便于调试协议切换问题。
- 指标：
  - 关注历史消息打开成功率、SSE 流式更新成功率、最后正文识别正确率。
- 告警：
  - 若出现消息为空、正文丢失、过程折叠跨 message 串组，应视为高优先级回归。

## 7. 验收标准（必须可验证）

- 功能验收：
  - [ ] 后端不再返回自定义 `contentBlocks/text` 投影作为消息主体协议。
  - [ ] Web 端直接消费 Pi 原始 message / 原始 stream event。
  - [ ] 过程折叠严格按 message 边界工作，不再跨 block/文本猜测混组。
  - [ ] 最后一条 assistant 正文消息单独显示，其余 assistant messages 进入过程区。
  - [ ] 不再出现“三层折叠”或 message 与 block 互相混淆。
- 非功能验收：
  - [ ] 性能：切协议后消息流滚动与贴底不明显退化。
  - [ ] 可靠性：历史会话与流式会话表现一致。
  - [ ] 安全：后端未新增对 message/content 的语义加工逻辑。
