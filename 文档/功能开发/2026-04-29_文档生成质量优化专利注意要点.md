# 文档生成质量优化机制专利注意要点

> 本文用于准备中国境内专利申请时内部讨论，不构成法律意见。正式申请前应进行完整专利检索，并由专利代理师确认权利要求范围。

## 一、总体判断

该机制可以尝试写成中国发明专利，但不宜以“理念”或“产品功能”方式申请。更稳妥的写法是把它表达为一个具体计算机实现方法：

```text
模型-技能组合统计推荐
+ 正反例文档评价检索
+ 动态生成提示构造
+ 生成后评分回写
= 文档生成质量闭环优化系统
```

申请重点不应是“大语言模型生成文档”本身，而应是闭环机制中的具体数据处理、检索、评分和组合推荐流程。

## 二、中国专利审查关注点

中国对 AI、算法、商业规则和计算机程序类发明通常重点关注：

1. 是否解决技术问题。
2. 是否采用技术手段。
3. 是否获得技术效果。
4. 算法特征与技术特征是否相互支持。
5. 权利要求是否仅仅是规则、想法、业务流程或人工智力活动。

因此，申请文本中应避免只写：

```text
根据用户反馈选择更好的模型。
根据相似文档让 AI 写得更好。
```

应改写为：

```text
对历史生成记录进行结构化采集，构建任务类型、模型标识、技能标识和质量评分之间的统计映射；
对文档样本和评价评论建立语义向量索引；
在生成前分别检索相似高分样本和相似低分样本；
基于评价信息提取正向写作指引和反向规避指引；
将统计推荐结果与检索指引共同转换为模型可执行的生成提示。
```

## 三、已有相邻专利风险

该方向已有大量相邻技术，不能宽泛主张。

常见相邻方向包括：

1. RAG 检索增强生成。
2. 基于历史文档生成新文档。
3. 文档质量评价。
4. 大模型生成结果评分。
5. 用户反馈闭环优化。
6. Prompt 优化。
7. Agent 或工具调用编排。

初步检索中已看到若干相邻公开：

| 方向 | 示例 |
| --- | --- |
| RAG 检索增强生成 | CN118394890B |
| 基于历史文档相似匹配生成文档 | CN113673210B |
| 文本质量评价 | CN117634468A |
| RAG + 质量评价 + 反馈闭环 | CN119993341A/B |
| 文档撰写质量评分/审核批注 | CN111832266A |

这些专利会影响新申请的新颖性和创造性。正式提交前需要让代理师做完整检索。

## 四、可主张的差异点

建议把差异集中在以下组合机制上：

### 1. 把 skill 作为可统计、可选择的生成能力单元

普通方案多统计模型或 prompt。本方案可以强调：

- skill 是可被调用的生成能力模块；
- 每次生成记录具体 skill 组合；
- 系统按任务类型统计不同模型-skill 组合的质量表现；
- 下一次生成前自动推荐组合。

### 2. 同时使用统计信号和评价检索信号

不是只做 RAG，也不是只做 A/B 统计，而是把两类信号合并：

- 统计信号回答：用哪个模型和 skill。
- RAG 信号回答：参考哪些高分经验，避开哪些低分问题。

### 3. 检索对象不是普通知识，而是带评分和评论的正反例

可强调：

- 高分文档用于提取可学习特征；
- 低分文档用于提取规避特征；
- 评论文本被结构化为优点、缺陷和改进建议；
- 生成提示中注入的是评价归纳，不是原文复制。

### 4. 生成后回写形成闭环

每次生成后的评分和评论会更新：

- 模型-skill 统计矩阵；
- 文档样本标签；
- 语义检索索引；
- 后续生成提示构造逻辑。

## 五、权利要求写法建议

### 独立权利要求应覆盖完整闭环

独立权利要求不要只写“检索高分文档并生成”，应同时覆盖：

1. 任务解析。
2. 模型-skill 质量统计。
3. 相似高分/低分样本检索。
4. 评价信息提取。
5. 动态提示构造。
6. 文档生成。
7. 评分回写。

这样整体组合更有创造性空间。

### 从属权利要求可拆细节

可拆为：

- 质量统计指标：平均分、方差、采纳率、近期趋势。
- 检索策略：分别检索高分样本和低分样本。
- 评价结构化：评论拆成优点、缺陷、改进建议。
- 提示构造：包含正向指引和反向规避指引。
- 多候选生成：多个组合 A/B 生成后评分。
- 人工评分与 AI 评分融合。
- 样本库更新和向量索引更新。

### 系统、设备、存储介质一起写

中国软件类发明通常建议同时写：

- 方法权利要求。
- 系统/装置权利要求。
- 电子设备权利要求。
- 计算机可读存储介质权利要求。

## 六、说明书写法注意

### 要写“技术问题”

不要写：

```text
提高用户满意度。
提升办公效率。
```

应写：

```text
现有文档生成系统无法根据历史生成质量自动选择模型和技能组合；
现有 RAG 系统未区分相似文档的质量等级和评价原因；
低质量历史文档可能被误作为生成参考，导致质量波动。
```

### 要写“技术手段”

要具体到：

- 数据表或数据结构；
- 向量索引；
- 评分维度；
- 聚合计算；
- 检索排序；
- 指引生成；
- prompt 构造；
- 回写更新。

### 要写“技术效果”

可以写：

- 降低低质量样本被召回并误用的概率；
- 提高模型和技能选择的稳定性；
- 减少同类文档生成质量波动；
- 提高文档生成系统对不同任务类型的适配能力；
- 使评分反馈转化为可计算的生成约束。

## 七、容易被驳回的写法

以下写法风险较高：

1. “一种提高 AI 文档生成质量的方法。”
2. “根据用户评分选择模型。”
3. “检索相似文档作为参考。”
4. “根据高分文档和低分文档生成提示词。”
5. “让 AI 学习历史评论后写文档。”

这些表述过宽，容易被认为是抽象规则或已有技术的简单组合。

## 八、较稳的写法

建议使用以下表达：

```text
一种文档生成优化方法，包括：
建立包含模型标识、技能标识、文档任务类型和质量评分的历史生成记录；
根据任务类型对历史生成记录进行聚合，得到模型-技能组合质量统计；
对待生成任务与样本文档分别构建语义向量；
从样本文档中分别召回相似高分样本和相似低分样本；
基于样本关联的评价评论生成正向写作指引和反向规避指引；
将模型-技能组合质量统计、正向写作指引和反向规避指引组合为文档生成提示；
基于所述文档生成提示生成候选文档；
根据候选文档的质量评价结果更新历史生成记录和样本文档索引。
```

## 九、产品实现上的留证建议

为了后续申请更有支撑，建议在产品开发中保留以下证据：

1. 生成任务记录截图或数据库结构。
2. 模型-skill 组合统计页面或接口结果。
3. 高分/低分样本检索结果。
4. 评论结构化结果。
5. 动态提示构造日志。
6. 生成前后评分变化对比。
7. 同一任务在不同组合下的 A/B 质量对比。

这些材料可用于完善实施例和证明技术效果。

## 十、保密与公开策略

在提交专利申请前，应避免公开以下细节：

1. 具体评分公式权重。
2. 样本检索排序细节。
3. 评论结构化模板。
4. 模型-skill 推荐策略。
5. 生成提示的完整拼接模板。

可以公开产品层概念，但不要提前公开核心算法流程、数据结构和实验结果。

## 十一、申请策略建议

### 第一件专利

主张完整闭环：

```text
模型-skill 统计推荐 + 正反例评价检索 + 动态提示 + 评分回写
```

### 第二件可选专利

如果后续实现深入，可拆出：

```text
基于低分相似文档评论的生成风险规避方法
```

或：

```text
面向智能体技能组合的文档生成质量评估与调度方法
```

### 第三件可选专利

如果 UI 和工作流有明显创新，可考虑：

```text
一种文档生成候选对比、评分和采纳回流的人机协同方法
```

## 十二、结论

该机制具备申请价值，但必须强调“具体闭环技术方案”，不能停留在“AI 学习好文档和坏文档”的概念层。

最值得保护的核心句是：

```text
系统同时利用模型-skill 历史质量统计和带评分评论的相似正反例文档检索结果，动态构造文档生成提示，并将生成后的质量评价回写到统计库和样本库中，从而形成面向文档生成质量的闭环优化。
```
