# 内嵌子代理运行时 方案

- 状态：候选
- 日期：2026-04-12
- Owner：AI Agent
- 目标版本/里程碑：子代理能力第二阶段（M3 前重规划）

## 1. 背景与问题

- 背景：当前 `pi_web` 已有主会话 agent 配置、真实 session 持久化、父子会话树恢复、工具消息桥接能力，也已落过一版子代理运行时原型。
- 现象/痛点：上一版 M2 把子代理建成“任务对象 + 专用 `/subagents` API + synthetic message 回写 + 父会话任务视图”模型。用户已明确否定这条路：对父会话而言，子代理不该表现成独立任务 UI，而应表现成普通工具调用；child session 只是底层执行与持久化载体。
- 成功定义：在不引入 task store、任务面板、专用 `/subagents` 前端协议的前提下，让 `task / steer_subagent / get_subagent_result` 三个工具驱动真实 child session，并让父会话只显示普通 tool call/tool result。

## 2. 目标与非目标

- 目标：
  - 把子代理运行时收敛到“真实 child session + 普通工具协议”模型。
  - 保留 child session 持久化与父子 lineage，但不把它投影成父会话任务区。
  - 工具集固定为 `task`、`steer_subagent`、`get_subagent_result`。
  - 删除或退役面向前端的 `/subagents` 专用 API，回归现有 session API 与普通工具结果渲染。
  - 前端不新增任务面板，只保证普通工具参数/返回值可看，child session 可从全局会话树进入。
- 非目标：
  - 不做任务面板、任务列表、任务详情页。
  - 不做独立 task store。
  - 不做额外 task 元信息、task 自定义消息投影、synthetic message 回写。
  - 不做 worktree、RPC、memory、TUI。
  - 不做兼容性双轨：旧的 `/subagents` 前端协议与新工具协议不并存。

  - `packages/server/src/index.ts:1022-1198` - `listWorkspaceSessions()` 会读取 session 的 `parentSessionPath` 并解析成 `parentSessionId`，当前左栏树恢复就是依赖这条链。
  - `packages/server/src/subagents.ts` - 当前已存在一版 `task` 工具与子代理 manager，但它仍偏向“任务对象 + 结果查询”的实现。
  - `packages/web/src/composables/usePiChat.ts:759-869` - 前端当前对工具调用/返回值已走普通消息流，不需要再发明第二套任务展示协议。
- 关键数据结构/状态：
  - `node_modules/@mariozechner/pi-coding-agent/dist/core/session-manager.js:480-490` - `SessionManager.newSession()` 真正把 `parentSession` 写进 session header。
  - `node_modules/@mariozechner/pi-coding-agent/dist/core/session-manager.js:367-376` - `buildSessionInfo()` 真正从 session header 读出 `parentSessionPath`。
  - `packages/server/src/types/index.ts` - 当前还保留 `SubagentTaskSummary / SubagentTaskSnapshot / CreateSubagentTaskPayload`，这是上一版任务模型的残留。
  - `packages/server/src/session-metadata.ts:83-104` - `session-sidebar.json` 现在会镜像 `parentSessionPath`，但它不是 lineage 真相源，只是补充索引层。
- 约束与坑点：
  - 不建 task store 后，运行中状态只能由当前进程内 active session / runtime 维护；服务重启后不存在“恢复中任务”。
  - 不做父会话任务区后，父会话不能再依赖 synthetic message 或额外 UI 协议，只能依赖普通工具消息。
  - 若保留旧 `/subagents` API，会和“工具即协议”形成双轨，违背当前收敛方向。
  - `steer_subagent` 仍要保留，但它应是运行时工具，不应再变成专门的前端页面协议。

## 4. 资料与依据（Librarian）

- 关键结论：
  - Pi SDK 的 session header 原生支持 `parentSession`，并且 session 列表恢复原生读取该字段，因此“父子关系放 session 自身”是有实现依据的，不是额外设计。依据：`node_modules/@mariozechner/pi-coding-agent/dist/core/session-manager.js:367-376,480-490`。
  - 参考插件的核心不是任务面板，而是工具集：`task`、`steer_subagent`、`get_subagent_result`。用户已明确要求沿这套工具语义，而不是保留一层独立任务 UI 心智。
  - 当前 Web 已经能正确桥接普通 `toolCall / toolResult` 历史与实时消息，只要工具结果结构化，父会话无需新增协议即可消费。依据：`文档/模块梳理/Web工作台与Pi服务桥接模块.md:807-836`。
  - 现有产品级用户存储统一走 `~/.ridge/`，但本次方案已明确不新增 task store，因此无需再为 task 单独设计新持久化文件。依据：`packages/server/src/utils/paths.ts:5-35`。

## 5. 方案概览（主方案）

### 5.1 方案摘要

- 一句话：把子代理能力收敛成 3 个普通工具，这些工具内部创建和操控真实 child session；父会话只看工具参数与结构化结果，不再看任务 UI。
- 取舍：放弃上一版“任务对象 / 专用 API / 父会话任务区”方案，换来更简单的协议、更少的状态层、更符合 Pi 原始工具语义的实现。
- 影响面：会重构 `packages/server/src/subagents.ts`、`packages/server/src/index.ts`、共享类型、前端工具消息消费逻辑；会移除或退役 `/subagents` 专用 API 与相关前端任务心智。

### 5.2 风险与回滚

- 风险：
  - 无 task store 后，后台运行态在服务重启后无法恢复：通过文档明确这是当前约束，并在运行期返回清晰状态缓解。
  - 保留 `steer_subagent` 说明仍存在“运行中外部插话”协议：通过严格限定为运行时工具、只接受运行中的 child session 缓解。
  - 旧的 `SubagentTask*` 类型和 `/subagents` 接口残留可能造成实现混乱：通过本轮直接删除或停用相关代码缓解。
  - 父会话不再显示任务面板后，用户入口完全依赖普通工具消息：需要保证 tool result 足够结构化、可读。
- 回滚：
  - 回滚策略：若新工具协议不稳定，先回退到只有 `task` 工具的最小实现，临时关闭 `steer_subagent` 与 `get_subagent_result`，但不恢复任务面板与专用 API。

## 6. 详细设计（Spec）

### 6.1 交互/行为（如适用）

- 用户流程：
  - 主会话中的 agent 调用 `task` 工具，提交 `agent / prompt / run_in_background / inherit_context / model / thinking / max_turns` 等参数。
  - 服务端为这次调用创建真实 child session，并把 `parentSession` 写入 child session header。
  - 若前台等待：工具阻塞到 child session 完成，然后返回 `{ sessionId, status: "completed", result }` 或 `{ sessionId, status: "error", error }`。
  - 若后台运行：工具立即返回 `{ sessionId, status: "running" }`。
  - 运行中的子代理如需转向，主会话或其他 agent 可调用 `steer_subagent`，向该 child session 发送 steering message。
  - 需要查询后台结果时，调用 `get_subagent_result`，按 `sessionId` 拉取当前状态与结果摘要。
  - 若用户想看完整过程，直接打开该 child session；父会话不显示额外任务区。
- 边界条件：
  - 父会话只显示普通 tool call/tool result，不追加 synthetic message。
  - child session 可以出现在全局会话树中，并继续保留真实父子关系；但这不是父会话的专属任务 UI。
  - `steer_subagent` 仅对运行中的 child session 生效，非运行态必须返回明确错误。
  - `get_subagent_result` 只负责查询，不承担持久化任务索引构建。
  - 不保留 `/subagents` 前端协议；前端若查看 child session 详情，走现有 `/api/sessions/:id`。

### 6.2 接口与契约

- API/函数签名：
  - `task(params) -> { sessionId: string, status: 'running' | 'completed' | 'error', result?: string, error?: string }`
  - `steer_subagent(params) -> { sessionId: string, status: 'running', ok: true }`
  - `get_subagent_result(params) -> { sessionId: string, status: 'running' | 'completed' | 'error', result?: string, error?: string }`
  - `createChildSessionFromTask(parentRecord, params) -> SessionRecord`：内部 helper，负责创建带 `parentSessionPath` 的真实 child session。
  - `resolveChildSessionResult(sessionId) -> { status, result?, error? }`：内部 helper，基于 child session 当前运行态和消息历史解析结果。
- 错误码/异常：
  - `SUBAGENT_AGENT_NOT_FOUND`：目标 agent 不存在或被禁用。
  - `SUBAGENT_SESSION_NOT_FOUND`：`sessionId` 不存在或不属于可访问范围。
  - `SUBAGENT_NOT_RUNNING`：`steer_subagent` 目标 session 当前不在运行态。
  - `SUBAGENT_RESULT_NOT_READY`：后台 child session 尚未完成，`get_subagent_result` 只能返回 `running`。
  - `SUBAGENT_INVALID_MODE`：目标 agent 不是 `task` / `all` 模式。

### 6.3 数据与存储（如适用）

- Schema/字段：
  - child session header：`parentSession`（真实 lineage 真相源）
  - `SessionSummary`：继续使用现有 `parentSessionId / agent / status / sessionFile`，不新增 task 专用字段
  - `tool result`：统一使用 `{ sessionId, status, result?, error? }`
  - agent frontmatter：继续沿用已确认字段 `name / description / display_name / mode / model / thinking / max_turns / skills / inherit_context / run_in_background / permission / enabled`
- 迁移策略：
  - 删除或停用 `SubagentTaskSummary / SubagentTaskSnapshot / CreateSubagentTaskPayload / SubagentTasksResponse` 相关共享类型。
  - 删除或停用 `/api/sessions/:sessionId/subagents*` 路由。
  - 删除 synthetic message 回写逻辑，回归普通工具结果。
  - `session-sidebar.json` 若继续镜像 `parentSessionPath`，只保留补充索引语义，不把它升级成 task 持久化层。

### 6.4 兼容性与发布

- 兼容策略：
  - 不保留旧的前端任务面板协议，也不保留 `/subagents` 作为正式前端接口。
  - 父会话中子代理只通过普通工具消息出现，因此前端只需要继续支持工具消息渲染。
  - child session 详情继续复用现有 session 页面与会话树。
- 发布步骤：
  - 第一步：后端收敛工具协议，移除 `/subagents` 路由与 synthetic message 方案。
  - 第二步：前端确认普通 tool result 可读，并让 child session 仍能从会话树进入。
  - 第三步：补齐 `steer_subagent / get_subagent_result` 的运行态约束与错误反馈。

### 6.5 可观测性

- 日志：
  - `task` 创建 child session、启动、完成、失败日志
  - `steer_subagent` 命中运行态/非运行态日志
  - `get_subagent_result` 查询命中、未命中、未完成日志
- 指标：
  - 当前运行中的 child session 数
  - `task` 调用次数、完成率、失败率
  - `steer_subagent` 调用次数与失败次数
- 告警：
  - child session 长时间 `streaming` 未结束
  - `get_subagent_result` 高频返回未命中或错误
  - 某类 agent 的 child session 失败率异常升高

## 7. 验收标准（必须可验证）

- 功能验收：
  - [ ] 主会话中调用 `task` 时，只出现普通 tool call/tool result，不出现任务面板或 synthetic message。
  - [ ] `task` 创建的 child session 为真实持久化 session，并带正确 `parentSession`。
  - [ ] 前台 `task` 完成后，tool result 返回 `{ sessionId, status: "completed", result }`；失败时返回 `{ sessionId, status: "error", error }`。
  - [ ] 后台 `task` 立即返回 `{ sessionId, status: "running" }`。
  - [ ] `steer_subagent` 只能作用于运行中的 child session，失败时返回明确错误。
  - [ ] `get_subagent_result` 能按 `sessionId` 返回 `running/completed/error` 结构化结果。
  - [ ] `/subagents` 专用前端协议被移除或停用，前端不再依赖它。
- 非功能验收：
  - [ ] 性能：child session 运行不阻塞父会话普通消息流渲染。
  - [ ] 可靠性：服务不重启前，运行中的 child session 可被 `steer_subagent / get_subagent_result` 正常访问。
  - [ ] 安全：child session 仍受 agent `permission` 与现有服务端工具约束，不因简化协议放大权限面。
