# 非 Git 项目内置 Git 隔离方案

## 背景

ridge 的右侧 Git 面板当前依赖真实 Git 仓库能力：项目不是 Git 项目时，Git 面板会隐藏。

新的目标是：即使用户添加的是普通文件夹，也要能使用 ridge 内置 Git 完成本地变更查看与提交。但这个内置 Git 不能污染用户项目目录，也不能影响项目后续接入真实 remote Git 后的提交记录。

## 目标

- 非 Git 项目也能使用 Git 状态、文件选择和本地提交能力。
- 内置 Git 不在用户项目目录创建 `.git`。
- 项目后来变成真实 Git / remote Git 项目后，ridge 自动切换到真实 Git。
- 内置 Git 的历史提交不进入真实 Git 的提交记录。
- 旧内置 Git 历史归档保留在 ridge 数据目录，不迁移、不 replay、不删除。

## 非目标

- 不把内置 Git 做成真实 remote Git 的替代品。
- 不为内置 Git 支持 push、pull、fetch、worktree。
- 不把内置 Git 注入终端或 agent shell 环境。
- 不提供兼容旧静态 `is_git` 状态的双路径逻辑；Git 能力以后由运行时解析结果决定。

## 核心设计

采用“外置裸仓库 + 用户项目 work-tree”的方式实现内置 Git：

```bash
git --git-dir ~/.pi/ridge-git/<project-id>/repo.git --work-tree <project-path> ...
```

这样 Git 可以追踪项目文件，但 Git 元数据完全保存在 `~/.pi/` 数据域中，项目目录不会出现 `.git`。

Git 仓库解析优先级：

1. 如果项目目录存在真实 Git 仓库，使用真实 Git。
2. 如果项目目录不是 Git 仓库，使用 ridge 内置 Git。
3. 如果项目后来变成真实 Git 仓库，立即切换到真实 Git。
4. 对应内置 Git 仓库标记为归档，只保留历史，不参与后续 Git API。

## 服务端改动

### Git 仓库解析层

新增统一的 Git 仓库解析服务，用于替代各处直接调用 `gitService.isGitRepository(cwd)` 的判断方式。

解析结果包含：

```ts
type GitRepositoryKind = "real" | "internal";

interface GitRepositoryContext {
  kind: GitRepositoryKind;
  projectRoot: string;
  workTree: string;
  gitDir?: string;
  isGitRepo: true;
  canCommit: boolean;
  canPushPull: boolean;
  canWorktree: boolean;
  label: string;
}
```

真实 Git 项目：

- `kind = "real"`
- `workTree = 项目真实 Git worktree`
- `gitDir = 真实 Git dir`
- `canPushPull = true`
- `canWorktree = true`

内置 Git 项目：

- `kind = "internal"`
- `workTree = 用户项目目录`
- `gitDir = ~/.pi/ridge-git/<project-id>/repo.git`
- `canPushPull = false`
- `canWorktree = false`

### 内置 Git 初始化

当解析到非 Git 项目时，按需初始化内置裸仓库：

```bash
git init --bare ~/.pi/ridge-git/<project-id>/repo.git
```

初始化后写入仓库级配置：

- 默认分支：`main`
- 提交身份：ridge 内置身份
- exclude：`.git`、`node_modules`、`dist`、`build`、`target`、`.next`、`.turbo`、`coverage`、`.pi-web`

内置 Git 的所有命令都必须显式传入：

```bash
--git-dir <internal-git-dir>
--work-tree <project-path>
```

禁止在项目目录执行 `git init`。

### Git API 行为

支持真实 Git 和内置 Git：

- `GET /api/git/is-repo`
- `GET /api/git/status`
- `GET /api/git/branches`
- `POST /api/git/commit`

仅支持真实 Git：

- `POST /api/git/fetch`
- `POST /api/git/pull`
- `POST /api/git/push`
- worktree 相关 API

如果内置 Git 项目调用远程或 worktree 能力，服务端返回明确的 400 错误，不做兜底。

### 项目状态

项目表中的静态 `is_git` 不再作为 Git 能力真源。

后续 API 应以运行时仓库解析结果为准，避免“添加项目时不是 Git，后来变成 Git”后状态失真。

## 前端改动

### Git 能力探测

`useIsGitRepo` 升级为仓库状态 composable，消费服务端解析结果，而不是只消费 `isGitRepo: boolean`。

建议前端状态：

```ts
interface GitRepositoryStatus {
  isGitRepo: boolean;
  kind: "real" | "internal";
  canCommit: boolean;
  canPushPull: boolean;
  canWorktree: boolean;
  label: string;
}
```

### 右侧 Git 面板

非 Git 项目也显示 Git 面板，但显示为“内置 Git”模式。

内置 Git 模式：

- 显示 status、changes、commit。
- 隐藏 fetch、pull、push。
- 隐藏 worktree 入口。
- 提交按钮只执行本地内置 Git commit。

真实 Git 模式：

- 保持现有完整 Git 能力。
- 显示 fetch、pull、push、worktree。

## 数据位置

内置 Git 仓库存储在：

```text
~/.pi/ridge-git/<project-id>/repo.git
```

项目后来变成真实 Git 后，旧内置仓库保留在原位置，并标记为归档。

归档策略：

- 不删除历史。
- 不自动迁移到真实 Git。
- 不 replay 内置 commit。
- 默认不展示归档内置历史。

## 验收标准

- 非 Git 项目添加后，右侧显示 Git 面板。
- 非 Git 项目可以查看变更、选择文件并提交。
- 内置 Git commit 后，项目目录中不存在 `.git`。
- 项目执行真实 `git init` 后，ridge 自动切换到真实 Git。
- 项目接入 remote 后，真实 Git 的 `git log` 不包含内置 Git 的历史提交。
- 内置 Git 模式下不出现 fetch、pull、push、worktree 操作。
- 真实 Git 模式保持现有 Git 能力不变。

## 测试计划

### 服务端测试

- 非 Git 项目解析为 `kind = "internal"`。
- 真实 Git 项目解析为 `kind = "real"`。
- 内置 Git commit 不创建项目目录 `.git`。
- 真实 Git 出现后，解析结果从 `internal` 切换为 `real`。
- 内置 Git 模式调用 push、pull、fetch、worktree 返回 400。
- 真实 Git 模式下现有 Git API 行为不回退。

### 前端测试

- internal 模式显示 Git 面板。
- internal 模式显示“内置 Git”状态。
- internal 模式隐藏远程操作按钮。
- real 模式显示完整 Git 操作。
- 从 internal 切换为 real 后，面板状态随服务端解析结果更新。

## 实施顺序

1. 新增内置 Git 存储路径工具，统一放在 `~/.pi/ridge-git/`。
2. 新增 Git 仓库解析层，先识别真实 Git，再按需解析或初始化内置 Git。
3. 改造 server Git API，使所有 Git 操作先拿仓库解析上下文。
4. 改造项目添加与项目列表逻辑，停止把静态 `is_git` 当能力真源。
5. 改造前端 Git 探测和 Git 面板能力显示。
6. 补齐服务端和前端测试。
7. 更新模块梳理与 `文档/记忆/MEMORY.md`。

## 风险与约束

- 内置 Git 必须只用于 ridge 自身 Git API，不能改变用户终端中的 `git` 行为。
- 内置 Git 不能配置 remote。
- 内置 Git 不能写入项目 `.git`。
- 真实 Git 一旦出现，必须立即优先使用真实 Git。
- 不做兼容式双写，不同时提交到 internal 和 real。
