# 闪念 Inbox 处理台与数据模型设计 v0.2

> 任务：task #7 闪念 Inbox 处理台与数据模型设计  
> 日期：2026-05-01  
> 产品决策：闪念不做长期微博流，而是 Inbox / 待处理输入队列。目标是捕捉后处理清空，最终流向任务、日记、笔记、日历或忽略。  
> v0.2 更新：采纳代码审查建议，收紧 MVP 边界：#7 不阻塞 #8，只新增 captures；日记写入幂等；忽略不硬删除；AI 不自动落地。

## 1. 核心结论

**闪念 = Inbox，不是微博。**

闪念是用户随手捕捉的未整理输入：情绪、想法、任务候选、日记片段、资料链接、临时备忘。它不是最终内容主页，而是一个待处理队列。

每条闪念最终应被处理为：

- 转成 Task；
- 写入 Journal；
- 归档成 Note / Reference；
- 安排到 Calendar / Event；
- 忽略 / 删除；
- 稍后处理。

默认 Inbox 只展示未处理项。处理完成后从主列表消失，但保留处理记录、来源追踪和 RAG 索引。

## 2. 产品心智

### 不再采用微博流

微博流强调长期浏览、评论、动态感。但当前产品更需要：

- 快速捕捉；
- 快速分流；
- 清空待处理内容；
- AI 辅助整理；
- 可追溯和可撤销。

因此闪念页面应像邮件 Inbox / Linear Inbox / GTD 收件箱，而不是社交动态墙。

### 目标心智

> **Inbox Zero for thoughts：把脑子里的临时输入倒出来，再处理到正确位置。**

## 3. 页面设计

### 3.1 页面结构

建议采用“列表 + 详情处理面板”的处理台结构。

```text
┌──────────────────────────────────────────┐
│ 快速捕捉：记下一点什么...                 │
├───────────────────┬──────────────────────┤
│ 未处理列表          │ 详情处理面板            │
│ - Capture A        │ 原文                   │
│ - Capture B        │ AI 判断                │
│ - Capture C        │ 建议动作                │
│                   │ 处理按钮                │
└───────────────────┴──────────────────────┘
```

### 3.2 顶部快速捕捉

输入框文案：`记下一点什么...`

行为：

- 回车创建 Capture；
- 不要求用户先选类型；
- 空内容不创建；
- 创建后进入未处理列表顶部；
- 可选：创建后异步触发 AI 分类。

### 3.3 未处理列表

默认只显示：`status = inbox`。

卡片信息：

- 原文摘要；
- 创建时间 / 相对时间；
- AI 类型建议：情绪、任务候选、日记候选、资料、日程候选、想法、未知；
- 来源：手动、AI、分享、语音、导入；
- 状态：未处理、稍后、处理失败。

筛选：

- 全部未处理；
- 任务候选；
- 日记候选；
- 情绪；
- 资料；
- 稍后。

### 3.4 详情处理面板

点击一条 Capture 后展示：

- 原始内容；
- 创建时间；
- AI 分类和理由；
- 建议动作；
- 处理按钮。

处理按钮：

- `转任务`；
- `写入今日日记`；
- `归档为笔记`；
- `安排到日历`；
- `稍后处理`；
- `忽略`。

### 3.5 空状态

当未处理列表为空：

> 收件箱已清空，当前没有待整理内容。

可提供 CTA：

- 写一条闪念；
- 查看已处理历史。

## 4. 数据模型

### 4.1 Capture / InboxItem

建议 Capture 使用 DB 真源，和 #6 的 Task DB 思路一致。

```ts
type CaptureStatus = "inbox" | "snoozed" | "processed" | "ignored" | "deleted";
type CaptureKindHint =
  | "emotion"
  | "thought"
  | "task_candidate"
  | "journal_candidate"
  | "reference"
  | "event_candidate"
  | "unknown";

type CaptureSourceType = "manual" | "ai" | "share" | "voice" | "import";
type ProcessedToType = "task" | "journal" | "note" | "event" | "ignored";

type Capture = {
  id: string;
  content: string;
  status: CaptureStatus;
  kindHint?: CaptureKindHint;
  sourceType: CaptureSourceType;
  tags?: string[];
  aiSummary?: string;
  aiSuggestedActions?: CaptureSuggestedAction[];
  processedTo?: {
    type: ProcessedToType;
    id?: string;
    path?: string;
    date?: string;
  };
  createdAt: string;
  updatedAt: string;
  snoozedUntil?: string;
  processedAt?: string;
};
```

### 4.2 SuggestedAction

```ts
type CaptureSuggestedAction = {
  id: string;
  action: "create_task" | "append_journal" | "archive_note" | "schedule_event" | "ignore";
  label: string;
  reason?: string;
  payload?: Record<string, unknown>;
  confidence?: number;
};
```

建议动作可以由 AI 生成，但最终落地需要用户点击，或在自动整理模式下由工具执行并记录日志。

## 5. 处理动作

### 5.1 转任务

流程：

1. 用户点击 `转任务`；
2. 系统基于 Capture 内容创建 Task；
3. Task.source = `{ type: "capture", id: captureId }`；
4. Capture.status = `processed`；
5. Capture.processedTo = `{ type: "task", id: taskId }`；
6. 默认 Inbox 列表不再显示该 Capture。

说明：Task 数据模型由 task #6 定义。

### 5.2 写入今日日记

流程：

1. 用户点击 `写入今日日记`；
2. 系统检查当天日记是否已有 `<!-- capture:<id> -->`；
3. 若不存在，追加内容到 `日记/YYYY/MM/YYYY-MM-DD.md` 的固定区块 `## 今日闪念`；
4. 写入内容带 source 标记：`<!-- capture:cap_123 -->`；
5. md 写入成功后，Capture.status = `processed`；
6. Capture.processedTo = `{ type: "journal", path, date }`；
7. 如果 md 写入失败，Capture 保持 `inbox`，并显示错误。

硬边界：写入必须幂等，重复点击不能把同一条 Capture 写入日记多次。

日记仍是 md 真源；Capture 只是来源记录。

### 5.3 归档为笔记

流程：

1. 用户点击 `归档为笔记`；
2. 系统创建或追加到 markdown Note；
3. 默认目录不使用 `项目/`；
4. 可选目录：`笔记/`、`阅读/`、`归档/` 或用户自建非冻结目录；
5. Capture.status = `processed`；
6. Capture.processedTo = `{ type: "note", path }`。

### 5.4 安排到日历

MVP 可后置。

两种路径：

- 如果是行动项：创建 Task 并设置 `dueDate/startDate`；
- 如果是独立事件：后续 Event 模型定义后创建 Event。

在 Event 模型未定义前，不建议强行做复杂日历事件。

### 5.5 忽略

流程：

1. 用户点击 `忽略`；
2. Capture.status = `ignored`；
3. Capture.processedTo = `{ type: "ignored" }`；
4. 默认 Inbox 列表不显示。

### 5.6 稍后处理

流程：

1. 用户点击 `稍后处理`；
2. Capture.status = `snoozed`；
3. 设置 `snoozedUntil`；
4. 到期后回到 Inbox。

MVP 可后置。

## 6. AI 参与方式

### 6.1 AI 是整理助手，不是微博评论

AI 主要做：

- 类型分类；
- 建议动作；
- 批量整理；
- 提取任务标题；
- 生成日记追加文本；
- 识别日期；
- 识别资料链接。

### 6.2 自动处理边界

MVP 第一阶段 AI 只写建议，不自动落地。也就是说，AI 可以写 `kindHint / aiSummary / suggestedActions`，但不直接转任务、写日记、忽略或归档。

未来如果开启自动处理，必须接 `ai_operations` 和撤销。

#### 可自动低风险

- 为 Capture 打 kindHint；
- 生成 aiSummary；
- 生成 suggestedActions；
- 将明显垃圾/重复项标记为建议忽略，但不直接删除；
- 将情绪类建议写入日记，等待用户或自动模式处理。

#### 自动执行但需日志/撤销

- 自动转任务；
- 自动追加到日记；
- 自动归档为笔记；
- 自动忽略。

#### 暂不自动

- 删除 Capture；
- 批量处理大量 Capture；
- 覆盖日记正文；
- 创建复杂日历事件。

### 6.3 操作日志

所有 AI 自动处理必须写入 `ai_operations`，并能追踪：

- AI 做了什么；
- 对哪个 Capture；
- 生成了哪个 Task / Journal / Note / Event；
- 修改前后状态；
- 是否可撤销。

## 7. RAG / 搜索

Capture 虽然是 DB 真源，但需要进入全局索引。

生成 IndexDocument：

```ts
type CaptureIndexDocument = {
  id: string;
  sourceType: "capture";
  sourceId: string;
  title: string;
  text: string;
  metadata: {
    status: CaptureStatus;
    kindHint?: CaptureKindHint;
    processedTo?: Capture["processedTo"];
    tags?: string[];
  };
  updatedAt: string;
};
```

索引策略：

- 未处理 Capture 可索引，但搜索展示优先级低于已沉淀对象；
- 已转 Task 的 Capture，搜索可优先展示 Task；
- 已写入 Journal 的 Capture，搜索可优先展示 Journal；
- Capture 作为来源记录保留。

## 8. MVP 范围

### MVP 做

- Capture / InboxItem 数据模型；
- Inbox 页面：快速输入、未处理列表、详情处理面板；
- 处理动作：转任务、写入今日日记、忽略；
- 处理后从默认 Inbox 消失；
- 已处理历史入口；
- AI 类型建议字段，但 AI 自动执行后置；
- Capture → IndexDocument 函数定义。

### 建议开发切片

#### Slice 1：CaptureRepository + Inbox 基础 CRUD/忽略

- 在 `<workspace>/.ridge/ridge.db` 新增 `captures` 表；
- 不改 `tasks` schema；
- 实现创建 Capture、列表未处理 Capture、读取详情、忽略 Capture；
- 忽略为软状态，不物理删除；

#### Slice 2：写入今日日记

- 写 md 成功后再标记 processed；
- 失败保持 inbox 并提示；
- 测试覆盖重复写入不重复、写入失败不改 Capture。

### MVP 不做

- 不做微博流；
- 不做点赞/评论；
- 不做复杂日历 Event；
- 不做批量自动处理；
- 不做 `项目/` 归档；
- 不做 Capture markdown 真源；
- 不直接和 B1/B2 笔记编辑任务混合；

- #7 不阻塞 #8：转任务动作依赖 task #8 的 `/api/workspace/tasks` DB 能力；如果 #8 未完成，#7 先做 Capture CRUD + 忽略 + 写日记；

- Capture 数据模型不嵌入 Task 内。

### 与 Journal

- Journal 是 md 真源；
- Capture 可追加到 Journal；
- Journal 不承载所有 Capture 的唯一真源。

### 与 Note

- Note 是 md 真源；
- Capture 可归档为 Note；
- Capture.processedTo 记录 note path。
