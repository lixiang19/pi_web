# 38 AI 对话主线闭环

## 目标

把 ridge 的第一主线明确收束为“用户发起 AI 对话并持续工作”的完整闭环：从主页输入、创建/恢复 Pi 会话、流式回复、工具事件、附件上下文、历史恢复、错误恢复到归档只读，全部端到端可验证。

## 背景

现有任务把 AI 对话拆散在多个功能里：

- `05 工作空间主页与会话创建` 只覆盖首次发送创建会话。
- `06 会话索引归档与只读状态` 只覆盖会话列表、索引和只读边界。
- `07 会话界面与右侧工作侧栏` 只覆盖界面操作和侧栏。

这些都不是完整主线。AI 对话能力必须单独作为主线任务管理，否则容易出现“会话能创建、界面能打开，但真实多轮 AI 工作不可验收”的断层。

## 范围

- 主页输入后创建真实 Pi session，并发送首条用户消息。
- 已有会话恢复后可继续发送消息。
- 服务端使用 Pi SDK session 作为消息真源，不在 Web 层伪造会话语义。
- 服务端通过 SSE/Web stream 桥接真实 runtime 事件。
- 前端展示用户消息、助手消息、thinking、tool call、tool result、错误状态。
- 支持多轮连续对话，历史消息分页单位按 user 轮次。
- 支持模型、Agent、thinkingLevel 的会话级选择与运行时传递。
- 支持附件 ID 进入首轮和后续消息上下文。
- 支持 ask / permission 的阻塞交互入口。
- 支持发送中、取消、失败重试、恢复订阅、重复事件去重。
- 支持归档会话只读：不能继续发送，服务端返回明确错误。
- 支持同一个会话在标签池/LRU 中保活，切换标签不丢流。

## 不做

- 不把任务系统、RAG、Wiki、图谱混入本任务。
- 不伪造 AI 回复、不 mock Pi runtime。
- 不把消息正文复制进 `ridge.db`。
- 不做多用户协作。
- 不做桌面离线设备的远程会话正文读取。

## 验收

- 未配置可用模型时，发送失败有明确错误，输入不丢失。
- 配置可用模型后，主页首发能创建 Pi session、写入 session index、原地切换为会话标签，并显示流式 assistant 回复。
- 恢复已有会话后继续发送，前端能收到新增 user/assistant/runtime 事件。
- 切换标签再切回，正在流式输出的会话不中断或能正确恢复订阅。
- 多轮历史按 user 轮次分页，展开历史不破坏当前流。
- tool call、tool result、thinking 可见且与对应 assistant 消息关联正确。
- ask 请求显示底部阻塞表单，提交后 runtime 继续推进。
- permission 请求显示独立审批卡片，允许/拒绝后 runtime 继续推进或明确失败。
- 附件发送后进入 runtime 上下文，后端不泄露 `storedPath`。
- 归档会话发送消息返回 403，前端禁用 composer 并显示只读原因。
- 网络中断、SSE 断开或刷新页面后，不重复显示用户消息，不丢失已持久化历史。
- 根目录 `npm run check` 通过。
- 后端、前端测试通过，并有 Playwright E2E 覆盖真实浏览器主链路。

## Spec 提取点

- Pi session 生命周期。
- 首发和后续消息 API 契约。
- SSE runtime event 协议。
- 消息快照装配边界。
- ask / permission 阻塞交互协议。
- 附件上下文协议。
- 标签池/LRU 与订阅生命周期。
- 归档只读服务端边界。

## Spec 草案

### API

- `POST /api/sessions` 创建真实 Pi session 和 ridge session index。
- `GET /api/sessions/:id/messages` 读取消息历史，不隐式打开完整 runtime。
- `POST /api/sessions/:id/messages` 向真实 Pi session 发送消息。
- `GET /api/sessions/:id/events` 订阅 runtime 事件。
- `POST /api/sessions/:id/cancel` 取消当前运行。
- `POST /api/sessions/:id/ask/:requestId` 提交 ask 响应。
- `POST /api/sessions/:id/permissions/:requestId` 提交权限审批。

### 前端状态

- `draft`：尚未绑定 Pi session 的主页输入。
- `creating`：首发创建 session 中。
- `active`：会话可发送。
- `streaming`：等待或接收 assistant/runtime 事件。
- `blocked_on_ask`：等待用户回答 ask。
- `blocked_on_permission`：等待用户审批工具权限。
- `readonly`：归档或设备不可用导致只读。
- `failed`：发送或订阅失败，可重试。

### 消息边界

- Pi 原始消息结构是协议真源。
- `pending/localId` 只能存在于前端 UI 包装层。
- user message 由前端乐观显示时，服务端 SSE 必须过滤重复 user start/end 事件。
- tool call 和 tool result 必须保留 `toolCallId`、`toolName`、`details`、`isError`。

### 测试

- 服务端集成测试：创建 session、发送消息、归档拒绝发送、附件 ID 校验、ask/permission 提交。
- 前端组件测试：composer 状态、流式消息合并、pending 去重、只读禁用、ask/permission 卡片。
- E2E：登录、主页首发、流式回复可见、继续追问、刷新恢复、归档后不可发送。
