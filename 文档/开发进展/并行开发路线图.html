闪念线 12–16 详解

这条线的核心目标是：快速捕获 anything，不打扰，用户确认后再正式化。 以下是 5 个环节的职责边界。

12
闪念 DB + Web 入口
数据模型：闪念头（文本/类型/来源）+ 附件关联。Web 提供快速输入框，不选项目、不分类。
13
临时附件生命周期
附件先落盘到 .ridge/fleeting-attachments。未处理前：不进 RAG、不进文件树、不给 MCP。
14
桌面采集入口
桌面端菜单栏一键采集：文字、截图、文件、剪贴板、选区、浏览器网址、录音。统一上传到服务器变成闪念。
15
fleeting agent 建议
依赖 25 后台队列。Agent 只读闪念内容，给出分类/任务/剪藏建议，不自动执行，等用户确认。
16
正式化处理
用户一键确认后，闪念迁移为：任务/里程碑、剪藏、正式附件。目标对象创建成功后删除原闪念记录；附件成功迁移为正式附件；失败则保留闪念和临时附件。
13 与 14 的关系

13 是后端边界：无论 Web 还是桌面采集，上传的附件都先进入 .ridge/fleeting-attachments，处于"隔离区"。未处理成功前，RAG 不索引、MCP 不可读、文件树不展示。处理成功后，才迁移到 附件/ 或对应项目目录，成为正式资产。

14 是前端入口：桌面端提供系统级采集能力（菜单栏、全局快捷键），把多种输入格式统一封装成闪念 API 的 payload 发到服务器。14 依赖 12 的 API 和 31 的桌面在线能力；采集到的文件/截图走的正是 13 定义的临时附件链路。
