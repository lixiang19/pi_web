# 文件处理流程

## 定位

文件处理流程负责把进入工作空间的文件变成长期可读、可检索、可引用的知识资产。

它不是“导入”。

上传只是文件进入 ridge 的入口。

## 文件入口

第一版文件入口包括：

- 用户上传到工作空间；
- 用户创建或编辑 Markdown；
- 闪念处理成功后迁入正式对象；
- 会话处理后产生正式文件；
- 剪藏保存；
- 内部项目中的文档文件。

未处理闪念附件、会话临时上传、处理中附件都不进入正式文件处理流程。

## 临时文件

临时文件放在 `.ridge` 下的系统隐藏目录。

临时文件包括：

- 闪念临时附件；
- 会话临时上传；
- 转换中的中间文件；
- 失败重试所需的临时缓存。

临时文件：

- 不进入 RAG；
- 不进入图谱；
- 不给 workspace MCP 读取；
- 不作为长期资产展示。

## 正式文件

正式文件位于工作空间可见目录。

正式文件包括：

- `笔记/`；
- `日记/`；
- `剪藏/`；
- `附件/`；
- `记忆/`；
- `Wiki/`；
- `空间/`；
- `项目/` 内部项目文件；
- 用户自定义可见目录。

正式文件可以被 RAG 索引。

正式文件可以被 workspace MCP 读取。

## 正式附件

`附件/` 是正式附件目录。

正式附件是长期资产。

正式附件可以被读取。

正式附件可以被笔记、日记、任务、项目、剪藏、会话引用。

正式附件进入 RAG。

## 标准化转换

RAG 和图谱不直接基于混杂原文件工作。

文件先转换为标准产物。

标准产物包括：

- Markdown；
- assets；
- metadata JSON。

示例：

```txt
paper.md
paper.assets/
paper.metadata.json
.originals/paper.pdf
```

PDF、Word、音频等原文件转换后归档到同目录隐藏 `.originals`。

转换产物留在原文件所在目录。

用户主要阅读和编辑 Markdown 产物。

转换失败时保留原文件并记录失败状态。

## PDF 和 Word 处理流程

PDF、Word 是第一版最重要的复杂文件类型。

它们进入正式处理流程后，必须经过标准化解析。

流程：

1. 保存原文件；
2. 创建文件处理任务；
3. 调用 Docling 或同类解析能力；
4. 生成 Markdown；
5. 提取图片、表格和其他 assets；
6. 生成 metadata；
7. 原文件归档到 `.originals`；
8. 基于 Markdown 和 metadata 构建 RAG；
9. 记录处理状态。

生成物必须和原文件在同一语义位置。

示例：

```txt
附件/paper.md
附件/paper.assets/
附件/paper.metadata.json
附件/.originals/paper.pdf
```

Markdown 是用户主要阅读、编辑和引用的对象。

原 PDF、Word 只作为归档原件保留。

RAG 默认索引 Markdown，不直接索引原 PDF、Word。

如果 Markdown 已被用户编辑，后续自动转换不得覆盖。

重新转换必须由用户手动触发并确认。

## PDF 解析要求

PDF 处理至少保留：

- 标题；
- 正文段落；
- 页码；
- 标题层级；
- 表格；
- 图片；
- 脚注或尾注；
- 原文件 metadata；
- 解析失败信息。

表格优先转换为 Markdown 表格。

无法可靠转换的表格，保留为图片 asset，并在 Markdown 中引用。

图片保存到 `.assets/`。

图片需要生成 metadata。

图片可进入图片向量索引。

图片 OCR 文本可以进入 RAG。

第一版不为图片自动生成视觉描述。

## Word 解析要求

Word 处理至少保留：

- 标题层级；
- 正文段落；
- 列表；
- 表格；
- 图片；
- 脚注或尾注；
- 文档属性 metadata。

修订、批注、复杂样式第一版不作为核心语义。

能保留就写入 metadata，不能稳定保留时明确降级。

Word 转换后的 Markdown 是 RAG 真源。

## RAG 构建流程

RAG 构建只面向标准产物。

输入包括：

- Markdown 正文；
- OCR 文本；
- 音频转写文本；
- metadata；
- 文件路径；
- 文件类型；
- 更新时间；
- 所属工作空间目录；
- 可引用 asset 信息。

RAG 构建流程：

1. 读取标准产物；
2. 解析 Markdown 结构；
3. 按标题、段落、表格和语义边界切块；
4. 为每个 chunk 生成 embedding；
5. 保存 chunk 文本和 metadata；
6. 建立路径、标题、类型、时间等过滤字段；
7. 标记文件已索引。

chunk 必须保留来源信息。

至少包括：

- 工作空间相对路径；
- 标题路径；
- chunk 序号；
- 更新时间；
- 内容 hash；
- 文件类型。

检索结果必须能回到原 Markdown 和对应位置。

## RAG 更新规则

上传文件后立即进入转换和 RAG 构建。

普通 Markdown 编辑不立即索引。

后台 Agent 改写 Markdown 不立即索引。

这些变更进入夜间统一索引。

用户可以手动触发立即索引。

重新索引时用内容 hash 判断是否需要更新。

文件删除后，必须删除对应 RAG chunk。

文件移动或改名后，必须更新路径 metadata。

转换失败时，不写入新的 RAG chunk。

索引失败时，保留转换产物并记录失败状态。

## RAG 与图谱关系

RAG 先于图谱运行。

图谱基于标准产物和必要 RAG 结果抽取实体关系。

图谱不直接读取混杂原文件。

夜间维护顺序固定为：

1. RAG；
2. 图谱；
3. Wiki。

## 类型处理

PDF、Word：

- 转为 Markdown；
- 提取图片和 assets；
- 生成 metadata；
- 原文件归档。

音频：

- 转为转写 Markdown；
- 保留必要分段时间戳；
- 生成 metadata；
- 原文件归档。

图片：

- 生成图片向量；
- 生成 OCR 文本；
- 生成 metadata；
- 不生成视觉描述。

Markdown：

- 直接作为可编辑文本资产；
- RAG 以当前 Markdown 为准。

## 索引策略

用户上传文件后立即转换和更新 RAG。

普通文件变更、用户编辑 Markdown、后台 Agent 改写 Markdown，不立即更新 RAG。

这些变更等待夜间统一索引。

用户可以手动触发立即更新索引。

上传后只立即更新 RAG。

图谱和 Wiki 等夜间维护。

夜间顺序固定为：

1. RAG；
2. 图谱；
3. Wiki。

## Markdown 编辑

转换生成的 Markdown 允许用户编辑。

RAG 以编辑后的 Markdown 为准。

如果 Markdown 已被用户编辑，自动转换不得覆盖。

重新转换必须由用户手动触发并确认。

## 状态

文件处理状态至少包括：

- 待处理；
- 转换中；
- 已转换；
- 已索引；
- 转换失败；
- 索引失败。

失败状态不吞掉原文件。

失败原因必须可见。

用户可以重试失败处理。

## 后台关系

文件处理本身是系统流程，不默认算一个后台 Agent。

需要 AI 判断内容归类、总结或生成 Wiki 时，交给对应后台 Agent。

- 闪念分析由 fleeting agent 做。
- 会话记忆由 summary agent 和 memory agent 做。
- 图谱维护由 graph agent 做。
- Wiki 维护由 wiki agent 做。
