# 闪念系统与桌面采集

## 定位

闪念是临时捕获队列。

- 入口和页面统一叫“闪念”。
- 不再使用“收件箱”概念。
- 闪念 DB 只保存未处理闪念。
- 处理成功后删除原闪念记录。
- 采集时不选择项目。
- 捕获阶段只保存，不分类、不打标签、不追问。

## 分析与处理

闪念保存成功后，可以交给闪念 Agent 后台分析。

- 闪念 Agent 只生成一个最佳处理建议。
- AI 建议不能自动执行。
- 用户必须点击处理按钮才会沉淀为正式对象。
- 固定处理按钮始终可用，不依赖 AI 分析结果。
- 用户在分析完成前手动处理成功时，取消该闪念的后台分析。

处理成功后删除原闪念。

处理失败时不得删除原闪念。

只有目标系统确认创建成功后，才删除原闪念和迁移附件。

## 状态

闪念主状态保持简单：

- 待处理；
- 处理中。

分析状态单独记录：

- 未分析；
- 分析中；
- 已建议；
- 分析失败。

分析失败不打扰用户；用户仍可使用固定处理按钮。

## 采集类型

桌面端支持：

- 文字；
- 截图；
- 文件；
- 剪贴板；
- 当前选区；
- 浏览器网址；
- 录音。

Web 端闪念入口也支持文字和附件上传。

Chrome 插件支持浏览器阅读采集：

- 插件注册为 `browser` 设备。
- 捕获阶段 `fleeting_notes.content` 只保存脱敏 URL。
- 页面标题、站点名、语言和阅读信号只作为 `metadata_json` 溯源信息保存。
- 自动捕获必须满足停留、滚动或重复访问等真实阅读信号。
- 手动捕获当前页始终可用。
- 采集结果统一进入闪念，`capture_type=browser_page`。
- 处理为剪藏时，Node 后端用 Python Converter 的 `document.markdown` + `markitdown` 按 URL 拉取网页并转 Markdown。
- 转换成功后写入 `剪藏/<标题>.md`，创建剪藏记录，触发 RAG；转换失败时保留原闪念。
- 插件不创建独立知识库，不直接写 RAG、图谱或 Wiki。
- 插件不采集网页正文、摘录文本、表单输入、cookie、按键、鼠标轨迹。

截图和文件闪念允许没有文字说明，也允许附带文字说明。

## 桌面入口

桌面端提供 mac 菜单栏常驻入口。

点击菜单栏入口后展开一列采集按钮：

- 文字；
- 截图；
- 文件；
- 剪贴板；
- 当前选区；
- 浏览器网址；
- 录音。

截图同时支持快捷键触发。

截图模式：

- 区域截图；
- 窗口截图；
- 全屏截图。

浏览器网址采集通过系统权限读取当前浏览器标签。

- 能读取 URL 和标题时只保存 URL，标题只作为 metadata。
- 权限不足或浏览器不支持时明确提示失败。

录音先作为音频闪念保存，不在采集当场转写。

## 附件目录

闪念临时附件和正式附件分开。

临时附件：

- 路径：`~/ridge-workspace/.ridge/fleeting-attachments`；
- 系统隐藏；
- 属于未处理闪念生命周期；
- 删除未处理闪念时，未被引用的临时附件一起删除。

正式附件：

- 路径：`~/ridge-workspace/附件`；
- 用户可见；
- 属于长期附件；
- 进入工作空间 RAG。

## 附件生命周期

- 采集截图或文件时，先上传副本到闪念临时附件目录。
- 闪念处理成功后，附件迁移到正式附件目录。
- 正式附件保留原文件名；冲突时追加短 ID 去重。
- 目标对象引用正式附件。
- 未处理闪念和临时附件不进入 RAG。
- 处理沉淀后的正式附件进入 RAG。

## 附件分析

闪念 Agent 第一版分析：

- 文字内容；
- 图片内容。

PDF、Word 等文档附件先作为原文件保留，不在闪念系统中决定是否转 Markdown。

PDF、Word、长文档解析与转换规则留给文件/RAG 设计。

## 处理关联项目

采集时不选择项目。

处理闪念为正式对象时，可以选择是否关联项目。

如果处理为任务或里程碑，可以选择项目。

如果处理为剪藏或附件，进入工作空间正式附件和 RAG 流程。
