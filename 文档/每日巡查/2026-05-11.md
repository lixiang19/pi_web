## 2. 总体判断

仓库当前处于**“多里程碑并行展开、文档状态严重滞后于代码”**的阶段。代码中已出现并部分接入了 M2 任务系统、M3 闪念、M9 自动化、M10 设置、文件/预览、终端、附件等后续能力，但 `功能开发/index.md` 仅将 04/05 标记为 ✅ 归档，00–03、06–37 均未标完成。这导致：

1. **无法从文档判断哪些能力已可端到端工作**——只能逐文件 grep 确认。
2. **存在多个 P0 级功能缺陷**——路由未挂载、依赖注入顺序颠倒、安全边界失效，这些问题若进入用户环境将直接 404/503/数据越界。
3. **数据模型双轨与 schema 漂移**——同一功能领域存在两套表结构，部分表不在版本化迁移中，未来演进风险高。

因此，当前不应再新增“占位式”功能入口；首要任务是：补齐端到端可用性、补文档状态矩阵、以集成测试为门禁收敛归档标准。

## 4. 需要修复的 Bug / 风险

### 4.1 P0 闪念 API 未挂载，收件箱入口会 404

- **现象**：前端 `useInbox.ts` 与 `api.ts` 已完整调用 `/api/fleeting/*`，但服务端 `index.ts` 未装配该路由。

  - `packages/server/src/index.ts` 全文无 `createFleetingRouter` import 或 `app.use('/api/fleeting', ...)`；
  - grep 结果仅在测试代码中发现挂载。  
- **影响**：用户点击“闪念”或尝试保存剪藏时，服务端返回 404，功能完全不可用。  
- **建议**：
  1. 先按 TDD 增加 app 级集成测试（覆盖列表、创建、处理）；
  2. 在 `index.ts` 中以真实的 `getRidgeDb()` 和 `defaultWorkspaceDir` 挂载 `createFleetingRouter`；
  3. 确认鉴权链路（`authMiddleware`）装配在其前面。

### 4.2 P0 自动化服务初始化顺序导致 `/api/automations` 可能一直 503 / no-op

- **现象**：`automationStore` 和 `automationScheduler` 在依赖注入时仍为 `null`，导致路由层和 session payload 层均无法正常工作。  
- **证据**：
  - `packages/server/src/index.ts` 行 405–415：`initSessionPayload({ automationStore })`，此时 `automationStore` 尚未赋值；
  - `packages/server/src/session-payload.ts` 行 100–108：`getAutomationStore()` 读取的是 deps 内保存的 `null`；
  - `packages/server/src/index.ts` 行 1246 之后：`automationStore = createAutomationStore(db)` 才真正创建；
  - `coreRouter` 创建时传入的 `automationScheduler` 也是当时的 `null`，路由内 `automationScheduler?.reschedule()` 永远 no-op。  
- **影响**：自动化列表/创建/运行接口在启动后始终处于“有路由无服务”状态；前端 `AutomationTabContent.vue` 保存自动化后不会重新调度。  
- **建议**：
  1. 将 `automationStore` / `automationScheduler` 改为 lazy getter 注入，或在 DB 初始化完成后再创建依赖自动化的 router；
  2. 补 API 集成测试覆盖：列表、创建、toggle 后调度触发；
  3. 在 `initSessionPayload` 调用前确保所有依赖已就绪。

### 4.3 P0/P1 通用 `/api/files/create` 没复用文件管理边界，且先 `mkdir` 再越界检查

- **现象**：文件创建接口自行拼接路径、先递归创建目录、后做越界检查，未复用 `file-manager.ts` 的 root/realpath/`.ridge` 拒绝逻辑。  
- **证据**：
  - `packages/server/src/index.ts` 行 445–460：
    ```ts
    targetPath = path.resolve(defaultWorkspaceDir, trimmed)
    // ...
    await fs.mkdir(parentDir, { recursive: true })
    // ...
    parentReal.startsWith(rootReal) // 检查在 mkdir 之后
    // ...
    writeFile
    ```
  - `文档/模块梳理/ridge系统目录边界.md` 行 23–27 要求：`.ridge` 系统目录拒绝、越界检查必须在任何 IO 前完成。  
- **影响**：
  - 恶意 `../` 路径可在 `startsWith` 检查失败前创建工作区外目录；
  - `.ridge` 边界可能被绕过；
  - 与模块文档直接冲突。  
- **建议**：
  1. 改用统一 `file-manager.ts` 的 `createEntry` 保存链路；
  2. 任何 `mkdir` / `writeFile` 前完成 lexical + `realpath` + `.ridge` 边界校验；
  3. 补越界路径的集成测试（含 `../`、`/etc/passwd`、`.ridge` 子路径）。

## 5. 优化建议

1. **建立“功能状态矩阵”**  
   在 `文档/功能开发/index.md` 或独立表格中维护：文档编号、导航入口、API、DB、前端、单元测试、集成测试、E2E、是否归档、未验收缺口。避免仅凭“代码存在”即判断功能完成。

2. **归档必须以端到端验收/自动化测试为门禁**  
   任何功能在 `index.md` 标 ✅ 前，必须满足：
   - 路由已挂载（服务端）
   - API 客户端调用路径可通（前端 `api.ts`）
   - 至少一条 API 集成测试通过（服务端）
   - 无 placeholder 组件独占该功能页面

3. **优先修复 P0，再处理 P1，最后做 P2**  
   按“可用性 > 数据安全 > 功能完整 > 文档同步”的顺序投入。具体见下方“建议执行顺序”。

4. **统一文件系统边界校验**  
   所有涉及路径的 API（`files/create`、`files/read`、`files/write`、`attachments` 等）都应复用同一 `file-manager.ts` 入口，禁止在路由层自行 `path.resolve` + `fs.mkdir`。

5. **认证密码脱离硬编码**  
   即使当前无外部用户，也应在部署文档中说明密码配置方式，避免镜像泄露即沦陷。
