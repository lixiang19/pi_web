# 项目目标

把 pi 包装、定制、封装为 AI 工作平台。

# 强制准则

- 功能必须完整设计，禁止因复杂而简化。
- 简洁优先：只写解决问题所需的最少代码，不做推测性扩展。
- 手术式变更：只改必要位置，不顺手重构、不清理无关代码；因本次修改产生的未用导入/变量/函数必须移除。
- 写代码前先思考：明确假设、歧义、权衡；不清楚就先问。
- 目标驱动：每步都有可验证标准，循环到确认完成。
- Bug 要看全链路，从根因修复；禁止用兜底、忽略、延迟、`any`、跳过检查等方式掩盖问题。

# 代码检测与质量门禁

- 修改 `.ts` / `.vue` / `.js` 后，必须在根目录运行 `npm run check`，禁止只跑测试就结束。
- `npm run check` 包含 ESLint 与 TypeScript 检测；前端类型检测可运行 `npm run typecheck` 或 `cd packages/web && npm run check`。
- ESLint 查规范和部分静态问题；TypeScript / `vue-tsc` 查未定义标识符、缺失 import、类型不匹配、Vue SFC 类型错误。
- 测试不能替代类型检测；`npm run test` 通过不代表可合并。
- `npm run check` 失败时不得完成任务，必须修复或向用户报告阻塞日志。

# 测试驱动开发

先写测试，再写代码。

流程：

1. 写测试；
2. 运行并确认失败；
3. 写最小实现；
4. 在测试保护下重构。

规则：

- 禁止提交无测试的功能代码。
- 禁止先实现后补测试，紧急线上 Bug 除外。
- 新 API 必须有集成测试；新 composable/工具函数必须有单元测试；新组件必须有组件测试。
- 测试必须覆盖 spec 的功能点和边界。

测试命令：

- 后端：`cd packages/server && pnpm test`
- 前端：`cd packages/web && pnpm test`
- 全量：`pnpm test`

# Spec 优先

开发前先在 `文档/spec/` 写清输入、输出、边界，测试从 spec 生成，不从实现倒推。

# 核心约束

- Tauri 只打包，不做功能开发。
- Node 做服务层，提供 API 和持久化。
- 界面纯前端，文件选择用 Web 原生方式。

# 品牌与数据

- 品牌名：ridge
- 数据目录：`~/.pi/`
- 配置与运行状态存储：`~/.pi/ridge.db`
- 默认工作空间目录：`~/ridge-workspace`

# 四愿景

- 双工作模式：代码模式专注工程开发，助手模式专注知识对话；支持一键切换与移动端触控、碎片场景、语音优先输入。
- 文件预览编辑：多格式预览、轻量编辑、与对话深度关联。

# 开发要求

- 文档/examples 是 pi 示例。
- 多用网络搜索和 Git 搜索 `@mariozechner/pi-coding-agent`。
- 多读 pi 源码和 pi-extensions。
- pi 已支持会话持久化、插件钩子等能力；禁止伪造、mock、模拟，先探索现有实现。

# 界面开发

- 必须使用 shadcn-vue：<https://www.shadcn-vue.com/>，禁止手写基础组件。
- 多安装 shadcn-vue 组件。
- 必须使用主题文件，并支持明暗模式。

# 文档要求

- 新任务在 `文档/功能开发/` 新建 md，完成后归档到 `文档/功能开发/归档/`。
- 开发经验更新到 `文档/记忆/`，优先 `MEMORY.md`。
- 修改前先读取 `文档/模块梳理/`；涉及模块职责、接口契约、主流程、配置边界、数据结构变化时，必须更新对应模块梳理文档。
